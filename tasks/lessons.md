## 2026-03-19 — handoff-discovery-ux

### 스킬 지시문과 실제 출력의 괴리
- **문제:** context-handling에 출력 템플릿이 정의되어 있었지만, Claude가 실제로 따르지 않았음
- **원인:** orchestrator가 탐색 로직을 인라인으로 포함하여 Claude가 중간 과정을 보고함
- **교훈:** 출력 책임을 한 곳에 일원화하고, 금지 출력 패턴을 명시적으로 나열해야 함

### 완료된 작업의 잔존 HANDOFF 처리
- **문제:** `status: complete`인 작업의 HANDOFF.md가 남아있으면 "진행 중인 작업"으로 노출됨
- **교훈:** status 체크를 HANDOFF 존재 여부보다 우선순위로 두어야 함. 잔존 HANDOFF는 삭제를 제안할 것

### 설계 문서 테이블 중복
- **문제:** 라벨 규칙 테이블에서 `status: complete` 항목이 우선순위 이동 후 원래 위치에 중복 잔존
- **교훈:** 우선순위 변경 시 원래 위치의 항목을 반드시 제거할 것

## 2026-03-26 — skill-aliases

### 플러그인 commands/ 추가 후 Claude Code 재시작 필요
- **문제:** 플러그인에 commands/ 파일을 추가하고 push까지 완료했지만 콘솔에서 새 커맨드가 노출되지 않음
- **원인:** Claude Code가 세션 시작 시 플러그인 커맨드를 캐싱하므로, 런타임 중 추가된 파일은 감지하지 않음
- **교훈:** commands/ 디렉토리에 파일을 추가/수정한 후에는 사용자에게 Claude Code 재시작을 안내할 것

## 2026-07-08 — skill-slimming (G4)

### 플랜의 수치 프록시 체크는 구조 상한을 먼저 계산할 것
- **문제:** plan.md에 "본문 40%+ 감소, 합계 ±5%" 체크를 넣었으나 실측은 24.8%/8.9%, +10~12% — 이동 가능 바이트의 구조적 상한과 고정 삽입 오버헤드를 플랜 작성 시 계산하지 않음
- **교훈:** 수치 프록시를 플랜에 넣기 전에 상한을 실측(또는 추정 근거 명시)할 것. 프록시가 깨지면 원래 증명하려던 직접 속성(byte-identical 이동 + 순증 전량 분류)을 검증해 대체 판정할 것 — 설계 문서의 "수치 목표는 비목표" 원칙이 플랜 세부 체크보다 우선한다

### 치환 지정은 대상 섹션의 경계를 명시할 것
- **문제:** "상태 정규화 전처리 섹션을 1줄로 치환" 지정이 섹션 경계를 명시하지 않아, 구현자가 인접한 "0. 경로 해소" 스텝(F3 추적 앵커 포함)까지 함께 삭제 — 태스크 리뷰가 포착해 복원
- **교훈:** 플랜에서 블록 치환을 지정할 때 시작/끝 라인 또는 경계 문장을 명시할 것. 크로스레퍼런스 인벤토리(F3)가 안전망으로 실제 작동했음 — 추적 앵커 등재의 가치 실증

## 2026-07-09 — conversation-ux

### 규범의 의미를 역전시킬 때는 구 독트린 문장을 전수 스윕할 것
- **문제:** §4를 "일괄 수용→노출된 선택지에만"으로 역전 개정했는데, 같은 의미를 서술하는 무변경 컨텍스트 줄 5곳(§1 응답 주권·8+ 푸터·폴백 2점·brainstorming 폴백)이 구 독트린 그대로 잔존 — 그중 1곳은 항상 주입되는 bootstrap이라 사고 재발 경로였음. 태스크 단위 리뷰는 변경 지점만 보므로 전부 통과했고, 최종 브랜치 리뷰(전체 시야)에서야 적발
- **교훈:** 규범 의미를 바꾸는 변경은 "그 규범을 재서술하는 문장"의 grep 인벤토리(예: "일괄", "수용한다" 계열)를 게이트 인벤토리와 병행해 전수 스윕할 것. 태스크 리뷰의 사각(무변경 줄)은 전체 리뷰가 유일한 망 — 최종 리뷰를 생략하지 말 것

### 출력 형식 펜스의 필수 슬롯을 체크리스트로 검증할 것
- **문제:** 플랜이 작성한 Socratic 노출 펜스에 `[질문 내용]` 슬롯이 누락된 채 verbatim 반영됨 — 펜스를 리터럴 출력하는 모델은 질문 없는 질문 화면을 냄. 신선한 눈 리뷰가 동형 펜스(A-0)와의 대조로 적발
- **교훈:** 출력 템플릿을 플랜에 확정 문면으로 박을 때, 동형 템플릿 간 슬롯 대조(제목/본문/안내줄)를 셀프 리뷰 항목에 포함할 것

## 2026-07-09 — gate-effectiveness (G5)

- **교훈:** 규범 의미를 바꾸는 개정의 스윕은 skills/ 외 사용자 문서(README·commands)까지 포함할 것. 스윕 사각은 최종 리뷰가 유일한 망이라는 conversation-ux 교훈의 확장판

### 조사 보고의 집계 수치는 표 행 실측과 대조할 것
- **문제:** 심사 보고가 "유지 22건"이라며 게이트 23개를 나열 — 오프바이원이 설계 문서 §4 집계로 승계됐고 구현 태스크가 표를 실측하다 발견
- **교훈:** 수치 요약과 그 근거 목록(행)을 받으면 개수를 직접 세어 대조할 것. 표가 원본이고 집계는 파생이다

### 서브에이전트 main 오염은 경고만으로 안 막힌다
- **문제:** 워크트리 경고에도 haiku 구현자가 main 체크아웃에 선편집·선커밋 (커밋은 자가 정정, 작업 트리 잔재는 컨트롤러 복구) — 두 릴리스 연속 재발
- **교훈:** 디스패치 프롬프트 경고 강화(branch+pwd 이중 확인 지시)에 더해, 태스크 리뷰·최종 검증에 "main 무오염 확인"을 상시 항목으로 넣을 것 (이번에 T3부터 적용해 재발 차단 확인)

## 2026-07-13 — issue-lifecycle

### 스윕 예외 판정은 검증자 재량이 아니라 plan 문면을 따른다
- **문제:** plan이 "hotfix·핫픽스 어휘 예외 없음, 전부 제거"를 명시했는데, Task 8 스윕 검증자가 잔존 1건("긴급 핫픽스용")을 "구 독트린과 무관한 일반 형용사"로 오탐 제외 판정 — 최종 브랜치 리뷰가 스펙 위반으로 뒤집음
- **교훈:** 스윕의 제외 판정 기준은 검증 시점의 의미 해석이 아니라 plan의 명문 범위다. "예외 없음" 지시가 있으면 의미가 무관해 보여도 제거하거나, 예외로 남기려면 사용자 확인을 거칠 것. 두 검증자(태스크 vs 최종)의 판정이 충돌하면 plan 문면이 이긴다

### 삭제 태스크의 검증 grep은 "모드명"이 아니라 "절차 어휘"까지 포함할 것
- **문제:** Mode 2·3 삭제 후 검증 grep이 모드명 패턴만 검사해, 모드명 없는 절차 불릿 4개(주의 사항의 domain.md 통합·pending·feature 삭제 서술)가 사각에 남음 — 같은 파일 안 자기모순으로 최종 리뷰에서 적발
- **교훈:** 절 삭제의 잔존 검사는 삭제된 절의 고유 어휘(대상·동작 명사)로도 grep할 것 — 헤딩·모드명만으로는 본문이 재서술한 절차가 안 잡힌다

## 2026-07-13 — README 현행화

### 사용자 문서에 내부 용어를 컨텍스트 없이 노출하지 말 것
- **문제:** README 각주를 "진행 모드 3종(하나씩/한꺼번에/알아서)"으로 재작성 — 스킬 내부 모드명을 그대로 노출해, 처음 읽는 독자가 무엇의 모드인지 문맥 파악 불가. 사용자 피드백으로 적발
- **교훈:** 사용자 문서(README·가이드)는 상황(독자가 언제 겪는가) → 기본 동작 → 전환 발화와 그 결과 순으로 풀어 쓸 것. 내부 용어(모드명·규범명·게이트명)는 정의 없이 쓰지 않는다 — 내부 문서(skills/)의 어휘를 복사하면 이 함정에 빠진다

## 2026-07-14 — persona-improvement

### 페르소나 발언을 개행 없는 벽 텍스트로 출력하지 말 것
- **문제:** 라운드 결과 정리에서 페르소나 발언을 문단 구분 없이 한 덩어리로 출력 — "이런 식으로 개행 없이 글을 쓰면 사람이 읽을 수가 없어" 피드백
- **교훈:** 페르소나 발언은 논점이 바뀔 때마다 빈 줄로 문단을 나누고, 병렬 나열은 불릿으로 쓸 것. 서브에이전트 원문이 문단을 갖췄더라도 본체가 정리하며 뭉개지 말 것. 어휘 수준(일상어)만이 아니라 문단 구조가 가독성의 절반이다

### 서브에이전트 main 오염 3차 재발 — 프롬프트 경고로는 안 막힌다 (구조적 확인 필요)
- **문제:** T11 haiku 구현자가 branch+pwd 이중 확인 지시·"다르면 BLOCKED 보고" 지시에도 main 체크아웃에 선편집·선커밋(2d64b33) 후 워크트리에 재커밋 — 세 릴리스 연속 재발. 구현자 리포트는 브랜치 확인을 통과했다고 허위 보고
- **교훈:** 경량 모델 태스크는 프롬프트 경고를 신뢰하지 말 것. 컨트롤러가 각 태스크 승인 직후 `git -C [main]` log를 상시 확인하고(이번에 감지 성공), 최종 검증만이 아니라 태스크 단위로 main 무오염을 게이트할 것. 근본 대안 검토 후보: 경량 모델 디스패치에서 절대 경로 Edit만 허용하는 지시 대신 워크트리 격리 실행(isolation) 사용

## 2026-07-16 — grounded-output

### 원문 유지 규범 확정 당일, 본체가 중계 압축으로 재위반
- **문제:** 국면 3 판정 3건을 화면에 옮기며 "길다"는 이유로 가이드라인 목록을 슬래시(/) 한 줄로 압축 — 기존 TD 형식 규칙("줄바꿈 목록 필수, 슬래시 금지")과 같은 세션에서 확정한 "원문 유지·고쳐 쓰기 금지"를 동시에 위반. 사용자 정정으로 적발
- **교훈:** 길이 부담은 생산 규격(발언 상한)에서 해결하는 것이지 전달 단계의 압축으로 해결하지 않는다 — 중계가 길면 원문 그대로 두는 것이 규범이다. 줄바꿈 규칙은 산출물만이 아니라 중계·요약 문면에도 적용된다

### 주석 마커 이중화 금지
- **문제:** 본체 검증 주석에 `- ↳`로 불릿과 화살표를 겹쳐 사용 — 사용자가 어색함 지적
- **교훈:** 주석 줄 마커는 `↳` 하나만 쓴다 (불릿 없이 들여쓰기). 접두 태그(리스크 아이콘·[추측] 등)는 줄 첫머리에 모아 배치한다

### 법조문식 절 기호(§)를 사용자 대면 문면에서 쓰지 말 것
- **문제:** 대화 내내 "§7", "§8"로 절을 지칭 — 사용자가 기호의 뜻 자체를 물었고, "라벨 병기" 제안도 "일반 사람이 아는 표기라는 가정이 틀렸다"로 기각됨
- **교훈:** 사용자 대면 문면에서는 절 기호·전문 표기를 아예 쓰지 않는다 — "결정 규범 문서의 블라인드 라운드 절"처럼 문서 이름+절 이름을 말로 풀어 쓴다. 언어 규범에 조항 추가 대상 (grounded-output 개정 범위 편입)

### 규범 신설 스윕의 grep 스코프는 규범의 적용 범위 전체로 잡을 것
- **문제:** 절 기호(§) 금지 규범을 신설하면서 T0 grep 스코프를 4개 경로(brainstorming·context-handling·commands·README)로 한정 — 규범의 실제 적용 범위(사용자 대면 문면 전체)에 있던 completion-protocol.md·orchestrator SKILL.md의 사용자 대면 리터럴 2건이 사각에 남아 최종 브랜치 리뷰에서 적발
- **교훈:** 새 규범의 스윕 스코프는 "지금 고치는 파일 목록"이 아니라 "규범이 적용되는 표면 전체"(references/ 포함 skills/ 전부 + 사용자 문서)로 잡는다. 스코프를 좁힐 때는 좁힌 근거를 인벤토리에 명시한다
