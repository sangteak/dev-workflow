## 2026-03-19 — handoff-discovery-ux

### 스킬 지시문과 실제 출력의 괴리
- **문제:** context-handling에 출력 템플릿이 정의되어 있었지만, Claude가 실제로 따르지 않았음
- **원인:** orchestrator가 탐색 로직을 인라인으로 포함하여 Claude가 중간 과정을 보고함
- **교훈:** 출력 책임을 한 곳에 일원화하고, 금지 출력 패턴을 명시적으로 나열해야 함

### 완료된 작업의 잔존 HANDOFF 처리
- **문제:** `status: complete`인 작업의 HANDOFF.md가 남아있으면 "진행 중인 작업"으로 노출됨
- **교훈:** status 체크를 HANDOFF 존재 여부보다 우선순위로 두어야 함. 잔존 HANDOFF는 삭제를 제안할 것

### 설계 문서 테이블 중복
- **문제:** 라벨 규칙 테이블에서 `status: complete` 항목이 우선순위 이동 후 원래 위치에 중복 잔존
- **교훈:** 우선순위 변경 시 원래 위치의 항목을 반드시 제거할 것

## 2026-03-26 — skill-aliases

### 플러그인 commands/ 추가 후 Claude Code 재시작 필요
- **문제:** 플러그인에 commands/ 파일을 추가하고 push까지 완료했지만 콘솔에서 새 커맨드가 노출되지 않음
- **원인:** Claude Code가 세션 시작 시 플러그인 커맨드를 캐싱하므로, 런타임 중 추가된 파일은 감지하지 않음
- **교훈:** commands/ 디렉토리에 파일을 추가/수정한 후에는 사용자에게 Claude Code 재시작을 안내할 것

### 순차 실행 가능한 작업을 선택지로 제시하지 말 것 (2026-07-08)
- **문제:** 푸시/도그푸딩/브랜치 정리처럼 상호 배타적이지 않고 자연스러운 순서가 있는 작업들을 "어떻게 진행할까요?" 선택지로 제시함
- **피드백:** "다 해보자. 이걸 굳이 선택해야해?"
- **교훈:** 선택지는 실제 트레이드오프가 있어 사용자 판단이 필요할 때만. 순차 실행 가능한 후속 작업들은 순서를 선언하고 바로 진행할 것

## 2026-07-08 — skill-slimming (G4)

### 플랜의 수치 프록시 체크는 구조 상한을 먼저 계산할 것
- **문제:** plan.md에 "본문 40%+ 감소, 합계 ±5%" 체크를 넣었으나 실측은 24.8%/8.9%, +10~12% — 이동 가능 바이트의 구조적 상한과 고정 삽입 오버헤드를 플랜 작성 시 계산하지 않음
- **교훈:** 수치 프록시를 플랜에 넣기 전에 상한을 실측(또는 추정 근거 명시)할 것. 프록시가 깨지면 원래 증명하려던 직접 속성(byte-identical 이동 + 순증 전량 분류)을 검증해 대체 판정할 것 — 설계 문서의 "수치 목표는 비목표" 원칙이 플랜 세부 체크보다 우선한다

### 치환 지정은 대상 섹션의 경계를 명시할 것
- **문제:** "상태 정규화 전처리 섹션을 1줄로 치환" 지정이 섹션 경계를 명시하지 않아, 구현자가 인접한 "0. 경로 해소" 스텝(F3 추적 앵커 포함)까지 함께 삭제 — 태스크 리뷰가 포착해 복원
- **교훈:** 플랜에서 블록 치환을 지정할 때 시작/끝 라인 또는 경계 문장을 명시할 것. 크로스레퍼런스 인벤토리(F3)가 안전망으로 실제 작동했음 — 추적 앵커 등재의 가치 실증
