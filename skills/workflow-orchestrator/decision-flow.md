# 결정 요청의 그룹별 단계적 흐름 (Decision Flow SSOT)

> dev-workflow 플러그인의 모든 스킬에서 사용자 결정을 요청할 때 따르는 단일 SSOT.

## 1. 메타 원칙

결정이 여러 개일 때 사용자에게 한 번에 하나만 묻는다. 의존성이 발견되면 묶어서 종합 질문한다. 이전 결정과 충돌이 발견되면 재논의 대기열에 추가하고 현재 결정을 마무리한다.

## 2. 출력 형식 (B+D 하이브리드)

### 진행 중 매 응답 상단 — 한 줄 헤더 (B)

```
📋 N/M 확정 · 진행 중: [현재 결정명] · 대기열 K건
─────────────────────────────────────────────────
```

### 국면 시작 시 사전 브리핑 (F)

```
🗒️ 결정 항목 브리핑 (페르소나 토론 없이 사전 정리)

이번 국면 결정 항목 N개 — 의존성 루트부터 정렬:
1. [항목명] — [정렬 근거 한 줄] ← 루트
2. [항목명] — [정렬 근거 한 줄]
...

각 항목은 페르소나 토론 후 순서대로 한 번에 하나씩 결정합니다.
정렬에 이의가 있으면 알려주세요. 없으면 진행합니다.
```

### 결정 요청 시 풀 박스 (D)

```
┌─────────────────────────────────────────────────┐
│ 📌 결정 요청: [결정명]                            │
│                                                  │
│  1. [선택지]                                     │
│  2. [선택지]                                     │
│  0. 기타                                         │
└─────────────────────────────────────────────────┘
```

## 3. 행위 지침 (E — 진행 중 새 결정 발견 시)

모델은 자율 추론으로 다음을 수행한다:
1. 신규 항목과 기존 항목 간 의존성을 한 줄로 기술
2. 의존성 루트부터 순서대로 사용자에게 묻는다
3. 의존성 불명확/동률 시 입력 순서를 유지

## 4. 가드 규칙

### 임계값 8개

결정 항목 누적 8개 도달 시 → "이 국면을 두 라운드로 분할할까요?" 사용자에게 제안한다.
- 거절 시 한 줄 헤더 자동 압축: `📋 진행 N건 · 완료 K건 · 대기 M건`
- 풀 박스는 가까운 결정 3개만 표시 + `...더 N개`

### 재논의 대기열

충돌 발견 시 즉시 처리하지 않고 카운터(K건)만 헤더에 표시한다. 현재 국면의 모든 결정 완료 후 일괄 노출:

```
┌─────────────────────────────────────────────────┐
│ 🔄 재논의 대기열 K건                              │
│  1. 모두 재논의                                  │
│  2. 일부만 (항목 지정)                           │
│  3. 모두 기각                                    │
└─────────────────────────────────────────────────┘
```

국면 전환 시 대기열이 비어있지 않으면 phase 파일 작성 직전 강제 노출한다.

### 클러스터 그룹핑 금지

클러스터 그룹핑은 도입하지 않는다 (사후 합리화 위험).

## 5. 페르소나 피드백 출력 형식 (brainstorming 통합)

### 에이전트 태그 (Enhanced Mode)

Enhanced Mode에서 서브에이전트로 실행된 페르소나에 에이전트명을 표시한다:
- `[Socratic]` — Socratic Interviewer 사고 제약으로 생성된 발언
- `[Contrarian]` — Contrarian 사고 제약으로 생성된 발언 (비판 담당)
- Standalone Mode 또는 서브에이전트가 아닌 발언에는 태그를 붙이지 않는다
- 라운드 헤더에도 현재 사용 중인 에이전트를 표시한다

### 결정 요청 분리 규칙

- 합의 시: `**합의:**` + 결론 요약
- 미합의 시: `**미합의:**` + 쟁점 요약
- 결정 항목이 없으면 (페르소나 간 합의 도출된 경우) 결정 요청 블록을 생략한다
- "이 두 가지에 대해" 같은 대명사 참조를 사용하지 않는다 — 각 결정 항목은 명시적 이름으로 가리킨다

## 6. 적용 범위 선언

본 SSOT는 dev-workflow 플러그인의 모든 스킬에서 사용자 결정을 요청할 때 따르는 단일 규범이다. 핵심 하이브리드 패턴은 **B+D 하이브리드** (출력 형식: 한 줄 헤더 + 풀 박스) 와 **F+E 하이브리드** (동작: 사전 브리핑 + 행위 지침) 이다.

- 결정이 0개면 결정 박스를 생략한다 (합의 도출됨)
- 결정이 1개면 풀 박스(D)로 즉시 요청한다 (사전 브리핑 생략 가능)
- 결정이 2개 이상이면 사전 브리핑(F) + 의존성 루트부터 한 번에 하나씩 요청한다
- 진행 중 새 결정 발견 시 행위 지침(E)을 적용한다
- 결정 8개+ 도달 시 가드 규칙의 임계값 처리를 따른다
- 충돌 발견 시 재논의 대기열에 추가한다

### 적용 예외

- 결정이 없는 정보 제공/조사 출력에는 적용하지 않는다
- 이미 자연스럽게 단계적인 흐름(예: document-consolidation의 매 단계 확인)은 풀 박스(D) 형식만 따르면 충분하다
