# Development Principles

모든 단계에 공통으로 적용되는 개발 철학이다.
PLAN 페르소나 피드백루프와 Superpowers 서브에이전트 모두 이 원칙을 따른다.

---

## 단순함 우선

- 모든 변경은 최대한 단순하게 유지한다
- 건드리는 코드는 최소화한다
- 꼭 필요한 부분만 수정하고 새로운 버그를 만들지 않는다
- 오버엔지니어링 금지 — 단순하고 명백한 수정엔 과도한 설계를 적용하지 않는다

---

## 근본 원인 추구

- 임시방편(workaround)을 사용하지 않는다
- 버그 리포트를 받으면 직접 로그/에러/실패 테스트를 찾아 해결한다
- 하나하나 물어보지 않고 자율적으로 처리한다
- 시니어 엔지니어 기준으로 판단한다: "시니어 엔지니어가 이 결과물을 승인할까?"

---

## 우아함 추구 (균형 있게)

- 사소하지 않은 변경 전: "더 우아한 방법이 없을까?" 자문한다
- 수정이 어설프게 느껴진다면: 현재 알고 있는 모든 것을 바탕으로 우아한 해결책을 다시 구현한다
- 결과물을 내놓기 전에 반드시 스스로 검토한다
- 단순한 수정에는 이 과정을 생략한다 — 균형이 중요하다

---

## 자율적 버그 수정

- 버그 리포트 수신 시: 사용자에게 하나하나 확인하지 않고 직접 해결한다
- 실패한 CI 테스트는 별도 지시 없이 직접 찾아서 수정한다
- 사용자가 컨텍스트를 전환할 필요가 없도록 처리한다

---

## Self-Improvement Loop

세션 간 학습을 누적하여 동일한 실수를 반복하지 않는다.

### 교훈 기록 규칙

사용자로부터 수정을 받을 때마다:

1. 어떤 패턴의 실수였는지 분석한다
2. `tasks/lessons.md` 에 아래 형식으로 기록한다:

```markdown
## [날짜] [카테고리]
- 실수: [무엇을 잘못했는가]
- 원인: [왜 발생했는가]
- 규칙: [앞으로 어떻게 할 것인가]
```

3. 같은 패턴의 실수가 반복될 경우 해당 규칙을 강화한다

### 세션 시작 시 교훈 검토

`tasks/lessons.md` 가 존재할 경우 Session Start Protocol 실행 후
반드시 파일을 읽고 주요 교훈을 숙지한 뒤 작업을 시작한다.
교훈 목록을 사용자에게 별도로 출력하지 않는다 — 내부적으로 참고만 한다.

### 카테고리 예시

- ARCHITECTURE: 설계 결정 관련 실수
- IMPLEMENTATION: 구현 방식 관련 실수
- TESTING: 테스트 누락 또는 잘못된 검증
- COMMUNICATION: 사용자 의도 오해
- WORKFLOW: 워크플로우 절차 위반

## 경로 해소 규칙 (Path Resolution)

모든 스킬에서 `docs/design/` 디렉토리를 탐색할 때 적용하는 공통 규칙이다.

### 탐색 시 (Search/Read)

`docs/design/` 경로를 탐색하기 전, 실제 경로를 확인한다:

1. 아래 명령을 실행한다:
   ```bash
   find . -maxdepth 2 -iname "docs" -type d
   ```
2. 결과에서 `design` 하위 디렉토리가 존재하는지 확인한다
3. 발견된 실제 경로를 이후 모든 탐색(glob, 존재 확인, 스캔)에 사용한다
4. `docs` 디렉토리를 찾지 못한 경우 → "docs/design/ 디렉토리를 찾을 수 없습니다" 안내

### 생성 시 (Create/Write)

파일 및 디렉토리 생성은 항상 표준 경로 `docs/design/`을 사용한다.
경로 해소 결과와 무관하게 소문자 표준을 강제한다.

### 실행 빈도

경로 해소는 **스킬 실행당 1회**만 수행한다.
해소된 경로를 해당 스킬의 전체 실행 동안 재사용한다.

## 프로젝트 규칙 (`.claude/rules/*.md`)

dev-workflow는 프로젝트별 코딩/커밋/리뷰 규칙을 `.claude/rules/*.md`로 분리하여 도메인 분리 + 단계별 자동 주입을 지원한다.

- **규칙 작성:** `docs/templates/rule-template.md` 참조
- **CLAUDE.md 분리 가이드:** `docs/guides/migrate-claude-md.md` 참조
- **새 규칙 생성:** `/dev-workflow:add-rule` 명령으로 인터랙티브 생성

자세한 메커니즘은 `skills/rules-injection/SKILL.md` 참조. 규칙 파일이 없는 프로젝트는 변경 없이 동일하게 동작한다 (후방 호환).

## 자동 결정 안전망 (공통 포맷)

게이트(사용자 확인 지점)를 생략·자동화한 자리에는 아래 3요소 중 최소 1개를 동반한다 — 안전망 없는 단순 생략 금지:

- **사후 고지 1줄**: 자동으로 처리된 결정을 답을 요구하지 않는 한 줄로 알린다. 표준형: `[이모지] [무엇]을 자동 [처리]했습니다 — [요지]. (수정하려면 말씀하세요)`
- **기록**: 자동 결정의 내용을 산출물(phase 파일·회계 목록·git 이력)에 남긴다
- **소급 수정 경로**: 사용자가 이의를 제기하면 해당 확인을 소급 제시하거나 결정을 되돌린다 — 재논의 대기열을 거치지 않고 즉시 처리한다

이 포맷은 merge-to-domain의 "자동 수정 회계" 패턴을 일반화한 것이다. 각 스킬은 안전망을 개별 서술하지 않고 본 절을 참조한다.

## 게이트 신설 체크리스트 (3단 결정 트리)

새 사용자 확인 지점을 만들기 전에, 그리고 기존 확인을 제거·자동화하기 전에 아래 3문항에 순서대로 답한다:

1. 이 확인이 없으면 되돌릴 수 없는 결과가 발생하는가? → Yes = 게이트를 둔다 (비가역 승인 — 어떤 진행 모드에서도 생략 불가)
2. 이 확인의 답이 사실상 항상 같은가? → Yes = 게이트 대신 위 "자동 결정 안전망"을 쓴다
3. 생략해도 사용자가 사후에 그 사실을 알 방법이 있는가? → No = 안전망부터 설계한다 — 설계할 수 없으면 게이트를 둔다
