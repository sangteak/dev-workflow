# Development Principles

모든 단계에 공통으로 적용되는 개발 철학이다.
PLAN 페르소나 피드백루프와 Superpowers 서브에이전트 모두 이 원칙을 따른다.

---

## 단순함 우선

- 모든 변경은 최대한 단순하게 유지한다
- 건드리는 코드는 최소화한다
- 꼭 필요한 부분만 수정하고 새로운 버그를 만들지 않는다
- 오버엔지니어링 금지 — 단순하고 명백한 수정엔 과도한 설계를 적용하지 않는다

---

## 근본 원인 추구

- 임시방편(workaround)을 사용하지 않는다
- 버그 리포트를 받으면 직접 로그/에러/실패 테스트를 찾아 해결한다
- 하나하나 물어보지 않고 자율적으로 처리한다
- 시니어 엔지니어 기준으로 판단한다: "시니어 엔지니어가 이 결과물을 승인할까?"

---

## 우아함 추구 (균형 있게)

- 사소하지 않은 변경 전: "더 우아한 방법이 없을까?" 자문한다
- 수정이 어설프게 느껴진다면: 현재 알고 있는 모든 것을 바탕으로 우아한 해결책을 다시 구현한다
- 결과물을 내놓기 전에 반드시 스스로 검토한다
- 단순한 수정에는 이 과정을 생략한다 — 균형이 중요하다

---

## 자율적 버그 수정

- 버그 리포트 수신 시: 사용자에게 하나하나 확인하지 않고 직접 해결한다
- 실패한 CI 테스트는 별도 지시 없이 직접 찾아서 수정한다
- 사용자가 컨텍스트를 전환할 필요가 없도록 처리한다

---

## Self-Improvement Loop

세션 간 학습을 누적하여 동일한 실수를 반복하지 않는다.

### 교훈 기록 규칙

사용자로부터 수정을 받을 때마다:

1. 어떤 패턴의 실수였는지 분석한다
2. `tasks/lessons.md` 에 아래 형식으로 기록한다:

```markdown
## [날짜] [카테고리]
- 실수: [무엇을 잘못했는가]
- 원인: [왜 발생했는가]
- 규칙: [앞으로 어떻게 할 것인가]
```

3. 같은 패턴의 실수가 반복될 경우 해당 규칙을 강화한다

### 세션 시작 시 교훈 검토

`tasks/lessons.md` 가 존재할 경우 Session Start Protocol 실행 후
반드시 파일을 읽고 주요 교훈을 숙지한 뒤 작업을 시작한다.
교훈 목록을 사용자에게 별도로 출력하지 않는다 — 내부적으로 참고만 한다.

### 카테고리 예시

- ARCHITECTURE: 설계 결정 관련 실수
- IMPLEMENTATION: 구현 방식 관련 실수
- TESTING: 테스트 누락 또는 잘못된 검증
- COMMUNICATION: 사용자 의도 오해
- WORKFLOW: 워크플로우 절차 위반
