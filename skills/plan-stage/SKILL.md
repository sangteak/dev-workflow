# PLAN Stage Protocol

PLAN 단계는 반드시 아래 순서로 진행한다. 순서를 건너뛰지 않는다.

**결정 요청 형식 (SSOT):** PLAN 전체(Step 1~4)의 사용자 결정 요청은 `decision-flow.md`를 따른다. 첫 서브에이전트 투입 전에 반드시 해당 파일을 Read한다 — 컨텍스트 압축(/compact) 복원 후에는 재적용한다 — 본 스킬 디렉토리 기준 `../workflow-orchestrator/decision-flow.md`, 미확보 시 경로 해소·Read 실패 폴백은 workflow-orchestrator "Output Format Rules"의 SSOT 규칙과 동일. Step 1~4 전체를 하나의 '국면'으로 간주하며, 재논의 대기열의 최종 안전망 노출 시점은 Superpowers writing-plans 핸드오프 직전이다.

## Step 1: 브레인스토밍 문서 분석

**동일 세션 내 브레인스토밍 직후 진입하는 경우:**
설계 문서 내용을 이미 알고 있으므로 파일을 다시 읽지 않는다.

```
── PLAN 단계 진입 ────────────────────────────────────
브레인스토밍에서 생성된 설계 문서를 기반으로 진행합니다.
──────────────────────────────────────────────────────
```

**신규 세션에서 진입하는 경우:**
경로 해소(`find . -maxdepth 2 -iname "docs" -type d`) 후 `[해소된 경로]/[카테고리]/[기능명]/` 의 설계 문서를 읽고 추출 후 사용자 확인
(상세: development-principles "경로 해소 규칙" 참조):

```
── 브레인스토밍 문서 분석 결과 ──────────────────────

✅ 확정 요구사항:
[목록]

⚠️ 미결 사항:
[목록]

🔧 기술 가이드라인 (TD 코멘트):
[목록 또는 "없음"]

위 내용을 바탕으로 PLAN 단계를 진행합니다. 계속할까요?
─────────────────────────────────────────────────────
```

사용자 확인 없이 다음으로 넘어가지 않는다.

**OPEN_QUESTIONS 처리 규칙:**

`OPEN_QUESTIONS` 항목이 하나라도 존재할 경우 Step 2 진입 전에 반드시 처리한다:

```
── 미결 사항 확인 ─────────────────────────────────────

아래 항목이 아직 결정되지 않았습니다 — 의존성 루트부터 정렬:

1. ❓ [미결 사항 1] — [정렬 근거 한 줄] ← 루트
2. ❓ [미결 사항 2] — [정렬 근거 한 줄]

첫 번째 항목부터 하나씩 여쭙습니다. (정렬 이의는 알려주세요 · 한 번에 보시려면 "다 보여줘" · 위임은 "알아서")
──────────────────────────────────────────────────────
```

모든 OPEN_QUESTIONS 해소 후 Step 2로 진행한다.
해소된 결정은 DECISIONS에 추가하고 OPEN_QUESTIONS에서 제거한 것으로 간주한다.

## 설계 문서 크로스레퍼런스

PLAN 진행 중 사용자가 기존에 구현 완료된 설계 문서를 참조하려 할 때,

## Step 2: Feasibility Assessment

페르소나 3인이 각 요구사항의 구현 가능 여부를 판정한다. **판정은 상시 블라인드로 실행한다** — 3인을 병렬 서브에이전트로 동시 투입하며(상호 미노출), 절차 원본은 decision-flow §7 「블라인드 라운드 공통 절차」다.

판정 프롬프트 구성 (사고 제약 원문 미포함 — 판정용 원문이 존재하지 않음을 설계로 확정): [도메인 페르소나 정의] + [판정 기준 ✅/⚠️/🚫] + [발췌: 설계 문서 요구사항 + Architect 구조 분석 결과] + ["--- 언어 규범 ---" 구획] + [정형 형식: 요구사항별 판정 + 근거 1~2문장] + [전제 반박 의무 1개] + [출력 제약: 결정 요청 문구 금지].

판정 취합은 본 절의 판정 출력 형식(로컬 템플릿)을 따른다 — §7 취합의 원문 나열은 의견 수집 라운드에 적용된다.

**📎 Architect 구조 분석:**
Feasibility Assessment 시작 전, Architect 에이전트를 서브에이전트로 실행하여 구조적 리스크를 사전 분석한다.
에이전트 역할 정의: 본 스킬의 `references/agent-roles.md`에서 해당 역할 블록을 Read하여 사용한다 (외부 탐색 없음. Read 실패 시 최소 요건: Architect=구조 리스크 4관점 분석, Researcher=판정 근거의 사실 확인·우회 경로 조사)

```
아래 에이전트 역할 정의를 읽고 그 역할을 수행하라.
형식이 예상과 다르더라도 DO/DON'T 섹션을 찾아 따르라.

[rules-injection 결과: applies-to: plan 매칭 규칙 본문 — `all` 규칙은 SessionStart에서만 노출되므로 단계별 첨부에는 제외됨]

--- 언어 규범 ---
[decision-flow.md 3부 「언어 규범」 펜스 전문 — 공통 조립 규칙 참조]

--- 설계 문서 ---
[확정된 설계 문서 전문]

--- 현재 코드베이스 구조 ---
[프로젝트 디렉토리 구조 요약]

아래 관점에서 구조적 리스크를 분석하라:
1. 기존 구조와의 결합도 (coupling)
2. 책임 분리가 적절한가 (responsibility)
3. 추상화 수준이 맞는가 (abstraction level)

```

Architect 분석 결과를 페르소나 판정의 참고 자료로 제공한다:
```
── 🏗️ 구조 분석 [Architect] ──────────────────────────
결합도: [낮음/중간/높음] — [설명]
책임 분리: [적절/주의] — [설명]
구조 변경 범위: [최소/보통/광범위] — [설명]
권장사항: [한 줄 요약]
──────────────────────────────────────────────────────
```

**판정 기준:**
- ✅ FEASIBLE: 현재 기술 스택으로 구현 가능, 리스크 낮음
- ⚠️ CAUTION: 구현 가능하나 난이도 높음, 접근 방향 논의 필요
- 🚫 RENEGOTIATE: 구현 불가 또는 비용 과다, 재협의 필요

**판정 출력 형식:**
```
── Feasibility Assessment ────────────────────────────

[요구사항명]
  [페르소나A]: [판정 + 근거]
  [페르소나B]: [판정 + 근거]
  [페르소나C]: [판정 + 근거]
  → 종합 판정: ✅ / ⚠️ / 🚫

── 판정 요약 ──────────────────────────────────────────
✅ FEASIBLE:     [n개]
⚠️ CAUTION:      [n개]
🚫 RENEGOTIATE:  [n개]
──────────────────────────────────────────────────────
```

```
--- 프로젝트 컨텍스트 ---

아래를 조사하라:
```

```
[CAUTION/RENEGOTIATE 항목명]:

  우회 가능성: [있음/없음] — [설명]
  유사 패턴: [발견/미발견] — [설명]
  결론: [FEASIBLE로 상향 가능 / CAUTION 유지 / RENEGOTIATE 유지]
──────────────────────────────────────────────────────
```

Researcher 조사 결과로 판정이 변경될 수 있다:
- RENEGOTIATE → CAUTION 또는 FEASIBLE로 상향: 근거와 함께 사용자에게 제안
- CAUTION → FEASIBLE로 상향: 근거와 함께 사용자에게 제안
- 판정 유지: 아래 선택지로 진행

### 재협의 선택지 (🚫 RENEGOTIATE 항목)

```
── 재협의 필요 항목 ───────────────────────────────────

🚫 [요구사항명]
이유: [기술적 근거]

어떻게 진행할까요?

1. 브레인스토밍 문서를 수정하고 다시 검토 — 고르면: 설계로 돌아가 해당 요구사항을 고칩니다
2. 해당 요구사항을 스코프에서 제외하고 진행 — 고르면: 이번 구현에서 빠지고 문서에 제외 기록이 남습니다
3. CAUTION으로 하향 조정하여 리스크를 감수하고 진행 — 고르면: 경고 상태로 구현 대상에 남습니다
4. 대안 접근 방향을 제안받고 논의 — 고르면: 페르소나가 대안을 만들어 다시 여쭙니다
──────────────────────────────────────────────────────
```

사용자 응답 없이 Step 4로 넘어가지 않는다.

## Step 4: 설계 방향 수립

FEASIBLE 및 CAUTION 항목에 대해 구현 방향을 설계한다.

CAUTION 항목은 반드시 Persona Feedback Loop를 통해 접근 방향 합의 후 진행한다:

```
── Persona Feedback ──────────────────────────────────

[페르소나A 이모지] [이름]:
[의견 - 2~4문장]

[페르소나B 이모지] [이름]:
[반론 또는 보완 - 2~4문장]

[페르소나C 이모지] [이름]:
[추가 관점 - 2~4문장]

── 합의 ───────────────────────────────────────────────
[세 관점을 종합한 최종 결론 및 권장 방향]
──────────────────────────────────────────────────────
```

**루프 규칙:**
- PLAN 단계의 주요 결정 전 피드백루프를 생략하지 않는다 — 단 페르소나 초기 의견이 전원 일치하면 합의 블록 기록으로 라운드를 종료한다 (반박·이견이 있을 때만 후속 라운드를 연다)
  전원 일치라도 판정 근거가 미실측 수치·상한 가정에 기대면, 합의 블록에 실측 값 또는 추정 근거 1줄을 기록해야 종료할 수 있다 (동일 컨텍스트를 공유한 페르소나의 일치는 독립 검증이 아니다).
- 페르소나 간 충돌 시 충돌 내용과 해결 근거를 명시한다
- 합의 후 사용자 확인을 받고 다음으로 진행한다
- 각 페르소나의 발언이 중복되지 않도록 한다

설계 방향 수립 완료 후 Superpowers `writing-plans` 를 실행한다.

**plan 파일 저장 경로:** `docs/design/[카테고리]/[기능명]/plan.md`
Superpowers가 다른 경로에 생성할 경우, 위 경로로 이동한다.

## 재계획 진입 (명시 라우팅 전용)

크리티컬 이슈의 "제자리 재계획" 경로에서 workflow-orchestrator가 명시적으로 invoke한다 (orchestrator 「Issue Lifecycle」 참조). **Stage Detection으로 진입하지 않는다** — plan.md가 이미 존재하므로 자동 감지가 불가한 경로다.

전제: 설계 문서 §4/§6/§10 갱신이 완료되고 사용자 리뷰를 통과한 상태.

절차 (Step 1~4 전체를 재실행하지 않는다 — 변경분만):

1. **변경분 Feasibility** — 갱신된 설계 부분에 한정해 페르소나 3인 판정 (Step 2의 판정 기준·출력 형식 재사용, 📎 Architect 구조 분석 재사용 가능)
2. **기존 plan 차분 분류** — 기존 plan.md의 태스크를 3분류해 사용자 확인을 받는다: **유효**(완료됐거나 그대로 진행) / **폐기**(바뀐 설계와 충돌 — 사유 기록) / **신규 필요**. 완료 태스크의 워크트리 커밋과 문서가 어긋나지 않게 한다
3. **Superpowers writing-plans 재호출** — 차분 분류를 컨텍스트로 전달해 plan.md를 재작성한다 (유효 태스크는 완료 표시 유지)

국면 규칙: 재계획 진입 1회가 하나의 국면이다 (decision-flow.md 국면 대응표 참조).
