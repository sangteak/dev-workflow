# PLAN Stage Protocol

PLAN 단계는 반드시 아래 순서로 진행한다. 순서를 건너뛰지 않는다.

---

## Step 1: 브레인스토밍 문서 분석

**동일 세션 내 브레인스토밍 직후 진입하는 경우:**
설계 문서 내용을 이미 알고 있으므로 파일을 다시 읽지 않는다.

```
── PLAN 단계 진입 ────────────────────────────────────
브레인스토밍에서 생성된 설계 문서를 기반으로 진행합니다.
──────────────────────────────────────────────────────
```

**신규 세션에서 진입하는 경우:**
경로 해소(`find . -maxdepth 2 -iname "docs" -type d`) 후 `[해소된 경로]/[카테고리]/[기능명]/` 의 설계 문서를 읽고 추출 후 사용자 확인
(상세: development-principles "경로 해소 규칙" 참조):

```
── 브레인스토밍 문서 분석 결과 ──────────────────────

✅ 확정 요구사항:
[목록]

⚠️ 미결 사항:
[목록]

🔧 기술 가이드라인 (TD 코멘트):
[목록 또는 "없음"]

위 내용을 바탕으로 PLAN 단계를 진행합니다. 계속할까요?
─────────────────────────────────────────────────────
```

사용자 확인 없이 다음으로 넘어가지 않는다.

**OPEN_QUESTIONS 처리 규칙:**

`OPEN_QUESTIONS` 항목이 하나라도 존재할 경우 Step 2 진입 전에 반드시 처리한다:

```
── 미결 사항 확인 ─────────────────────────────────────

아래 항목이 아직 결정되지 않았습니다. 하나씩 확인하겠습니다.

❓ [미결 사항 1]: [내용]
❓ [미결 사항 2]: [내용]

각 항목에 대해 결정을 내려주시면 진행하겠습니다.
──────────────────────────────────────────────────────
```

모든 OPEN_QUESTIONS 해소 후 Step 2로 진행한다.
해소된 결정은 DECISIONS에 추가하고 OPEN_QUESTIONS에서 제거한 것으로 간주한다.

---

## 설계 문서 크로스레퍼런스

PLAN 진행 중 사용자가 기존에 구현 완료된 설계 문서를 참조하려 할 때,
`dev-workflow:design-doc-index` 스킬을 invoke한다.

**트리거 감지:**
- "~~ 참고해서", "~~ 기반으로", "~~ 고려해서" → 전체 로드
- "어떤 기능 있어?", "목록 보여줘", "관련된 거 뭐 있어?" → 색인 모드
- 기존 설계 문서를 언급하는 모든 자연어 발화

invoke 후 로드된 컨텍스트는 Feasibility Assessment 및 설계 방향 수립에서 활용한다.

---

## Step 2: Feasibility Assessment

페르소나 3인이 각 요구사항의 구현 가능 여부를 판정한다.

**📎 Enhanced Mode — Architect 구조 분석:**
Feasibility Assessment 시작 전, Architect 에이전트를 서브에이전트로 실행하여 구조적 리스크를 사전 분석한다:

```
아래 에이전트 역할 정의를 읽고 그 역할을 수행하라.
형식이 예상과 다르더라도 DO/DON'T 섹션을 찾아 따르라.

--- 에이전트 역할 정의 ---
[Ouroboros agents/architect.md 전문]

--- 설계 문서 ---
[확정된 설계 문서 전문]

--- 현재 코드베이스 구조 ---
[프로젝트 디렉토리 구조 요약]

아래 관점에서 구조적 리스크를 분석하라:
1. 기존 구조와의 결합도 (coupling)
2. 책임 분리가 적절한가 (responsibility)
3. 추상화 수준이 맞는가 (abstraction level)
4. 최소한의 구조 변경으로 구현 가능한가
```

Architect 분석 결과를 페르소나 판정의 참고 자료로 제공한다:
```
── 🏗️ 구조 분석 [Architect] ──────────────────────────
결합도: [낮음/중간/높음] — [설명]
책임 분리: [적절/주의] — [설명]
구조 변경 범위: [최소/보통/광범위] — [설명]
권장사항: [한 줄 요약]
──────────────────────────────────────────────────────
```

Standalone Mode에서는 Architect 분석을 생략하고, TD 페르소나가 구조적 관점을 포함하여 발언한다.

**판정 기준:**
- ✅ FEASIBLE: 현재 기술 스택으로 구현 가능, 리스크 낮음
- ⚠️ CAUTION: 구현 가능하나 난이도 높음, 접근 방향 논의 필요
- 🚫 RENEGOTIATE: 구현 불가 또는 비용 과다, 재협의 필요

**판정 출력 형식:**
```
── Feasibility Assessment ────────────────────────────

[요구사항명]
  [페르소나A]: [판정 + 근거]
  [페르소나B]: [판정 + 근거]
  [페르소나C]: [판정 + 근거]
  → 종합 판정: ✅ / ⚠️ / 🚫

── 판정 요약 ──────────────────────────────────────────
✅ FEASIBLE:     [n개]
⚠️ CAUTION:      [n개]
🚫 RENEGOTIATE:  [n개]
──────────────────────────────────────────────────────
```

---

## Step 3: 재협의 플래그 처리

🚫 RENEGOTIATE 또는 ⚠️ CAUTION 항목 존재 시 처리한다.

### 📎 Enhanced Mode — Researcher 증거 조사

RENEGOTIATE 또는 CAUTION 항목이 있을 때, 사용자에게 선택지를 제시하기 전에 Researcher 에이전트로 실제 제약 조건을 조사한다:

```
아래 에이전트 역할 정의를 읽고 그 역할을 수행하라.
형식이 예상과 다르더라도 DO/DON'T 섹션을 찾아 따르라.

--- 에이전트 역할 정의 ---
[Ouroboros agents/researcher.md 전문]

--- 조사 대상 ---
[CAUTION/RENEGOTIATE 항목과 판정 근거]

--- 프로젝트 컨텍스트 ---
[코드베이스 구조, 기존 패턴, 관련 문서]

아래를 조사하라:
1. 판정 근거가 사실인지 (실제 기술적 제약 확인)
2. 우회 가능한 경로가 있는지
3. 유사 패턴이 코드베이스에 이미 존재하는지
4. 추가 정보가 필요한 항목은 무엇인지
```

Researcher 결과를 사용자 선택지와 함께 제시한다:
```
── 🔍 제약 조건 조사 결과 [Researcher] ───────────────
[CAUTION/RENEGOTIATE 항목명]:
  확인된 사실: [조사 결과]
  우회 가능성: [있음/없음] — [설명]
  유사 패턴: [발견/미발견] — [설명]
  결론: [FEASIBLE로 상향 가능 / CAUTION 유지 / RENEGOTIATE 유지]
──────────────────────────────────────────────────────
```

Researcher 조사 결과로 판정이 변경될 수 있다:
- RENEGOTIATE → CAUTION 또는 FEASIBLE로 상향: 근거와 함께 사용자에게 제안
- CAUTION → FEASIBLE로 상향: 근거와 함께 사용자에게 제안
- 판정 유지: 아래 선택지로 진행

Standalone Mode에서는 Researcher 조사를 생략하고 즉시 선택지를 제시한다.

### 재협의 선택지 (🚫 RENEGOTIATE 항목)

```
── 재협의 필요 항목 ───────────────────────────────────

🚫 [요구사항명]
이유: [기술적 근거]

어떻게 진행할까요?

1. 브레인스토밍 문서를 수정하고 다시 검토
2. 해당 요구사항을 스코프에서 제외하고 진행
3. CAUTION으로 하향 조정하여 리스크를 감수하고 진행
4. 대안 접근 방향을 제안받고 논의
──────────────────────────────────────────────────────
```

사용자 응답 없이 Step 4로 넘어가지 않는다.

---

## Step 4: 설계 방향 수립

FEASIBLE 및 CAUTION 항목에 대해 구현 방향을 설계한다.

CAUTION 항목은 반드시 Persona Feedback Loop를 통해 접근 방향 합의 후 진행한다:

```
── Persona Feedback ──────────────────────────────────

[페르소나A 이모지] [이름]:
[의견 - 2~4문장]

[페르소나B 이모지] [이름]:
[반론 또는 보완 - 2~4문장]

[페르소나C 이모지] [이름]:
[추가 관점 - 2~4문장]

── 합의 ───────────────────────────────────────────────
[세 관점을 종합한 최종 결론 및 권장 방향]
──────────────────────────────────────────────────────
```

**루프 규칙:**
- PLAN 단계의 주요 결정 전 피드백루프를 절대 생략하지 않는다
- 페르소나 간 충돌 시 충돌 내용과 해결 근거를 명시한다
- 합의 후 사용자 확인을 받고 다음으로 진행한다
- 각 페르소나의 발언이 중복되지 않도록 한다

설계 방향 수립 완료 후 Superpowers `writing-plans` 를 실행한다.

**plan 파일 저장 경로:** `docs/design/[카테고리]/[기능명]/plan.md`
Superpowers가 다른 경로에 생성할 경우, 위 경로로 이동한다.
