# Design Document Index

브레인스토밍/플랜 중 기존에 구현 완료된 설계 문서를 자연어로 참조하여 컨텍스트에 로드한다.
별도 인덱스 파일을 관리하지 않으며, 호출 시 `[해소된 경로]`를 실시간 스캔한다.

---

## 상태 정규화 전처리

색인/로드 실행 전에 대상 문서의 상태값을 정규화한다. 이 전처리는 필터링보다 먼저 실행한다.

**절차:**
0. 경로 해소: `find . -maxdepth 2 -iname "docs" -type d` 실행 후 design 하위 확인
   (상세: development-principles "경로 해소 규칙" 참조)
1. `[해소된 경로]/[카테고리]/[기능명]/[기능명].md` 패턴의 모든 파일을 탐색한다
1-1. domain.md(category 직속 .md)는 상태 정규화 대상에서 제외한다
2. 각 파일의 프론트매터 `status` 값을 읽는다
3. `complete` → 변경 없음
4. `completed` → 해당 파일의 프론트매터를 `complete`로 수정한다
5. 그 외 값 또는 `status` 없음 → 스킵

수정 시 사용자에게 별도 안내하지 않는다 (silent fix).

---

## 색인 대상

구현 완료된 설계 문서를 대상으로 하며, domain.md를 우선 참조한다.

### 우선순위

1. **domain.md** (1순위) — category 디렉토리 직속 .md 파일
   - 탐색 경로: `[해소된 경로]/[카테고리]/*.md`
   - 판별: category 디렉토리 직속에 위치한 .md 파일
   - 프론트매터 status 필터링 없음 (domain.md는 항상 활성 상태)

2. **feature 설계 문서** (2순위) — domain에 통합되지 않은 독립 설계 문서
   - 탐색 경로: `[해소된 경로]/[카테고리]/[기능명]/[기능명].md`
   - 프론트매터 `status: complete` 필터링 유지
   - domain.md가 존재하는 시스템의 feature 문서는 domain.md에 통합되어 삭제되므로,
     여기에 남아있는 feature 문서는 domain 미통합 또는 독립 기능

## 트리거 조건

brainstorming 또는 plan-stage 스킬에서 사용자 발화를 감지하여 이 스킬을 invoke한다.
사용자가 요청할 때만 동작한다 (세션 시작 시 자동 안내 없음).

### 모드 자동 판단

사용자 발화 패턴으로 색인 모드와 전체 로드 모드를 자동 판단한다.
확실한 패턴만 자동 판단하고, 나머지는 사용자에게 확인한다.

**색인 모드 트리거 (한국어):**
- "어떤 기능 있어?", "어떤 시스템이 있어?"
- "목록 보여줘", "리스트 보여줘"
- "관련된 거 뭐 있어?", "참고할 만한 거 있어?"
- "구현된 기능 뭐 있어?"

**색인 모드 트리거 (영어):**
- "what features exist?", "what systems are there?"
- "show me the list", "list completed features"
- "what's available for reference?"

**전체 로드 트리거 (한국어):**
- "~~ 참고해서", "~~ 참고하고"
- "~~ 기반으로", "~~ 바탕으로"
- "~~ 고려해서", "~~ 고려하고"
- "~~ 보고 설계하자", "~~ 읽어줘"

**전체 로드 트리거 (영어):**
- "referencing ~~", "based on ~~"
- "considering ~~", "taking ~~ into account"
- "load ~~", "read ~~"

**모호한 경우:**
"전체 문서를 로드할까요, 목록만 볼까요?" 사용자에게 확인한다.

## 색인 모드

기존에 구현 완료된 설계 문서의 메타데이터 목록을 제시한다.

### 절차

1. `[해소된 경로]` 하위의 모든 설계 문서를 glob 탐색한다 (domain.md + feature 문서)
2. domain.md는 필터링 없이 포함, feature 문서는 `status: complete` 필터링
3. 아래 형식으로 목록을 제시한다:

### 출력 형식

```
📚 설계 문서 색인 ([N]건)

[Domain]
  1. [카테고리]/[domain명] — "[한 줄 요약]"
  2. [카테고리]/[domain명] — "[한 줄 요약]"

[독립 Feature]
  3. [카테고리]/[기능명] — "[한 줄 요약]"
     dependencies: [목록]
     affects: [목록]
```

- domain.md의 "한 줄 요약"은 문서의 첫 번째 blockquote(`>`)에서 추출한다
- domain.md에는 `dependencies`/`affects` 프론트매터가 없으므로 생략한다
- feature 문서의 기존 메타데이터 표시는 유지한다
- 대상 문서가 없으면: "설계 문서가 없습니다."
- domain.md가 없고 같은 접두사를 공유하는 feature 문서가 3개 이상이면 통합 요약 힌트를 표시한다: "💡 관련 문서가 [N]건 있습니다. 통합 요약이 필요하시면 `/dev-workflow:design-summary [접두사]`를 사용해주세요."

### 색인 → 전체 로드 전환

사용자가 목록을 본 후 특정 문서를 참조하려 하면 전체 로드 모드로 전환한다:

```
사용자: "1번이랑 2번 참고해서 진행하자"
→ 해당 문서 전체 로드
```

## 전체 로드 모드

키워드로 매칭된 설계 문서 전체를 컨텍스트에 주입한다.

### 매칭 전략

디렉토리명 및 파일명 부분 매칭(substring)으로 검색한다.

```
키워드: "aggro"

domain 레벨:   [해소된 경로]/[카테고리]/*aggro*.md
                [해소된 경로]/*aggro*/*.md
feature 레벨:  [해소된 경로]/[카테고리]/*aggro*/[기능명].md
                [해소된 경로]/*aggro*/[기능명]/[기능명].md

→ 모든 결과를 합쳐서 중복 제거
→ domain.md 우선 표시
```

기능명 디렉토리와 파일명은 동일하므로 (`[기능명]/[기능명].md`), 디렉토리명 매칭으로 파일을 특정할 수 있다.

### 단계적 로드 전략

매칭된 문서 수에 따라 동작이 분기된다:

**1~3건 매칭 → 즉시 전체 로드:**
```
📖 [N]건의 설계 문서를 로드합니다.
  - [카테고리]/[기능명]
  - [카테고리]/[기능명]

[각 문서 전체 내용을 컨텍스트에 주입]
```

**4건 이상 매칭 → 목록 제시 후 사용자 선택:**
```
📖 "[키워드]" 관련 설계 문서가 [N]건 매칭되었습니다.

  1. [카테고리]/[기능명] — "[한 줄 요약]"
  2. [카테고리]/[기능명] — "[한 줄 요약]"
  3. [카테고리]/[기능명] — "[한 줄 요약]"
  ...

전부 로드할까요, 아니면 필요한 문서를 선택하시겠습니까?
```

사용자 선택 방식 (다중 선택 가능):
- "전부 로드해" → 전체 로드
- "1, 2, 3번만" → 번호 선택
- "core랑 combat만" → 키워드 선택

### 매칭 실패 시

키워드에 해당하는 `status: complete` 문서가 없으면:
```
📖 "[키워드]" 관련 관련 설계 문서를 찾지 못했습니다.
```

### 다중 키워드

사용자가 여러 키워드를 동시에 언급하면 각각 매칭하여 합산한다:
```
사용자: "currency랑 inventory 참고해서 상점 설계하자"
→ "currency" 매칭 + "inventory" 매칭 → 합산 후 단계적 로드 전략 적용
```

## 로드 후 동작

문서가 로드된 후 호출한 스킬(brainstorming/plan-stage)로 제어가 돌아간다.
로드된 내용은 현재 세션의 컨텍스트에 포함되어, 이후 페르소나 토론 및 설계 작업에서 활용된다.
