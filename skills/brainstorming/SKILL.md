# Brainstorming Workflow

브레인스토밍은 5개 국면을 순서대로 진행한다 (국면 0은 카테고리 결정).
국면 전환은 Claude가 감지해서 제안하되, 강제하지 않는다.
모든 브레인스토밍은 Claude Code에서 진행한다 (파일 생성 필요).

**모든 파일 경로 규칙:** `docs/design/[카테고리]/[기능명]/`

## Ouroboros 연동 모드

세션 시작 시 workflow-orchestrator의 Ouroboros Detection 결과를 사용한다:
- **Enhanced Mode** (Path A/B): Ouroboros 플러그인 연동 활성화. 서브에이전트가 Ouroboros 에이전트 역할을 차용한다.
- **Standalone Mode** (Path C): Ouroboros 미설치. 내장 소크라테스 질문 패턴과 시드 템플릿으로 동작한다.

이하 국면별 설명에서:
- 📎 표시 = Enhanced Mode 전용 (Path A 또는 B)
- 표시 없음 = 모든 모드 공통

---

## 국면 0: 카테고리 결정

브레인스토밍 시작 전, 기능이 속할 카테고리를 결정한다.
issues/ 서브 워크플로우에서는 부모 카테고리를 상속하므로 이 국면을 생략한다.

**절차:**
1. 사용자의 요구사항을 분석하여 적합한 카테고리를 추천한다
2. 기존 카테고리 조회:
   - 경로 해소: `find . -maxdepth 2 -iname "docs" -type d` 실행 후 design 하위 확인
     (상세: development-principles "경로 해소 규칙" 참조)
   - 해소된 경로의 1depth 디렉토리를 스캔한다
3. 추천 결과를 제시한다:

**추천이 명확한 경우:**
```
📂 카테고리 추천: [카테고리명]
   기존 카테고리에서 [근거]와 일치합니다.
   이 카테고리로 진행할까요?
```

**추천이 모호한 경우:**
```
📂 기존 카테고리 목록:
  1. [카테고리A] — [소속 기능 수]개 기능
  2. [카테고리B] — [소속 기능 수]개 기능
  3. ✨ 새 카테고리 생성

어떤 카테고리에 속할까요?
```

**신규 카테고리 생성 시:**
- 도메인/시스템 단위인지 확인하는 질의응답 진행
- 소문자 kebab-case로 네이밍
- 확정 후 디렉토리 생성

4. 카테고리 확정 후 국면 1로 진행한다

## 페르소나 피드백 루프

모든 국면(국면 1~3)에 공통 적용되는 피드백 루프 규칙이다.

### 발동 조건
- 매 주제/질문마다 피드백 루프를 실행한다
- 최대 3회 루프 제한. 초과 시 사용자에게 "추가 토론이 필요할까요?" 승인을 받는다

### 비판적 관점 할당
- 비판적 역할은 라운드마다 다른 페르소나가 맡는다 (라운드 로빈)
- 라운드 시작 시 비판적 역할 페르소나를 명시한다: `[비판: 🏛️ 페르소나명]`
- 비판 시 반드시 대안 또는 출구를 함께 제시한다

**📎 Enhanced Mode (Ouroboros 연동):**
- 비판 담당 페르소나는 서브에이전트로 실행한다
- 서브에이전트는 Ouroboros 에이전트 역할 정의를 읽고, 페르소나 도메인 컨텍스트를 주입받아 응답을 생성한다
- 비판 담당이 아닌 페르소나는 메인 컨텍스트에서 자유 발언한다
- 서브에이전트 프롬프트 구조:
  1. Ouroboros 에이전트 역할 정의 전문 (agents/*.md)
  2. 페르소나 도메인 설명
  3. 현재 논의 주제 + 시드 컨텍스트 (있는 경우)

**Standalone Mode (Ouroboros 미설치):**
- 기존 라운드 로빈 방식 유지
- 비판 담당에게 아래 DO/DON'T 제약을 적용한다:
  - DO: 가정을 의심한다, 반사실적 질문을 던진다, 대안을 제시한다
  - DON'T: 동의로 시작하지 않는다, "좋은 의견입니다"를 쓰지 않는다

### 에이전트 매칭 규칙 (📎 Enhanced Mode 전용)

두 가지 규칙으로 매칭을 결정한다:

**규칙 1 — Step이 기본 에이전트를 결정한다:**
| Step | 에이전트 | 적용 범위 |
|---|---|---|
| Step A-0 (본질 질문) | Ontologist | Optional, 1회 서브에이전트 |
| Step A (인터뷰) | Socratic Interviewer | 전원 서브에이전트 |
| Step B (시드 추출) | Seed-Architect | 시드 구조화 (Enhanced Mode) |
| Step C (확장 토론) | Contrarian | 비판 슬롯만 서브에이전트 |
| 전환 시점 (Phase 1→2) | Simplifier | Optional, 스코프 정리 |

**규칙 2 — 비판 담당만 서브에이전트로 실행한다:**
- Step A는 예외: 해결책 제안 차단을 위해 전원 서브에이전트
- Step C: 비판 담당 1명만 서브에이전트, 나머지는 메인 컨텍스트

### 출력 형식
- 의견 본문은 자유 형식 (페르소나 이모지 + 이름 접두사)
- 정형 태그(찬성/반대 등)를 사용하지 않는다
- 라운드 카운터를 포함한다

```
[라운드 1/3] [비판: 🔧 페르소나명] [에이전트: Socratic / Contrarian / Standalone]

🏛️ 페르소나A [Socratic]:
[의견]

🔧 페르소나B [Socratic]:
[의견]

👤 페르소나C [Socratic]:
[의견]

**합의:** [결론 요약]

---
📌 결정이 필요한 항목:
1. [한 줄 요약] — [선택지 또는 질문]
2. [한 줄 요약] — [선택지 또는 질문]
```

- **에이전트 태그**: Enhanced Mode에서 서브에이전트로 실행된 페르소나에 에이전트명을 표시한다
  - `[Socratic]` — Socratic Interviewer 사고 제약으로 생성된 발언
  - `[Contrarian]` — Contrarian 사고 제약으로 생성된 발언 (비판 담당)
  - Standalone Mode 또는 서브에이전트가 아닌 발언에는 태그를 붙이지 않는다
  - 라운드 헤더에도 현재 사용 중인 에이전트를 표시한다

- 합의 시: `**합의:**` + 결론 요약
- 미합의 시: `**미합의:**` + 쟁점 요약
- **결정 요청 분리**: 페르소나 토론 후 구분선(`---`) 아래에 `📌 결정이 필요한 항목:` 블록을 배치한다
  - 토론 중 사용자 답변이 필요한 항목을 번호 + 한 줄 요약으로 명시한다
  - "이 두 가지에 대해" 같은 대명사 참조를 사용하지 않는다
  - 결정 항목이 없으면 (페르소나 간 합의 도출된 경우) 블록을 생략한다

### 국면별 방향성 가이드
강제 비율 없이 방향성으로 자연 조절한다:
- 국면 1 (탐색): 비판보다 호기심으로 — 아이디어를 확장한다
- 국면 2 (발견): 기본 비율 유지 — 빈틈을 찾아 채운다
- 국면 3 (검증): 리스크를 놓치지 말 것 — 기술적 실현 가능성에 집중한다

brainstorming의 발산적 성격을 훼손하지 않는다. 자유로운 분위기를 유지한다.

## 설계 문서 크로스레퍼런스

국면 1~3 진행 중 사용자가 기존에 구현 완료된 설계 문서를 참조하려 할 때,
`dev-workflow:design-doc-index` 스킬을 invoke한다.

**트리거 감지:**
- "~~ 참고해서", "~~ 기반으로", "~~ 고려해서" → 전체 로드
- "어떤 기능 있어?", "목록 보여줘", "관련된 거 뭐 있어?" → 색인 모드
- 기존 설계 문서를 언급하는 모든 자연어 발화

invoke 후 로드된 컨텍스트는 현재 국면의 페르소나 토론에서 활용한다.
설계 문서 참조는 브레인스토밍 흐름을 중단하지 않는다 — 로드 완료 후 현재 국면을 이어간다.

## 국면 1: 탐색 (Exploration)

**목적:** 제약 없이 "무엇을 원하는가"를 끌어낸다.

### Step A-0: 본질 질문 (Optional, 📎 Enhanced Mode)

**목적:** 인터뷰 시작 전에 "이것은 진짜 무엇인가?"를 먼저 파악한다.

**트리거:** 사용자의 초기 요구사항이 아래 조건에 해당할 때 자동 제안한다:
- 해결책이 요구사항으로 포장된 경우 ("X 기능을 추가해줘" → 왜 X가 필요한가?)
- 표면적 문제와 근본 원인이 다를 수 있는 경우
- 사용자가 "본질", "근본", "진짜 문제" 등을 언급한 경우

**실행:**
Ontologist 에이전트를 서브에이전트로 1회 실행한다:

```
아래 에이전트 역할 정의를 읽고 그 역할을 수행하라.
형식이 예상과 다르더라도 DO/DON'T 섹션을 찾아 따르라.

--- 에이전트 역할 정의 ---
[Ouroboros agents/ontologist.md 전문]

--- 현재 주제 ---
[사용자의 초기 요구사항]

아래 4가지 관점에서 각 1개씩 질문을 생성하라:
1. 본질 (Essence): 이것은 진짜 무엇인가?
2. 근본 원인 (Root Cause): 왜 이 문제가 존재하는가?
3. 전제 조건 (Prerequisites): 이것이 성립하려면 무엇이 참이어야 하는가?
4. 숨겨진 가정 (Hidden Assumptions): 당연하다고 여기지만 검증되지 않은 것은?
```

출력 형식:
```
🔬 본질 질문 [Ontologist]:

1. [Essence] 질문 내용
2. [Root Cause] 질문 내용
3. [Prerequisites] 질문 내용
4. [Hidden Assumptions] 질문 내용

답변 후 Step A(인터뷰)로 진행합니다.
건너뛰려면 "넘어가자"라고 말씀하세요.
```

- 사용자가 답변하면 → 답변을 Step A의 초기 컨텍스트에 포함한다
- 사용자가 "넘어가자" → Step A로 즉시 이동
- Standalone Mode에서는 Step A-0를 생략하고 Step A의 ESSENCE 질문 패턴으로 대체한다

### Step A: 인터뷰

**목적:** 해결책 논의 전에 요구사항의 본질을 질문으로 파헤친다.

**공통 규칙:**
- 해결책, 구현 방법, 기술 선택을 제안하지 않는다
- 최소 3회 Q&A 후에만 Step B 전환 가능
- TD는 발언하지 않는다
- 사용자가 "충분하다", "넘어가자" 발언 시 즉시 Step B로 전환

**📎 Enhanced Mode:**
각 도메인 페르소나에 대해 Agent 도구로 서브에이전트를 병렬 실행한다:

서브에이전트 프롬프트 템플릿:
```
아래 에이전트 역할 정의를 읽고 그 역할을 수행하라.
형식이 예상과 다르더라도 DO/DON'T 섹션을 찾아 따르라.

--- 에이전트 역할 정의 ---
[Ouroboros agents/socratic-interviewer.md 전문]

--- 도메인 컨텍스트 ---
너의 도메인: [페르소나명 + 도메인 설명]
예: "🎮 Game Designer — 게임 디자인, 플레이어 경험, 재미 요소 관점"

--- 현재 주제 ---
[사용자의 기능 설명 또는 현재까지의 논의 요약]

이 도메인 관점에서 Socratic Interviewer로서 질문을 2~3개 생성하라.
```

서브에이전트 결과를 **통합 정제**하여 사용자에게 제시한다:

1. 모든 서브에이전트의 질문을 수집한다
2. 의미가 겹치는 질문을 병합한다 (출처 페르소나를 태그로 보존)
3. 주제별로 그룹핑하고 번호를 매긴다
4. 각 질문에 응답 힌트를 추가한다

출력 형식:
```
📋 핵심 질문 (N개로 정리) [에이전트: Socratic]:

1. [주제] 질문 내용
   ← [페르소나A 이모지] [Socratic] + [페르소나B 이모지] [Socratic] (겹치는 경우)
   > 힌트: 답변 시 고려할 관점

2. [주제] 질문 내용
   ← [페르소나 이모지] [Socratic] 단독
   > 힌트: 답변 시 고려할 관점

번호별로 답변하거나, 자유롭게 설명해 주세요.
```

**Standalone Mode:**
아래 4가지 소크라테스 질문 패턴을 적용하여 페르소나가 질문한다:
- **WHY-CHAIN**: "왜 X가 필요한가?" → 답변 → "그 이유는 왜 중요한가?" (최대 3단계)
- **COUNTERFACTUAL**: "X가 없다면 무엇이 달라지는가?"
- **BOUNDARY**: "X가 절대 해서는 안 되는 것은?"
- **ESSENCE**: "이것은 진짜 무엇인가? 본질은?"

페르소나는 질문 패턴 중 1~2개를 선택하여 도메인 관점의 질문을 한다. 해결책 제안 금지 규칙은 동일.

질문 생성 후, Enhanced Mode와 동일한 **통합 정제** 규칙을 적용한다 (중복 병합, 주제별 그룹핑, 번호 매기기, 응답 힌트). 출력 형식도 동일.

### Step B: 시드 추출

**목적:** 인터뷰 결과를 구조화된 시드로 변환하고 사용자 확인을 받는다.

**공통 규칙:**
- 시드 추출 후 반드시 사용자 확인을 받는다. 확인 없이 Step C로 넘어가지 않는다.
- 시드와 함께 명확도 체크리스트를 표시한다.

**📎 Enhanced Mode:**
Seed-Architect 에이전트를 서브에이전트로 실행하여 시드를 생성한다:

```
아래 에이전트 역할 정의를 읽고 그 역할을 수행하라.
형식이 예상과 다르더라도 DO/DON'T 섹션을 찾아 따르라.

--- 에이전트 역할 정의 ---
[Ouroboros agents/seed-architect.md 전문]

--- 인터뷰 결과 ---
[Step A의 Q&A 전체 내용]

인터뷰 결과를 구조화된 시드 YAML로 변환하라.
모호한 항목은 open_questions에 배치하라.
```

서브에이전트가 생성한 시드를 검토하고, 가능한 경우 Ouroboros validate_seed 도구로 형식을 검증한다 (Path A만).

**Standalone Mode:**
아래 시드 형식으로 대화 내용을 구조화한다 (templates/seed.yaml 참조):

```yaml
goal: "[핵심 목표 1문장]"
constraints:
  - "[제약조건]"
non_goals:
  - "[명시적 비목표]"
success_criteria:
  - "[측정 가능한 성공 기준]"
assumptions:
  - "[검증되지 않은 가정]"
open_questions:
  - "[미해결 질문]"
context: "[배경 설명]"
```

**시드 추출 후 출력 형식:**
```
── 시드 추출 결과 ────────────────────────────
  Goal: [핵심 목표]
  Constraints: [제약조건 목록]
  Non-goals: [비목표 목록]
  Success Criteria: [성공 기준]
  Assumptions: [노출된 가정]
  Open Questions: [미해결 질문]

  이것이 핵심 요구사항입니다. 맞습니까?
  1. Yes
  2. 수정이 필요하다
───────────────────────────────────────────────
```

사용자가 "수정이 필요하다" 선택 시 → 수정 사항 반영 후 재확인.

**명확도 체크리스트 (시드 확인과 함께 표시):**
```
── 명확도 체크 ────────────────────────────────
  ✅ / ⚠️ / ❌  Goal Clarity: [상태 설명]
  ✅ / ⚠️ / ❌  Constraint Clarity: [상태 설명]
  ✅ / ⚠️ / ❌  Success Criteria: [상태 설명]
  → 다음 국면 집중 영역: [⚠️/❌ 항목 목록]
───────────────────────────────────────────────
```

### Step C: 확장 토론

**목적:** 시드를 기반으로 요구사항을 확장하고 페르소나 간 토론을 진행한다.

**규칙:**
- 도메인 페르소나가 주도한다
- TD는 발언하지 않는다
- 요구사항이 모순되거나 비현실적이어도 일단 수용하고 기록한다
- Claude는 실현 가능성을 언급하지 않는다
- 한 번에 최대 2개 질문만 한다
- 시드의 open_questions와 ⚠️/❌ 항목을 우선 토론 주제로 삼는다

**📎 Enhanced Mode 추가 규칙:**
- 비판 담당 페르소나는 Contrarian 서브에이전트로 실행한다 (에이전트 매칭 규칙 참조)
- 대화가 교착 상태일 때 → Hacker 서브에이전트를 추가 투입하여 비정통적 대안 탐색

**Standalone Mode 추가 규칙:**
- 비판 담당에게 DO/DON'T 제약을 적용한다 (비판적 관점 할당 참조)

**전환 감지 (하나라도 충족 시 제안):**
- 사용자가 "이정도면 될 것 같아", "다른 건 없을 것 같아", "이게 다야" 류의 발언
- 요구사항 목록이 윤곽을 갖춘 시점:
  "추가로 논의할 요구사항이 있으신가요, 아니면 다음 단계로 넘어갈까요?"
- 명확도 체크리스트의 Goal과 Constraint가 모두 ✅인 경우 전환을 적극 제안한다

**전환 확정 시 즉시:**
`docs/design/[카테고리]/[기능명]/phase1_exploration.md` 생성 — 이후 절대 수정하지 않는다.

포함 내용:
- 탐색된 요구사항 전체 목록 (확정/미확정 구분 없이 언급된 모든 것)
- 사용자가 명시적으로 제외한 항목과 이유
- 확정된 페르소나 구성
- 페르소나 피드백 결과 (주제별 합의/미합의 요약)
- **시드 (Step B에서 사용자 확인 완료된 버전)**
- **명확도 체크리스트 (전환 시점 기준)**

### Simplifier 스코프 정리 (Optional, 📎 Enhanced Mode)

Phase 1→2 전환 확정 후, phase1_exploration.md 생성 직후에 실행한다.

**트리거:** 아래 조건에 해당할 때 자동 제안한다:
- 요구사항 수가 8개 이상일 때
- 사용자가 "너무 많다", "줄여야", "핵심만" 등을 언급한 경우
- 명확도 체크리스트에서 ⚠️/❌ 항목이 3개 이상일 때

**실행:**
Simplifier 에이전트를 서브에이전트로 1회 실행한다:

```
아래 에이전트 역할 정의를 읽고 그 역할을 수행하라.
형식이 예상과 다르더라도 DO/DON'T 섹션을 찾아 따르라.

--- 에이전트 역할 정의 ---
[Ouroboros agents/simplifier.md 전문]

--- 현재 요구사항 목록 ---
[phase1_exploration.md의 요구사항 전체]

YAGNI 원칙을 적용하여:
1. 핵심 요구사항과 부수 요구사항을 분류하라
2. 제거해도 핵심 가치가 유지되는 항목을 식별하라
3. 복잡도를 50% 줄이는 최소 스코프를 제안하라
```

출력 형식:
```
✂️ 스코프 정리 제안 [Simplifier]:

핵심 (제거 불가):
- [요구사항 목록]

선택적 (제거 가능):
- [요구사항] — 제거 시 영향: [설명]
- [요구사항] — 제거 시 영향: [설명]

제안: 선택적 항목을 v2로 미루면 복잡도를 약 N% 줄일 수 있습니다.

1. 제안 수용 (선택적 항목 v2로 미룸)
2. 일부만 수용 (항목 지정)
3. 유지 (전체 스코프 유지)
```

- 사용자 결정에 따라 시드의 non_goals에 미룬 항목을 추가한다
- 트리거 조건에 해당하지 않으면 자동으로 스킵한다
- Standalone Mode에서는 동일한 출력 형식으로 Claude가 직접 분석한다

## 국면 2: 발견 (Discovery)

**목적:** 대화에서 드러나지 않은 요구사항의 빈틈을 찾아 채운다.

**규칙:**
- Claude가 지금까지 논의된 내용을 정리하고 미정의 영역을 명시한다
- 도메인 페르소나들이 피드백 루프를 통해 요구사항을 구체화한다 (페르소나 피드백 루프 규칙 참조)
- TD는 여전히 침묵한다
- 한 번에 최대 2개 질문만 한다

**국면 2 시작 시 출력:**
```
── 현재까지 논의된 내용 ─────────────────────────────

확인된 요구사항:
- [목록]

아직 정의되지 않은 영역:
- [목록]

이 부분들을 하나씩 구체화하겠습니다.
────────────────────────────────────────────────────
```

**전환 감지 (하나라도 충족 시 제안):**
- 미정의 영역이 모두 논의됨
- 기술적 실현 가능성 검토가 필요한 시점
- 사용자가 "기술적으로 가능한지 보고 싶어" 류의 발언

전환 제안: "요구사항이 구체화되었습니다. TD를 불러서 기술적 실현 가능성을 검토할까요? [국면 3: 검증]"

**전환 확정 시 즉시:**
`docs/design/[카테고리]/[기능명]/phase2_discovery.md` 생성 — 이후 절대 수정하지 않는다.

포함 내용:
- 국면 2에서 새로 발견된 미정의 영역 목록
- 각 항목에 대한 Q&A 결정 사항
- 페르소나 피드백 결과 (주제별 합의/미합의 요약)

## 국면 3: 검증 (Validation)

**목적:** TD가 최소한의 기술 가이드라인을 제공한다. 대안 강요 금지.

**규칙:**
- TD 페르소나가 합류한다 (도메인 페르소나 유지)
- TD는 구현 비용/난이도/리스크 언급만 한다. 대안을 강요하지 않는다
- 사용자가 리스크를 감수하겠다면 그 결정을 존중한다

**TD 발언 형식:**
```
── 🔧 TD 기술 검토 ──────────────────────────────────

[요구사항명]
판단: [기술적 판단]
가이드라인: [권장 방향 또는 주의사항]
리스크: 낮음 / 중간 / 높음

────────────────────────────────────────────────────
```

**전환 감지 (하나라도 충족 시 제안):**
- TD 검토가 모든 요구사항에 대해 완료됨
- 재협의 항목이 정리되고 최종 방향이 확정됨
- 사용자가 "정리해줘", "문서로 만들어줘" 류의 발언

전환 제안: "기술 검토까지 완료되었습니다. 지금까지의 논의를 설계 문서로 정리할까요? [국면 4: 정리]"

**전환 확정 시 즉시:**
`docs/design/[카테고리]/[기능명]/phase3_validation.md` 생성 — 이후 절대 수정하지 않는다.

포함 내용:
- TD가 검토한 각 요구사항별 판단 결과 및 리스크 수준
- 재협의가 발생한 항목과 최종 결정
- TD가 제시한 기술 가이드라인
- 페르소나 피드백 결과 (주제별 합의/미합의 요약)

## 국면 4: 정리 (Consolidation)

**목적:** 전체 대화를 표준 설계 문서로 변환한다.

**규칙:**
- Claude가 전체 대화 기반으로 설계 문서 초안을 자동 생성한다
- 생성 후 사용자 확인을 받는다
- 수정 요청 시 반영 후 재확인한다
- 확정 시 `docs/design/[카테고리]/[기능명]/[기능명].md` 파일을 직접 생성한다
  - `docs/design/[카테고리]/[기능명]/` 디렉토리는 phase1 생성 시점에 이미 존재
- 📎 **Enhanced Mode — Seed YAML 생성:** 설계 문서 확정 후, Seed-Architect 서브에이전트로 최종 시드 YAML을 생성한다.
  - 입력: 확정된 설계 문서 전문 + phase1~3 핵심 결정 사항
  - 출력: `docs/design/[카테고리]/[기능명]/seed.yaml`
  - 이 시드는 Ouroboros `ooo run` 또는 PLAN 단계의 참조 자료로 활용할 수 있다
  - Standalone Mode에서는 Step B에서 생성한 시드를 설계 문서에 인라인으로 포함하고, 별도 파일은 생성하지 않는다
- 파일 생성 완료 후 PLAN 단계로 연속 진행을 제안한다:
  "설계 문서가 `docs/design/[카테고리]/[기능명]/[기능명].md`에 저장되었습니다.
   PLAN 단계로 바로 진행할까요?"
- PLAN 진행 시 Persona Resolution은 PLAN 페르소나만 새로 확정한다

**디렉토리 구조:**
```
docs/design/[카테고리]/[기능명]/
```

## issues/ 서브 워크플로우 (hotfix 전략)

개발 중 설계 보완이 필요한 서브 문제가 발생하면 issues/ 디렉토리에서 격리 처리한다.
git hotfix 브랜치와 동일한 전략이다.

### 진입 조건

사용자가 설계 논의가 필요하다고 판단하여 명시적으로 요청한 경우에만 진입한다.
Claude가 자동으로 issues/를 생성하지 않는다.

### 진입 절차

1. 부모 기능 디렉토리 확인: `docs/design/[카테고리]/[기능명]/`
2. 이슈 디렉토리 생성: `docs/design/[카테고리]/[기능명]/issues/[문제명]/`
3. 국면 0(카테고리 결정)은 생략 — 부모 카테고리를 상속한다
4. 정규 4단계(국면 1~4) 브레인스토밍을 진행한다
5. phase 파일 및 설계 문서를 issues/[문제명]/ 하위에 생성한다

### 이슈 디렉토리 구조

```
docs/design/[카테고리]/[기능명]/issues/[문제명]/
├── phase1_exploration.md
├── phase2_discovery.md
├── phase3_validation.md
├── [문제명].md              ← 이슈 설계 문서
└── HANDOFF.md               ← 이슈 미완료 시
```

### 완료 후 통합

이슈 해결이 완료되면 `dev-workflow:document-consolidation` 스킬의
consolidate-issue 모드를 사용하여 부모 design 문서에 통합한다.
통합은 "원래 하나였던 것처럼" 자연스럽게 병합한다 (참조 링크 아님).
통합 후 issues/[문제명]/ 디렉토리를 삭제한다.

### 중첩 금지

issues/ 내에서 추가 문제 발견 시:
- 같은 맥락 확장 → 현재 이슈의 phase를 리셋하여 재진행
  - 기존 phase 파일 삭제, 유효 결정사항을 새 Phase 1 컨텍스트로 이월
- 새로운 기능 필요 → 별도 기능 디렉토리로 격상 (feature 브랜치 전략)
- 무관한 문제 → 메모 후 별도 세션에서 처리

### 개발 중 문제 판단 플로우

```
테스트 중 문제 발생
  │
  ├─ 코드 수정만으로 해결? → 즉시 수정 (commit)
  │
  └─ 설계 논의 필요?
       ├─ 현재 기능의 설계? → issues/ (hotfix)
       ├─ 새로운 기능 필요? → 별도 기능 디렉토리 (feature)
       └─ 현재 기능과 무관? → 메모 후 별도 세션
```
