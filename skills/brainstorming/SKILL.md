# Brainstorming Workflow

브레인스토밍은 5개 국면을 순서대로 진행한다 (국면 0은 카테고리 결정).
국면 전환은 Claude가 감지해서 제안하되, 강제하지 않는다.
모든 브레인스토밍은 Claude Code에서 진행한다 (파일 생성 필요).

**모든 파일 경로 규칙:** `docs/design/[카테고리]/[기능명]/`

## Ouroboros 연동 모드

workflow-orchestrator의 Ouroboros Detection 결과를 사용한다 (지연 감지 — 첫 서브에이전트 투입 직전에 1회 실행됨):
- **Path A (MCP 연동)**: Ouroboros MCP 도구 사용 가능 — validate_seed 등 MCP 전용 도구를 추가로 사용한다.
- **Path B (내장 역할)**: MCP 미사용 — 내장 역할 정의만으로 동일하게 동작한다.

에이전트 역할 정의는 Path와 무관하게 본 스킬의 `references/agent-roles.md` 내장본을 사용한다 (v1.16.0 — 외부 플러그인 파일 탐색 제거). 구 Standalone Mode(Path C)는 소멸한 경로다 — 본문에 남아 있는 "Standalone Mode" 분기 서술은 적용하지 않는다 (후속 릴리스에서 정리 예정).

이하 국면별 설명에서:
- 📎 표시 = 서브에이전트 실행 구간 (내장 역할 정의 사용)
- 표시 없음 = 메인 컨텍스트 공통

## 국면 0: 카테고리 결정

브레인스토밍 시작 전, 기능이 속할 카테고리를 결정한다.
issues/ 서브 워크플로우에서는 부모 카테고리를 상속하므로 이 국면을 생략한다.

**절차:**
1. 사용자의 요구사항을 분석하여 적합한 카테고리를 추천한다
2. 기존 카테고리 조회:
   - 경로 해소: `find . -maxdepth 2 -iname "docs" -type d` 실행 후 design 하위 확인
     (상세: development-principles "경로 해소 규칙" 참조)
   - 해소된 경로의 1depth 디렉토리를 스캔한다
3. 추천 결과를 제시한다:

**추천이 명확한 경우:**
```
📌 결정 요청 : 카테고리 확정

**배경:** 이 기능의 설계 문서가 저장될 분류 폴더입니다. 기존 카테고리 [카테고리명]이 [근거]와 일치합니다.

1. Yes — 고르면: docs/design/[카테고리명]/ 아래에 기능 폴더가 생깁니다
2. No — 고르면: 다른 카테고리를 지정하거나 새로 만듭니다
```

**추천이 모호한 경우:**
```
📂 기존 카테고리 목록:
  1. [카테고리A] — 소속 기능 [n]개
  2. [카테고리B] — 소속 기능 [n]개
  0. ✨ 새 카테고리 생성 — 고르면: 이름 질의로 진행

어떤 카테고리에 속할까요?
```

**신규 카테고리 생성 시:**
- 도메인/시스템 단위인지 확인하는 질의응답 진행
- 소문자 kebab-case로 네이밍
- 확정 후 디렉토리 생성

4. 카테고리 확정 후 국면 1로 진행한다

## 페르소나 피드백 루프

모든 국면(국면 1~3)에 공통 적용되는 피드백 루프 규칙이다.

### 발동 조건
- 매 주제/질문마다 피드백 루프를 실행한다
- 최대 3회 루프 제한. 초과 시 사용자에게 "추가 토론이 필요할까요?" 승인을 받는다

### 비판적 관점 할당
- 비판적 역할은 라운드마다 다른 페르소나가 맡는다 (라운드 로빈)
- 라운드 시작 시 비판적 역할 페르소나를 명시한다: `[비판: 🏛️ 페르소나명]`
- 비판 시 반드시 대안 또는 출구를 함께 제시한다

**📎 Enhanced Mode (Ouroboros 연동):**
- 비판 담당 페르소나는 서브에이전트로 실행한다
- 서브에이전트는 Ouroboros 에이전트 역할 정의를 읽고, 페르소나 도메인 컨텍스트를 주입받아 응답을 생성한다
- 비판 담당이 아닌 페르소나는 메인 컨텍스트에서 자유 발언한다
- 서브에이전트 프롬프트 구조:
  1. 에이전트 역할 정의 전문 (본 스킬 references/agent-roles.md의 해당 역할 블록)
  2. 페르소나 도메인 설명
  3. 현재 논의 주제 + 시드 컨텍스트 (있는 경우)
  4. 출력 제약: 사용자를 향한 결정 요청 문구를 포함하지 않는다 — 의견·비판·대안만 반환한다 (결정 요청은 메인 컨텍스트가 decision-flow 형식으로 재구성한다)
- 에이전트 역할 정의 로드: 본 스킬의 `references/agent-roles.md`에서 해당 역할 블록을 Read한다 (외부 탐색 없음. Read 실패 시 최소 요건: 해당 역할의 사고 제약을 요약 적용 — Socratic=질문만·해결책 금지, Contrarian=가정 반박+대안)

**Standalone Mode (Ouroboros 미설치):**
- 기존 라운드 로빈 방식 유지
- 비판 담당에게 아래 DO/DON'T 제약을 적용한다:
  - DO: 가정을 의심한다, 반사실적 질문을 던진다, 대안을 제시한다
  - DON'T: 동의로 시작하지 않는다, "좋은 의견입니다"를 쓰지 않는다

### 에이전트 매칭 규칙 (📎 Enhanced Mode 전용)

두 가지 규칙으로 매칭을 결정한다:

**규칙 1 — Step이 기본 에이전트를 결정한다:**
| Step | 에이전트 | 적용 범위 |
|---|---|---|
| Step A-0 (본질 질문) | Ontologist | Optional, 1회 서브에이전트 |
| Step A (인터뷰) | Socratic Interviewer | 전원 서브에이전트 |
| Step B (시드 추출) | Seed-Architect | 시드 구조화 (Enhanced Mode) |
| Step C (확장 토론) | Contrarian | 비판 슬롯만 서브에이전트 |
| Step C (교착 시) | Hacker | Ad-hoc, 교착 상태 돌파용 |
| 전환 시점 (Phase 1→2) | Simplifier | Optional, 스코프 정리 |

**규칙 2 — 비판 담당만 서브에이전트로 실행한다:**
- Step A는 예외: 해결책 제안 차단을 위해 전원 서브에이전트
- Step C: 비판 담당 1명만 서브에이전트, 나머지는 메인 컨텍스트

### 출력 형식

- 의견 본문은 자유 형식 (페르소나 이모지 + 이름 접두사)
- 정형 태그(찬성/반대 등)를 사용하지 않는다
- 라운드 카운터를 포함한다: `[라운드 N/3]`

**결정 요청 및 피드백 루프 출력 형식 (SSOT):** 한 줄 헤더·결정 박스·사전 브리핑·진행 모드·에이전트 태그·합의/미합의 마커·결정 요청 분리의 형식은 `decision-flow.md`가 정의한다. 페르소나 토론 시작 전 반드시 해당 파일을 Read하고 준수한다. 파일은 사용자 프로젝트가 아닌 플러그인 디렉토리에 있다:
- 본 스킬 디렉토리 기준: `../workflow-orchestrator/decision-flow.md`
- 미확보 시 동적 탐색: `find "$HOME/.claude/plugins" -path "*dev-workflow*/skills/workflow-orchestrator/decision-flow.md" -not -path "*/worktrees/*" -type f | head -1` (캐시는 `.../dev-workflow/<버전>/skills/...` 구조)

**Read 실패 시 최소 규칙 (폴백):**
- 서브에이전트로 실행된 페르소나 발언에 `[Socratic]`/`[Contrarian]` 등 에이전트 태그를 부착한다
- 합의는 `**합의:**` · 미합의는 `**미합의:**` 마커로 요약한다
- 결정 항목은 대명사 없이 명시적 이름으로 지칭하고, 한 번에 하나씩 요청한다. 노출된 항목에만 답할 수 있으며, 하나씩 모드의 다중 값 응답은 첫 값만 반영한다

### 국면별 방향성 가이드
강제 비율 없이 방향성으로 자연 조절한다:
- 국면 1 (탐색): 비판보다 호기심으로 — 아이디어를 확장한다
- 국면 2 (발견): 기본 비율 유지 — 빈틈을 찾아 채운다
- 국면 3 (검증): 리스크를 놓치지 말 것 — 기술적 실현 가능성에 집중한다

brainstorming의 발산적 성격을 훼손하지 않는다. 자유로운 분위기를 유지한다.

## 설계 문서 크로스레퍼런스

국면 1~3 진행 중 사용자가 기존에 구현 완료된 설계 문서를 참조하려 할 때,
`dev-workflow:design-doc-index` 스킬을 invoke한다.

**트리거 감지 (의도 기반):** 특정 문서를 근거로 쓰려는 의도(예: "~~ 참고해서") → 전체 로드, 무엇이 있는지 탐색하려는 의도(예: "목록 보여줘") → 색인 모드. 기존 설계 문서를 언급하는 자연어 발화 전반이 대상이다.

invoke 후 로드된 컨텍스트는 현재 국면의 페르소나 토론에서 활용한다.
설계 문서 참조는 브레인스토밍 흐름을 중단하지 않는다 — 로드 완료 후 현재 국면을 이어간다.

## 국면 1: 탐색 (Exploration)

**목적:** 제약 없이 "무엇을 원하는가"를 끌어낸다.

### Step A-0: 본질 질문 (Optional, 📎 Enhanced Mode)

**목적:** 인터뷰 시작 전에 "이것은 진짜 무엇인가?"를 먼저 파악한다.

**트리거:** 사용자의 초기 요구사항이 아래 조건에 해당할 때 자동 제안한다:
- 해결책이 요구사항으로 포장된 경우 ("X 기능을 추가해줘" → 왜 X가 필요한가?)
- 표면적 문제와 근본 원인이 다를 수 있는 경우
- 사용자가 "본질", "근본", "진짜 문제" 등을 언급한 경우

**실행:**
Ontologist 에이전트를 서브에이전트로 1회 실행한다:

전문은 본 스킬의 references/templates.md 「Step A-0 Ontologist 프롬프트」을 Read하여 사용하라.
(Read 실패 시 최소 요건: 본질/근본원인/전제/숨은가정 4관점 질문 각 1개)

- 사용자가 답변하면 → 답변을 Step A의 초기 컨텍스트에 포함한다
- 사용자가 "넘어가자" → Step A로 즉시 이동
- Standalone Mode에서는 Step A-0를 생략하고 Step A의 ESSENCE 질문 패턴으로 대체한다

**미답변 질문 조사 (A-0 전용):**
Step A-0의 4개 본질 질문에 대해서도 미답변 조사 패턴을 적용한다. 단, A-0 출력 직후 1회만 실행한다.
처리 절차는 `### 미답변 질문 조사` 섹션과 동일하다 (2-pass: 직접 답변 반영 → 미답변 확인 → 조사 실행 → 결과 제시).
단, 조사 완료 후 Step B가 아닌 Step A로 진행한다.
조사 결과 확정 후 Step A의 초기 컨텍스트에 포함한다.

### Step A: 인터뷰

**목적:** 해결책 논의 전에 요구사항의 본질을 질문으로 파헤친다.

**공통 규칙:**
- 해결책, 구현 방법, 기술 선택을 제안하지 않는다
- 최소 3회 Q&A 후에만 Step B 전환 가능
- TD는 발언하지 않는다
- 사용자가 "충분하다", "넘어가자" 발언 시 즉시 Step B로 전환

**📎 Enhanced Mode:**
각 도메인 페르소나에 대해 Agent 도구로 서브에이전트를 병렬 실행한다:

서브에이전트 프롬프트 템플릿:
전문은 본 스킬의 references/templates.md 「Step A Socratic 프롬프트」을 Read하여 사용하라.
(Read 실패 시 최소 요건: 도메인 관점 소크라테스 질문 2~3개, 해결책 제안 금지)

서브에이전트 결과를 **통합 정제**하여 사용자에게 제시한다:

1. 모든 서브에이전트의 질문을 수집한다
2. 의미가 겹치는 질문을 병합한다 (출처 페르소나를 태그로 보존)
3. 주제별로 그룹핑하고 번호를 매긴다
4. 각 질문에 응답 힌트를 추가한다

**Standalone Mode:**
아래 4가지 소크라테스 질문 패턴을 적용하여 페르소나가 질문한다:
- **WHY-CHAIN**: "왜 X가 필요한가?" → 답변 → "그 이유는 왜 중요한가?" (최대 3단계)
- **COUNTERFACTUAL**: "X가 없다면 무엇이 달라지는가?"
- **BOUNDARY**: "X가 절대 해서는 안 되는 것은?"
- **ESSENCE**: "이것은 진짜 무엇인가? 본질은?"

페르소나는 질문 패턴 중 1~2개를 선택하여 도메인 관점의 질문을 한다. 해결책 제안 금지 규칙은 동일.

질문 생성 후, Enhanced Mode와 동일한 **통합 정제** 규칙을 적용한다 (중복 병합, 주제별 그룹핑, 번호 매기기, 응답 힌트). 출력 형식도 동일.

### 미답변 질문 조사

**목적:** 사용자가 답변하지 않은 질문에 대해 코드/문서 기반 잠정 답변을 제공한다.

**트리거:** 통합 정제된 질문에 대한 사용자 응답 후, 미답변 질문이 존재할 때.
모든 질문에 답변한 경우 이 섹션을 스킵하고 Step B로 진행한다.

**처리 순서 (2-pass):**

**Pass 1 — 직접 답변 반영:**
사용자가 답변한 질문을 먼저 처리하고 반영 결과를 표시한다:
```
✅ 답변 반영: [답변한 질문 번호 목록]
```

**Pass 2 — 미답변 확인 + 조사:**
미답변 질문이 있으면 사용자에게 조사 여부를 확인한다:
```
미답변 질문이 있습니다: [번호 목록]

1. 전부 조사 — 고르면: 모든 미답변 질문을 코드/문서에서 조사해 잠정 답을 제시합니다
2. 선택 조사 (번호 지정) — 고르면: 지정한 것만 조사합니다
3. 스킵 — 고르면: 조사 없이 다음 단계로 넘어갑니다
```

- 1(전부 조사) → 모든 미답변 질문을 조사한다
- 2(선택 조사) → 사용자가 지정한 번호만 조사한다
- 3(스킵) → 조사 없이 Step B(또는 다음 라운드)로 진행한다

**조사 실행:**

**📎 Enhanced Mode:**

## issues/ 서브 워크플로우 (hotfix 전략)

개발 중 설계 보완이 필요한 서브 문제가 발생하면 issues/ 디렉토리에서 격리 처리한다.
git hotfix 브랜치와 동일한 전략이다.

### 진입 조건

사용자가 설계 논의가 필요하다고 판단하여 명시적으로 요청한 경우에만 진입한다.
Claude가 자동으로 issues/를 생성하지 않는다.

### 진입 절차

1. 부모 기능 디렉토리 확인: `docs/design/[카테고리]/[기능명]/`
2. 이슈 디렉토리 생성: `docs/design/[카테고리]/[기능명]/issues/[문제명]/`
3. 국면 0(카테고리 결정)은 생략 — 부모 카테고리를 상속한다
4. 정규 4단계(국면 1~4) 브레인스토밍을 진행한다
5. phase 파일 및 설계 문서를 issues/[문제명]/ 하위에 생성한다

### 이슈 디렉토리 구조

전문은 본 스킬의 references/templates.md 「issues/ 이슈 디렉토리 구조」을 Read하여 사용하라.
(Read 실패 시 최소 요건: phase1~3.md + [문제명].md + HANDOFF.md 구성)

### 완료 후 통합

이슈 해결이 완료되면 `dev-workflow:document-consolidation` 스킬의
consolidate-issue 모드를 사용하여 부모 design 문서에 통합한다.
통합은 "원래 하나였던 것처럼" 자연스럽게 병합한다 (참조 링크 아님).
통합 후 issues/[문제명]/ 디렉토리를 삭제한다.

### 중첩 금지

issues/ 내에서 추가 문제 발견 시:
- 같은 맥락 확장 → 현재 이슈의 phase를 리셋하여 재진행
  - 기존 phase 파일 삭제, 유효 결정사항을 새 Phase 1 컨텍스트로 이월
- 새로운 기능 필요 → 별도 기능 디렉토리로 격상 (feature 브랜치 전략)
- 무관한 문제 → 메모 후 별도 세션에서 처리

### 개발 중 문제 판단 플로우

```
테스트 중 문제 발생
  │
  ├─ 코드 수정만으로 해결? → 즉시 수정 (commit)
  │
```
