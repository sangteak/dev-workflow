# Brainstorming Workflow

브레인스토밍은 4개 국면을 순서대로 진행한다.
국면 전환은 Claude가 감지해서 제안하되, 강제하지 않는다.
모든 브레인스토밍은 Claude Code에서 진행한다 (파일 생성 필요).

---

## 국면 1: 탐색 (Exploration)

**목적:** 제약 없이 "무엇을 원하는가"를 끌어낸다.

**규칙:**
- 도메인 페르소나가 주도한다
- TD는 발언하지 않는다
- 요구사항이 모순되거나 비현실적이어도 일단 수용하고 기록한다
- Claude는 실현 가능성을 언급하지 않는다
- 한 번에 최대 2개 질문만 한다

**전환 감지 (하나라도 충족 시 제안):**
- 사용자가 "이정도면 될 것 같아", "다른 건 없을 것 같아", "이게 다야" 류의 발언
- 요구사항 목록이 윤곽을 갖춘 시점:
  "추가로 논의할 요구사항이 있으신가요, 아니면 다음 단계로 넘어갈까요?"

**전환 확정 시 즉시:**
`docs/design/[기능명]/phase1_exploration.md` 생성 — 이후 절대 수정하지 않는다.

포함 내용:
- 탐색된 요구사항 전체 목록 (확정/미확정 구분 없이 언급된 모든 것)
- 사용자가 명시적으로 제외한 항목과 이유
- 확정된 페르소나 구성

---

## 국면 2: 발견 (Discovery)

**목적:** 대화에서 드러나지 않은 요구사항의 빈틈을 찾아 채운다.

**규칙:**
- Claude가 지금까지 논의된 내용을 정리하고 미정의 영역을 명시한다
- 도메인 페르소나들이 서로 충돌/보완하면서 요구사항을 구체화한다
- 충돌 시 양측 입장을 명확히 제시하고 사용자가 결정한다
- TD는 여전히 침묵한다
- 한 번에 최대 2개 질문만 한다

**국면 2 시작 시 출력:**
```
── 현재까지 논의된 내용 ─────────────────────────────

확인된 요구사항:
- [목록]

아직 정의되지 않은 영역:
- [목록]

이 부분들을 하나씩 구체화하겠습니다.
────────────────────────────────────────────────────
```

**전환 감지 (하나라도 충족 시 제안):**
- 미정의 영역이 모두 논의됨
- 기술적 실현 가능성 검토가 필요한 시점
- 사용자가 "기술적으로 가능한지 보고 싶어" 류의 발언

전환 제안: "요구사항이 구체화되었습니다. TD를 불러서 기술적 실현 가능성을 검토할까요? [국면 3: 검증]"

**전환 확정 시 즉시:**
`docs/design/[기능명]/phase2_discovery.md` 생성 — 이후 절대 수정하지 않는다.

포함 내용:
- 국면 2에서 새로 발견된 미정의 영역 목록
- 각 항목에 대한 Q&A 결정 사항
- 페르소나 간 충돌이 있었던 항목과 최종 결정

---

## 국면 3: 검증 (Validation)

**목적:** TD가 최소한의 기술 가이드라인을 제공한다. 대안 강요 금지.

**규칙:**
- TD 페르소나가 합류한다 (도메인 페르소나 유지)
- TD는 구현 비용/난이도/리스크 언급만 한다. 대안을 강요하지 않는다
- 사용자가 리스크를 감수하겠다면 그 결정을 존중한다

**TD 발언 형식:**
```
── 🔧 TD 기술 검토 ──────────────────────────────────

[요구사항명]
판단: [기술적 판단]
가이드라인: [권장 방향 또는 주의사항]
리스크: 낮음 / 중간 / 높음

────────────────────────────────────────────────────
```

**전환 감지 (하나라도 충족 시 제안):**
- TD 검토가 모든 요구사항에 대해 완료됨
- 재협의 항목이 정리되고 최종 방향이 확정됨
- 사용자가 "정리해줘", "문서로 만들어줘" 류의 발언

전환 제안: "기술 검토까지 완료되었습니다. 지금까지의 논의를 설계 문서로 정리할까요? [국면 4: 정리]"

**전환 확정 시 즉시:**
`docs/design/[기능명]/phase3_validation.md` 생성 — 이후 절대 수정하지 않는다.

포함 내용:
- TD가 검토한 각 요구사항별 판단 결과 및 리스크 수준
- 재협의가 발생한 항목과 최종 결정
- TD가 제시한 기술 가이드라인

---

## 국면 4: 정리 (Consolidation)

**목적:** 전체 대화를 표준 설계 문서로 변환한다.

**규칙:**
- Claude가 전체 대화 기반으로 설계 문서 초안을 자동 생성한다
- 생성 후 사용자 확인을 받는다
- 수정 요청 시 반영 후 재확인한다
- 확정 시 `docs/design/[기능명]/[기능명].md` 파일을 직접 생성한다
  - `docs/design/[기능명]/` 디렉토리는 phase1 생성 시점에 이미 존재
- 파일 생성 완료 후 PLAN 단계로 연속 진행을 제안한다:
  "설계 문서가 `docs/design/[기능명]/[기능명].md`에 저장되었습니다.
   PLAN 단계로 바로 진행할까요?"
- PLAN 진행 시 Persona Resolution은 PLAN 페르소나만 새로 확정한다

**디렉토리 최종 구조:**
```
docs/design/[기능명]/
├── phase1_exploration.md    ← 국면 1 완료 시 생성, 불변
├── phase2_discovery.md      ← 국면 2 완료 시 생성, 불변
├── phase3_validation.md     ← 국면 3 완료 시 생성, 불변
└── [기능명].md              ← 국면 4 확정 시 생성, 최종 설계 문서
```

**표준 설계 문서 포맷:**
```markdown
---
feature: [기능명]
status: ready-for-plan
created: [날짜]
---

## CONFIRMED_REQUIREMENTS
- ...

## OPEN_QUESTIONS
- ...

## TECHNICAL_GUIDELINES
- ...

## PERSONAS_USED
- [목록]

## DECISIONS
- Q: [질문] → A: [결정]
```
