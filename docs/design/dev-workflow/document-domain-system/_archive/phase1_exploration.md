# Phase 1: Exploration — Document Domain System

## 배경

SearchTarget 기능 개발 중 Aggro 시스템의 상태 전이 정책을 변경해야 하는 상황 발생. 변경 내용을 기록할 적절한 문서를 찾지 못함. 기존 feature 단위(aggro-*, peek-*) 문서 구조가 크로스 feature 변경을 수용하지 못하는 문제 확인.

## 핵심 문제

1. **Stale 문서** — 다른 feature 작업으로 시스템이 변경되었으나 기존 feature 문서는 과거 상태를 기술
2. **정책 충돌** — feature별 독립 문서가 서로 다른 버전의 현실을 기술
3. **캐시 실패** — feature.md가 "코드베이스 캐시" 역할을 하도록 설계했으나, stale로 인해 결국 코드를 다시 읽게 됨. 문서 비용 + 코드 비용 이중 지불

## feature.md 용도 정리

- 기존 작업에 대한 정책, 결정, 결정 사유를 명확히 기록
- 유지보수 시 기존 문서를 읽어 활용 (전체 플로우, 정책, 예외 상황 등)
- 관리 파일 목록과 용도가 기입되어 코드베이스 파악에 도움
- 불필요한 토큰 낭비를 막고 정확한 정보를 제공

## 설계 문서의 정체성 토론

### 검토한 방향

| 방향 | 핵심 아이디어 | 트레이드오프 |
|---|---|---|
| **Eager 업데이트** | 변경 시점에 영향받는 문서 즉시 수정 | 현재 feature가 폐기될 수 있어 위험 |
| **Lazy 마킹** | 변경 사실만 표시, 나중에 반영 | 표시 누락 위험 |
| **문서 분리** | 불변(결정기록) + 가변(현재명세) | 문서 증가 |
| **문서 축소** | feature.md를 결정 기록으로 한정 | 원래 문제로 회귀 |

### 결론

- **Eager 기각** — 현재 feature가 완료되지 않은 시점에서 의존성 문서에 반영하는 건 위험 (폐기 가능성)
- **Lazy 마킹 + 문서 분리 결합** — 별도 pending 파일에 기록, feature 완료 시 통합
- 핵심 원칙: **"현재 작업이 완료되지 않은 시점에 의존성 feature에 내용을 반영하는 건 위험하다"**
