# Phase 2: Discovery — 입력 상호작용 일관성

> 생성일: 2026-03-17 | 불변 스냅샷

## 국면 2에서 새로 발견된 미정의 영역

1. 번호 목록 표준 포맷
2. Yes/No 질문 처리 방식
3. 자유 텍스트 입력과의 경계
4. 규칙의 위치 (어디에 명시할 것인가)
5. 선택지 5개 이상일 때 표현 규칙

## 각 항목에 대한 Q&A 결정 사항

### 1. 번호 목록 표준 포맷
- **결정:** 모든 닫힌 선택은 번호 목록으로 통일
- **형식:**
  ```
  [질문 텍스트]

  1. 선택지A
  2. 선택지B
  3. 선택지C
  ```

### 2. Yes/No 질문 처리
- **결정:** Yes/No도 번호화하여 완전 통일
- **형식:**
  ```
  [질문 텍스트]

  1. Yes
  2. No
  ```
- **근거:** 모든 선택 상황에서 "번호를 입력한다"는 단일 멘탈 모델 형성

### 3. 자유 텍스트 입력과의 경계
- **결정:** 번호가 있으면 번호 입력, 번호가 없으면 자유 입력
- **형식:**
  ```
  [질문 텍스트]
  > 예: "[예시 입력]"
  ```
- **근거:** 시각적 단서(번호 유무)만으로 입력 방식을 즉시 인식 가능

### 4. 닫힌 선택에서 직접 입력 탈출구
- **결정:** 0번을 "직접 입력" 예약 번호로 표준화
- **형식:**
  ```
  [질문 텍스트]

  1. 선택지A
  2. 선택지B
  0. 직접 입력
  ```
- **근거:** 닫힌 선택에서 열린 입력으로의 일관된 탈출구 확보

### 5. 선택지 5개 이상
- **결정:** 동일한 번호 목록 형식 유지, 개수 제한 없음
- **근거:** 텍스트 기반이므로 AskUserQuestion의 2~4개 제한이 없음

### 6. 규칙의 위치
- **결정:** `workflow-orchestrator`의 Output Format Rules 옆에 Input Format Rules 섹션 추가
- **근거:** orchestrator는 모든 세션에서 로드되므로 별도 참조 없이 항상 적용됨. 각 스킬 파일의 프롬프트 텍스트도 새 형식으로 일괄 수정

### 7. 금지 사항
- AskUserQuestion 도구 사용 금지
- 문자 선택(A/B/C/D) 금지 — 번호만 사용
- 슬래시 구분(Yes / No) 금지 — 번호 목록만 사용
- 괄호 안 선택지 나열 금지 — 번호 목록 형식 준수

## 페르소나 피드백 결과

### 주제 1: 번호 목록 포맷 + Yes/No 처리
- **미합의 → 사용자 결정:** Yes/No를 번호화할지 vs 별도 텍스트 형식으로 둘지
- **최종:** Yes/No도 번호화 (완전 통일)

### 주제 2: 자유 텍스트 경계 + 0번 예약 패턴
- **합의:** "번호 있으면 번호, 없으면 자유 입력" 기본 규칙
- **합의:** 0번을 "직접 입력" 예약 번호로 표준화

### 주제 3: 규칙의 위치
- **합의:** C안 채택 — workflow-orchestrator에 Input Format Rules 추가, 각 스킬 프롬프트도 일괄 수정
