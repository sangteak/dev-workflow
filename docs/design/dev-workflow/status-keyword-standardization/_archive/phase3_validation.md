## TD 기술 검토 결과

### REQ-1: 상태 키워드 정규값 통일
- 판단: 단순 명명 규칙. 기술적 제약 없음.
- 가이드라인: `ready-for-plan`, `complete` 2개만 정규값
- 리스크: 낮음

### REQ-2: 쓰기 시점 정규값 강제
- 판단: 스킬 텍스트 수정만으로 해결 가능
- 가이드라인: "plan.md 등 다른 소스에 명시된 값에 관계없이 반드시 `complete`를 사용한다" 우선순위 규칙 추가
- 리스크: 낮음

### REQ-3: 읽기 시점 방어적 수용 + 자동 수정
- 판단: 실현 가능. 전처리 분리 방식으로 리스크 완화.
- 가이드라인:
  - 허용 변형: `complete`, `completed` (2개만, 한국어 제외)
  - 인덱싱 본 로직 전에 "상태 정규화 전처리" 단계를 분리
  - 전처리: `completed` 발견 시 해당 파일 프론트매터를 `complete`로 수정
  - 기존 `status: complete` 필터링 로직은 변경하지 않음
- 리스크: 낮음 (전처리 분리로 완화)

### REQ-4: 기존 불일치 문서 수정
- 판단: 수동 편집 2건. 즉시 실행 가능.
- 리스크: 낮음

## 재협의 사항

### 한국어 상태값(`완료`) 제외
- 최초 제안: `complete`, `completed`, `완료` 3개 허용
- 최종 결정: `complete`, `completed` 2개만 허용
- 사유: 프론트매터는 시스템(Claude)이 쓰는 메타데이터이므로 한국어 값이 들어갈 가능성 없음

### 자동 수정 위치: 인라인 vs 전처리
- 최초 제안: 필터링 중 인라인 수정
- 최종 결정: 인덱싱 전 전처리 단계로 분리
- 사유: 기존 필터링 로직 불변, Claude 이행률 향상, 부수 효과 스킵 리스크 감소

## 페르소나 피드백 요약

### 주제: REQ-3 자동 수정 리스크
- **합의**: 전처리 분리 방식이 인라인보다 단순하고 이행률이 높다. 기존 로직 불변 원칙 유지.

### 주제: 한국어 허용 범위
- **합의**: 제외. 시스템이 쓰는 메타데이터에 한국어 값이 들어갈 가능성 없음.
