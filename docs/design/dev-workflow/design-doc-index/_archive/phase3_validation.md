# Phase 3: 검증 (Validation)

> 기능: design-doc-index (설계 문서 색인 및 크로스레퍼런스)
> 카테고리: dev-workflow
> 날짜: 2026-03-16

## TD 기술 검토 결과

### 1. 실시간 스캔 방식 (glob + 프론트매터 파싱)
- **판단:** 문제 없음
- **가이드라인:** `docs/design/**/*/*.md` glob 후 프론트매터 YAML 파싱 (파일 상단 `---` 블록만)
- **리스크:** 낮음

### 2. 키워드 매칭 (부분 매칭 + 카테고리/기능 양쪽 레벨)
- **판단:** 부분 매칭(substring) 필수. 카테고리 레벨과 기능 레벨 양쪽 검색
- **가이드라인:** glob 패턴 `*키워드*`로 카테고리명/기능명 동시 매칭
- **리스크:** 낮음 (glob 기본 도구 활용)

### 3. 다중 문서 전체 로드 — 단계적 로드 전략
- **판단:** 상한선 없음. 단계적 로드로 효율성 확보
- **가이드라인:**
  - 1~3건 매칭: 즉시 전체 로드 + 로드 안내
  - 4건 이상 매칭: 색인 모드로 목록 제시 → 사용자 선택 → 전체 로드
- **리스크:** 낮음 (단계적 로드로 컨텍스트 압박 해소)

### 4. brainstorming/plan-stage 자동 호출 통합
- **판단:** 스킬 문서에 트리거 규칙 텍스트 추가. 코드 수정 없음
- **가이드라인:** "사용자가 기존 설계 문서를 참조하려 할 때 design-doc-index 스킬을 invoke"
- **리스크:** 낮음

### 5. 스킬 문서 구조
- **판단:** `skills/design-doc-index/SKILL.md` 생성, `plugin.json` skills 배열 등록
- **가이드라인:** 기존 7개 스킬과 동일한 패턴
- **리스크:** 낮음

## 재협의 항목

### 카테고리 레벨 문서 통합
- **제안:** 개발 완료 후 카테고리 하위 기능 문서를 하나의 `[카테고리].md`로 통합
- **TD 판단:** 방향은 좋으나 design-doc-index의 범위를 넘어섬 (document-consolidation 확장 영역)
- **최종 결정:** 별도 피처로 분리. 현재 설계(부분 매칭 + 단계적 로드)는 향후 카테고리 통합 추가 시에도 호환됨
