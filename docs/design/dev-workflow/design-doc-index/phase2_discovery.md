> 카테고리: dev-workflow

## 국면 2에서 발견된 미정의 영역

1. 스킬 형태 (독립 vs 내장)
2. 키워드 매칭 로직
3. 다중 문서 로드
4. status 통일 방안
5. 워크플로우 통합 지점
6. 인덱싱 파일 관리 (추가 발견)

## Q&A 결정 사항

### 1. 스킬 형태
- **결정:** 독립 스킬(`design-doc-index`)로 분리
- brainstorming/plan-stage에서 사용자 발화를 감지하면 자동으로 호출하는 구조
- 로직 중복 방지 + 향후 DEVELOP 등 다른 단계 확장 가능

### 2. 키워드 매칭 로직
- **결정:** 1차 매칭은 디렉토리명(기능명) 기반
- `docs/design/*/[키워드]/` 패턴으로 매칭
- 프론트매터 별칭(aliases)/태그(tags) 확장은 향후 고려사항으로 남겨둠

### 3. 다중 문서 로드
- **결정:** 다중 문서 전체 로드 지원
- "재화, 인벤토리, 메일 시스템 참고해서 상점 설계하자" → 세 문서 모두 로드
- 로드 시 문서 수를 안내하여 사용자가 인지할 수 있게 함

### 4. status 통일 방안
- **결정:** 별도 통일 작업 불필요
- document-consolidation 스킬에서 이미 `status: ready-for-plan` → `status: complete`로 업데이트하는 규칙 존재
- `status: complete`를 색인 대상 기준으로 사용

### 5. 워크플로우 통합 지점
- **결정:** brainstorming/plan-stage 스킬에 트리거 규칙 추가
- 오케스트레이터 수정 불필요
- 사용자가 요청할 때만 동작 (자동 안내 없음)

### 6. 인덱싱 파일 관리
- **결정:** 별도 인덱스 파일을 생성하지 않음
- 트리거 시점마다 `docs/design/` 실시간 스캔
- 파일시스템 자체가 인덱스 역할
- 인덱스 깨짐/리인덱싱 문제 자체가 발생하지 않는 구조

## 페르소나 피드백 결과

### 주제: 스킬 형태
- **합의:** 독립 스킬 + brainstorming/plan-stage에서 자동 호출

### 주제: 키워드 매칭 로직
- **합의:** 디렉토리명 기반 1차 매칭. 별칭/태그는 향후 확장

### 주제: 다중 문서 로드
- **합의:** 지원. 컨텍스트 비용은 사용자 감수 트레이드오프, 로드 수 안내로 인지 제공

### 주제: status 통일
- **합의:** 기존 `status: complete` 규칙 활용, 추가 작업 없음

### 주제: 워크플로우 통합
- **합의:** brainstorming/plan-stage에 트리거 규칙 추가, 오케스트레이터 수정 불필요

### 주제: 인덱싱 파일
- **합의:** 실시간 스캔 방식, 별도 인덱스 파일 없음
