# issue-lifecycle 설계 문서

> 한 줄 요약: 구현 후 테스트하며 문제를 발견·해결·보완하는 유지보수 과정을 feature 사이클의 정식 일부로 만든다 — 실사용에서 버려진 issues/ 서브워크플로우를 "이슈 카드 경량 사이클 + 크리티컬 2경로"로 재설계한다.

## 1. 배경과 동기

issues/ 서브워크플로우는 사용자가 실사용했다가 결함으로 버린 기능이다 — 현재는 별도 feature 생성으로 우회 중이다. 실측된 결함:

- issue 대응 후 불변 문서(phase/plan)가 갱신되지 않아 계획과 구현이 어긋난 채 남음 (plan.md 갱신 경로 전무, "phase 불변 vs 이슈발 변경" 충돌 미규정)
- 이슈 A 테스트 중 발견된 결함 B가 분류 없이 A 안에서 함께 처리 → feature 오염·제어 불능 (형제 이슈 무규정, 3분기 규칙은 발동 장치 부재로 사문화)
- 우회의 아픔: 같은 워크트리 작업인데 새 작업 초기화 비용 + 원 문서와 단절
- 간단한 문제를 말하면 설계 검토 없이 즉시 코드부터 나옴
- issues 서술 자체가 v1.13 이후 본류 진화(질문 규범·게이트·역할 내장·시드 상시)에서 격리 방치 (11개 파일 ~165줄)

사용자 핵심 비전: **구현 후 테스트·보완까지가 하나의 feature 사이클** — 그 과정이 끝나야 feature가 마무리된 것이다.

## 2. 목표와 비목표

### 목표
- GOAL-001: 유지보수(문제 발견→분류→해결→문서 반영)를 feature 사이클의 정식 일부로 — 우회 없이 한 흐름으로 종결
- GOAL-002: "말하면 즉시 코드" 차단 — 경미한 문제에도 가벼운 설계 검토(분석 카드)가 선행되는 구조적 강제
- GOAL-003: "이슈 한 번에 하나" — 발견 순간에 작동하는 분류·대기 장치로 다중 이슈 오염 재발 방지
- GOAL-004: 문서 진실 계층 확립 — 설계 문서 SSOT·phase 동결 스냅샷·plan 영향분 갱신, 변경 즉시 반영

### 비목표
- 다중 이슈 병행 처리 지원 — 병행은 요구가 아니라 사고였음 (목표는 "하나씩 + 즉시 분류")
- 발화 키워드 감지 트리거 — 표현이 무한히 다양해 신뢰 불가 (사용자 직접 기각)
- 이슈의 정규 4국면 브레인스토밍 유지 — 무거움이 우회의 원인 (경량 사이클로 대체)
- phase 파일 불변 원칙 폐기 — "국면 완료 시점의 동결 스냅샷"으로 의미 재정의만 (조사: 원 근거는 최초 커밋에 무기재 공리, 실질 근거인 /clear 복구는 "완료 시점 동결"만 요구)
- 제3의 status 값 신설 — 기각 확정된 결정의 재론 (닫기는 `_archive/` 이동으로)
- 신규 코드/MCP 도입 · 도메인 문서 2건(document-management·session-management)의 issues 절 갱신 — 본 기능의 도메인 머지 시점에 수행 (지배 관행)

## 3. 확정된 요구사항

- REQ-001 (이슈 카드): `issues/NNN-[문제명].md` 단일 파일, 5필드 — 심각도(경미/중간/크리티컬)·증상은 대기 시점 기록(2줄 비용), 원인·영향 범위·수정 방향은 활성화 시점 기록. **구조가 곧 상태**: 증상·심각도만=대기 / 분석 절 채워짐=활성(항상 1장) / 파일 없음=해소. frontmatter·상태값·phase·seed·PLAN·HANDOFF 없음 — HIGH
- REQ-002 (경량 3박자 사이클): ① 분석 — 카드 작성, 카드의 "수정 방향"이 수정 작업의 입력 프롬프트(구조적 강제 — 카드 없이 수정 시작 불가) ② 수정 — 같은 워크트리에서 구현+테스트 ③ 반영 — 부모 §10 요약 1~2줄 + 설계 본문은 설계가 실제로 바뀐 경우만 + plan은 남은 태스크 영향 시만 → 카드 삭제(git-mode 자동+고지 / no-git 확인 — v1.15 조건부). 반영 검증 항목: "분석이 수정에 선행했는가" — HIGH
- REQ-003 (행동 기반 트리거): "DEVELOP 중, 현재 plan 태스크 범위 밖의 결함 수정을 위해 코드 수정 도구를 사용하려는 시점 → 이슈 카드 생성 선행". 강제 3겹: bootstrap 불변식 1줄(~80B) + orchestrator DEVELOP 규칙 상세(예방) / 카드=수정 입력물(진행 차단) / 리뷰 검증 항목(사후 적발) — HIGH
- REQ-004 (발견 경로 경계 — 태스크 기준): 현재 태스크 범위 내 발견 = SDD 기존 fix 루프(카드 없음, 외부 위임 경계 존중) / 태스크 경계 밖(완료 태스크 회귀·범위 밖 부작용·Evaluator FAIL·사용자 리포트) = 카드. 서브에이전트 리포트 수신 처리에 "범위 밖 발견물 → 대기 카드" 명문 (orchestrator가 SDD에 전달하는 컨텍스트 규칙로 — Superpowers 본문 수정 불가) — HIGH
- REQ-005 (대기열 = 카드): 별도 목록 파일 없음. 번호 접두가 순서 고정(mtime 불신), FIFO 추천 + 심각도 새치기 허용. 활성 카드 존재 중 신규 태스크 디스패치 보류 — HIGH
- REQ-006 (크리티컬 2경로): 심각도 "크리티컬" 기록 = 판별문 발동 조건. 판별문 **"설계 문서 §2(목표·비목표)가 그대로 유효한가?"** — Yes → 제자리 재계획(기본) / No → 아카이브+승계(예외). **감지·분류 제안은 자동, 경로 전환 실행은 확인 게이트** (3단 트리 1번 비가역) — HIGH
- REQ-007 (제자리 재계획): 같은 feature·워크트리·브랜치 유지 — ① 설계 문서 §4/§6(폐기 대안 기록)/§10 갱신 + 사용자 리뷰 ② plan-stage "재계획 진입"(명시 라우팅 예외 경로 — 자동 감지 아님): 변경분 Feasibility + 기존 plan 차분 분류(유효/폐기/신규 태스크 — 워크트리 커밋과 문서 정합) ③ writing-plans 재호출. decision-flow 국면 대응표 1행 추가 — HIGH
- REQ-008 (아카이브+승계): 원 문서 §9/§10 중단·승계 기록 → 디렉토리 `docs/design/[카테고리]/_archive/[기능명]/` 이동(기존 세션 목록 제외 메커니즘 재사용) → 새 feature 브랜치는 원 워크트리 브랜치 HEAD에서 분기(커밋 히스토리 승계, 무효 구현은 revert 기록, 원 워크트리 정리) — MEDIUM
- REQ-009 (Completion 접점): 마무리 시퀀스 Step 1 앞에 미해소 카드 잔존 검사 — 잔존 시 목록 제시 후 처리/의식적 보류 확인 ("보완이 끝나야 마무리"의 강제 지점) — HIGH
- REQ-010 (복구 접점): context-handling 수집 규칙에 카드 추가(필수 — 현행 파일명 고정 수집이라 미수집). 활성 카드는 부모 하위 들여쓰기 표시+선택 시 재개, 대기 카드는 카운트만. 신형 카드는 HANDOFF 불요, is-issue 규칙은 구형 전용화 — HIGH
- REQ-011 (Mode 2·3 폐기): document-consolidation Mode 2 삭제(반영은 3박자에 흡수, 포인터 잔존) + Mode 3 삭제(선행: _pending 소비 경로를 merge-to-domain에 이관 — 명시 조건 충족) → Mode 1 단일 — MEDIUM
- REQ-012 (merge-to-domain 보강): feature 삭제 절차에 issues/ 잔존 검사(미해소 카드·구형 이슈 발견 시 경고+확인) + _pending 소비 절 — MEDIUM
- REQ-013 (완료 후 문제 경계선): "feature가 열려 있으면 issues 카드, 닫혔으면(complete/머지됨) 별도 feature" — 기존 설계·도메인 문서를 참조 컨텍스트로. 발생 시점 정의 불일치 해소 — MEDIUM
- REQ-014 (구형 호환): 구형 이슈 디렉토리 공존 허용 + 세션 탐색 발견 시 안내 1줄 ("구 형식 — 이어서 마무리하거나 정리"). 자동 변환·이중 지원 없음 — MEDIUM
- REQ-015 (구 서술 정비): issues 서술 11개 파일 처분(phase3 실측 목록이 체크리스트) + "hotfix·핫픽스" 어휘 전면 제거(은유 폐기 — 실제 동작 서술로 대체). 규범 변경 스윕 절차 적용(범위: skills/ 밖 README·commands·CLAUDE.md까지) — HIGH
- REQ-016 (검증): 게이트·카드 시나리오 드라이런 + 확정 문면 grep + 구 어휘 스윕 — HIGH

## 4. 설계 개요

**이슈 생애주기 (전체 그림):**

```
DEVELOP 중 문제 인지 (발화·리포트·테스트)
  │
  ├─ 현재 태스크 범위 내? → SDD 기존 fix 루프 (카드 없음)
  │
  └─ 범위 밖 → 이슈 카드 생성 (심각도+증상 2줄 — 대기)
       │    ※ 트리거: 계획 밖 결함 수정에 코드 도구를 잡는 순간, 카드 선행 (bootstrap 불변식)
       │
       ├─ 활성 이슈 있음 → 대기 카드로 적재 (FIFO 추천·심각도 새치기)
       │
       └─ 활성화: 원인·영향·수정 방향 기록 (분석)
            │
            ├─ 심각도 경미/중간 → 경량 3박자: 수정 → 반영(§10 1줄+조건부) → 카드 삭제
            │
            └─ 심각도 크리티컬 → 판별문 "§2 유효?" (분류 자동·전환은 확인 게이트)
                 ├─ Yes → 제자리 재계획 (설계 갱신·리뷰 → plan-stage 재계획 진입 → writing-plans)
                 └─ No  → §9/§10 기록 → _archive/ 이동 → 원 브랜치에서 분기해 새 feature

닫힌 feature의 사후 문제 → 별도 feature (issues 아님 — 경계선)
Completion 진입 → 미해소 카드 검사 (잔존 시 처리/보류 확인)
```

**문서 진실 계층:** 설계 문서([기능명].md) = 단일 진실(변경 즉시 반영) / phase 파일 = 국면 완료 시점 동결 스냅샷 / plan.md = 살아있는 실행 원장(영향분만 갱신) / 이슈 카드 = 임시 과정 로그(해소 시 삭제 — SSOT 이중화 금지).

## 5. 의존성 맵

| 컴포넌트 | 의존 대상 | 영향받는 컴포넌트 |
|----------|-----------|------------------|
| 행동 트리거 (REQ-003) | bootstrap 불변식·orchestrator DEVELOP 규칙 | 카드 생성, SDD 위임 컨텍스트 |
| 이슈 카드 (REQ-001) | — | 경량 사이클·대기열·복구·Completion 검사 |
| 크리티컬 2경로 (REQ-006~008) | 카드 심각도 필드, plan-stage·_archive 메커니즘 | orchestrator 라우팅, decision-flow 대응표 |
| Mode 2·3 폐기 (REQ-011) | 경량 사이클(반영 흡수), merge-to-domain(_pending 이관) | document-consolidation, Completion 무영향 |
| 복구 접점 (REQ-010) | context-handling 수집 규칙 | 세션 시작 목록, 구형 호환(REQ-014) |

## 6. 기술 결정 및 대안 검토

| 결정 사항 | 선택 | 근거 | 검토한 대안 | 기각 사유 |
|-----------|------|------|-------------|-----------|
| 크리티컬 처리 | 판별문 + 2경로 (제자리 재계획 기본 / 승계 예외) | 두 아픔(초기화 비용·문서 단절)은 갈아타기 자체의 비용 — 정체성 유지 시 안 떠나는 게 정답 | 항상 닫고 새 feature (원안) | 자기유발 비용, status 표현 불가 (3갈래 전부 소비자 오작동) |
| "닫기"의 실체 | `_archive/` 이동 | 세션 목록 제외 메커니즘 기존재 — 새 규범 0 | 제3의 status 값 | 기각 확정 결정의 재론 + 소비 스킬 전부가 모르는 값 |
| 경미 이슈 흐름 | 경량 3박자 (카드→수정→반영) | 무거움이 우회의 원인 — 고민은 강제하되 의식은 제거 | 정규 4국면 유지 | phase·페르소나 토론이 경미 수정에 과함 (실증) |
| 검토 강제 방식 | 구조적 강제 (카드=수정 입력물) + 검증 항목 | 경고문 단독은 2릴리스 연속 실패 실측 — 데이터 흐름 강제는 게이트 신설도 아님 | 순서 규칙 산문 | 세 번째 경고문 — 관성 최대 지점에서 무력 |

## 7. 제약조건과 가정

- 다중 모델 하드 제약: 판단형 규칙 금지 — 트리거·판별문·경계 전부 예/아니오 판정형
- 확정 문면(카드 템플릿·트리거 문면·판별 게이트·불변식 1줄·판단 플로우 이전분)은 plan.md에 완성형 수록 — 구현은 verbatim 치환만
- Superpowers 본문 수정 불가 — 리포트 수신 규칙은 orchestrator 위임 컨텍스트로만
- 가정: 행동 트리거의 판정("계획 밖 수정인가")은 plan.md 대조로 기계적 — 발화 해석보다 신뢰 가능 (도그푸딩 관측 대상)
- 가정: 경미 카드의 왕복 2회·쓰기 1줄 비용이면 사용자가 우회 대신 이 경로를 쓴다 (시뮬레이션 승인 기반 — 실사용 관측 대상)
- 가정: 활성 카드 중 태스크 보류가 체감 지연을 만들지 않는다 (경미 카드는 짧다는 전제)
