## 1. TD 기술 검토 결과 (요구사항별)

### [A] 요구사항 1 — 5단계 머지 알고리즘 (intelligent)

**판단:** 실현 가능. Markdown 플러그인이 추론을 하는 것이 아니라 SKILL.md 프롬프트로 호스트 Claude 추론을 가이드. feature-dev/Compound Engineering 등이 검증한 패턴. 단, 머지는 SSOT를 **변형**하는 작업이므로 비결정성 위험이 직접적임.

**핵심 위험:**
- 자유 서술 요약은 비결정성 — 같은 입력에 다른 머지 계획
- domain.md가 커지면 dry-run diff 60줄로 인간이 의미 보존 검증 불가
- 인지 한계로 사용자가 dry-run rubber-stamping할 위험

**가이드라인:**
- 학습 산출물을 **structured digest** (ID 기반 구조화)로 강제
- domain.md 토큰 ≥ 80K이면 섹션 단위 분할 학습 fallback
- 검증 페르소나(5)는 🏛️ Architect와 별개 페르소나가 담당 (🔧 Tech Lead 또는 📋 PM)
- 검증 체크리스트 구체화: 정책 ID 보존, supersede 없는 정책 누락 X, 변경 이력 행 추가 확인, 라인 수 변화율 ±50% 이내

**리스크: 높음** (Structured digest 안전장치 필수)

### [B] 요구사항 2 — Architect 라운드 적응형 (1~3)

**판단:** 설계 건전. 단, "Architect 자율 종료"는 자기 신뢰 편향 위험.

**가이드라인:**
- 라운드 등급을 **외부 분류 규칙**으로 고정 (Architect 진입 전 결정)
- 분류표:
  - 자동 수정 only + 사용자 결정 0 + append만 → 1라운드
  - 사용자 결정 1~2 + 의미 충돌 0 → 2라운드
  - 의미 충돌 ≥ 1 OR 의존성 재정의 OR 기존 결정 무효화 → 3라운드
- 1라운드 종료 시 출력 템플릿 강제, "발견된 우려" 비어있지 않으면 자동 2라운드 승급
- `--review-merge=N` 플래그는 **정확값** (디버깅 모드)으로 해석

**리스크: 중간** (외부 분류 규칙 적용 시 낮음으로 강하)

### [C] 요구사항 3 — 자동 수정 vs 사용자 결정 분류

**판단:** 분류 자체는 합리적. 단, "자동 수정" 항목의 **사일런트 손상** 위험 존재. 특히 ID renumbering 시 참조 갱신 누락.

**가이드라인:**
- dry-run에서 자동 수정도 **회계 형식 노출** (수정 전/후 + 참조 갱신 위치):
  ```
  자동 수정 (renumbering):
  - feature.REQ-001 → domain.REQ-008
    참조 갱신: feature.md 섹션 4 (2건), 섹션 6 (1건)
  ```
- "표 형식 통일"은 컬럼 헤더 동일할 때만 자동, 헤더 다르면 사용자 결정으로 강등
- 분류 애매한 경우 → **사용자 결정으로 fallback** (보수 원칙)
- 검증 체크리스트: ID renumbering 시 모든 인용 갱신 확인
- 푸시 전 dry-run 재확인 UX를 SKILL.md에 강제

**리스크: 높음** (회계 형식 dry-run + 보수 fallback 안전장치 필수)

### [D] 요구사항 4 — 카테고리 간 병렬 풀 max 5

**판단:** 풀 크기 3~5는 dev-workflow brainstorming Enhanced Mode에서 이미 검증된 동시성 범위. 단, 머지는 brainstorming보다 무거운 작업 (학습 + 쓰기 + 인터랙션).

**위험:**
- 5개 카테고리 병렬 중 3개에서 동시 사용자 결정 필요 → 큐 충돌
- 카테고리별 결과 도착 순서 비결정적 → 사용자가 결과 추적 어려움

**가이드라인:**
- **풀 크기 기본 3** (상한 5), `MERGE_POOL_SIZE` 환경 변수로 조정
- **사용자 결정 큐 직렬화**: 한 번에 하나의 결정 질문만 노출. 결정 대기 카테고리는 일시 정지, 다른 카테고리는 계속 진행
- **출력 직렬화**: 카테고리 결과를 buffer로 모은 뒤 카테고리 시작 순서대로 출력 (실시간 인터리브 금지)
- 카테고리 시작 순서: git-mode = main 머지 순서 / no-git-mode = mtime
- 한 카테고리 abort 시 다른 카테고리 영향 X (독립 abort)

**리스크: 중간** (운영 정의 위험 — 메커니즘 자체는 검증됨)

### [E] 이월 1 — 사용자 사전 승인 발화 패턴

**판단:** LLM 의도 분류는 비용/비결정성 문제. 플래그 + 키워드 확인 하이브리드 권장.

**가이드라인:**
- **1차 정문: `--auto` 플래그** (다른 플래그 `--no-review`, `--review-merge=N`와 일관)
- **2차 보조: 키워드 + 확인 프롬프트**:
  - 키워드 매칭 시 즉시 사용자에게 1차 confirm 발생 ("자동 모드로 전환하시겠습니까? 1. Yes 2. No")
  - 인식 키워드 목록을 SKILL.md에 표 형식 명시 (한국어/영어 5~10개)
- **자동 모드 의미 경계**: "자동 수정 항목의 사후 확인 생략"만. 사용자 결정 항목은 여전히 질문.
- **영속성**: 세션 종료 시 자동 리셋. 파일 저장 없음.

**리스크: 낮음**

### [F] 이월 2 — 기존 도메인 호환성

**판단:** 마이그레이션 별도 작업 만들지 않음 권장. 첫 머지에서 점진적 보완이 안전.

**가이드라인:**
- 별도 `/dev-workflow:migrate-domains` 명령 X (신규 스킬의 가장 위험한 첫 사용 시나리오 회피)
- 첫 머지에서 점진 도입:
  - ID 체계 부재 → 사용자 결정 (자동 부여 vs 보존)
  - 섹션 10 부재 → 시작점 행만 추가 (소급 기록 금지)
  - 의존성 맵 부재 → 머지 필요 시만 질문
- **호환성 체크리스트**를 SKILL.md에 명시:
  | 항목 | 부재 시 처리 |
  |---|---|
  | 섹션 10 변경 이력 | 시작점 행만 추가하고 진행 |
  | 정책 ID | 첫 머지에서 사용자 결정 (부여 vs 보존) |
  | 의존성 맵 | 머지에 필요한 경우에만 질문 |
  | frontmatter | 없어도 진행 |
- 첫 머지 발동 정책은 **해당 도메인에서 1회만** 발생.

**리스크: 낮음**

## 2. TD 종합 권고 (3건) — 모두 수용 확정

### 권고 1: Structured Digest 스키마 정의

**의도:** LLM 추론 비결정성을 머지에서 제거.
**목적:** 머지 계획을 "자유 텍스트 비교"가 아닌 "ID 집합 연산"으로.

**스키마 예시:**
```yaml
domain_digest:
  policies:
    - id: POL-001
      statement: "큐잉 우선순위는 player MMR 기준"
      source_section: "4. 설계 개요"
      immutability_level: high | medium | low
  decisions:
    - id: DEC-001
      statement: "Redis pub/sub 사용"
      supersedes: null
  requirement_ids: [REQ-001, REQ-002, ...]
  section_index:
    "1. 배경": [1, 15]
    "4. 설계 개요": [50, 120]
```

**적용 시점:**
- 정의: Phase 4 (설계 문서) → DEVELOP (SKILL.md 작성)
- 적용: 실사용 시 매 머지의 (1)(2) 학습 단계

### 권고 2: Architect 라운드 등급 외부 분류 규칙

**의도:** Architect 자기 신뢰 편향 차단.
**목적:** 라운드 수가 머지 계획의 객관적 특성으로 결정.

**규칙 표 + 1라운드 출력 템플릿** (위 [B] 가이드라인 참조).

**적용 시점:**
- 정의: Phase 4 → DEVELOP (SKILL.md 작성)
- 적용: 실사용 시 매 머지의 (3) 머지 계획 단계 직후, Architect 진입 전

### 권고 3: 실측 호환성 검증 (PLAN 중 1회)

**의도:** 구현 전 실제 데이터로 호환성 위험 사전 발견.
**목적:** "이론적으로 동작" → "실측에서 동작"으로 검증.

**적용 시점:** PLAN 단계 중 1회만.

**산출물:** plan.md에 "가상 머지 호환성 검증" 섹션 추가
- 대상: `docs/design/dev-workflow/document-management.md` (또는 다른 기존 도메인)
- 가상 feature 작성 + 5단계 알고리즘 시뮬레이션
- 발견된 호환성 이슈 → SKILL.md 작성 시 반영

**PLAN의 success_criteria에 추가:**
> "PLAN 단계에서 기존 도메인 1개에 대한 가상 머지 dry-run을 1회 실행하여 호환성 위험을 식별한다."

## 6. 핵심 위험 분포 (Phase 3 결론)

| 위험 영역 | 리스크 | 안전장치 |
|---|---|---|
| LLM 비결정성으로 SSOT 변형 (요구사항 1) | 🔴 높음 | Structured digest + 검증 페르소나 분리 + 토큰 fallback |
| 자동 수정 사일런트 손상 (요구사항 3) | 🔴 높음 | 회계 형식 dry-run + 보수 fallback + 검증 체크리스트 |
| Architect 자기 신뢰 편향 (요구사항 2) | 🟡 중간 → 🟢 낮음 | 외부 분류 규칙 + 1라운드 템플릿 강제 |
| 병렬 풀 UX 무너짐 (요구사항 4) | 🟡 중간 | 사용자 결정 큐 직렬화 + 출력 직렬화 |
| 사용자 사전 승인 발화 (이월 1) | 🟢 낮음 | `--auto` 플래그 + 키워드 보조 |
| 기존 도메인 호환성 (이월 2) | 🟢 낮음 | 마이그레이션 X + 점진 보완 |

**잔존 위험:** Structured digest 적용 후에도 머지 시 의미 보존을 인간이 모두 검증하기 어려운 인지 한계는 본질적으로 남는다. 이는 (5) 검증 단계의 자동 체크리스트와 git 위임 롤백으로 부분 완화. 사용자에게 "도구의 한계" 인지를 SKILL.md 본문에 명시 권장.
