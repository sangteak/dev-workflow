# persona-feedback-loop 설계 문서

> 한 줄 요약: brainstorming 스킬에 구조화된 페르소나 피드백 루프를 추가하여, 매 주제마다 체계적인 다관점 검증을 수행한다.

## 1. 배경과 동기
- workflow-orchestrator가 "persona-based feedback loops"를 선언하지만, 실제로는 plan-stage에만 구현되어 있음
- brainstorming 스킬에는 "충돌/보완" 기술만 존재하며, 정형화된 피드백 루프가 부재
- brainstorming 단계에서도 체계적인 페르소나 피드백이 필요

## 2. 목표와 비목표
### 목표
- GOAL-001: brainstorming 스킬에 독립적인 페르소나 피드백 루프 메커니즘 추가
- GOAL-002: brainstorming의 발산적 성격을 보존하면서 비판적 검증 수행

### 비목표
- plan-stage 피드백 루프와 통일/공유하지 않음 (각 스킬 독립 유지)
- persona-resolution의 기본 페르소나 범용화 (별도 이슈로 분리)

## 3. 확정된 요구사항

### 피드백 루프 메커니즘
- REQ-001: 매 주제/질문마다 피드백 루프 발동 — 우선순위: HIGH
- REQ-002: 최대 3회 루프 제한, 초과 시 사용자 승인 후 추가 토론 — 우선순위: HIGH
- REQ-003: 라운드 카운터 `[라운드 N/3]`을 출력에 포함 — 우선순위: MEDIUM

### 비판적 관점
- REQ-004: 라운드 로빈으로 비판적 관점 할당. 라운드 시작 시 비판적 역할 페르소나를 명시한다 — 우선순위: HIGH
- REQ-005: 비판적 사고 비율 — 비판 60% / 건설적 40% — 우선순위: MEDIUM
- REQ-006: 비판 시 반드시 대안 또는 출구를 함께 제시 — 우선순위: HIGH

### 토론 리드
- REQ-007: 매 주제 시작 시 사용자에게 토론 리드를 묻는다 (선택지에 "없음" 포함) — 우선순위: HIGH
- REQ-008: 리드 선택 시 선정 이유 한 줄 명시, 비판적 비중 약간 높게 — 우선순위: MEDIUM
- REQ-009: "없음" 선택 시 모든 페르소나 동등 참여 (기본 비율) — 우선순위: MEDIUM
- REQ-010: 연속 2회 리드 제한 — 우선순위: MEDIUM
- REQ-011: 구체적 비율 수치는 스킬 지침에 미포함, 설계 문서에만 기록 (~65:35 / ~55:45) — 우선순위: LOW

### 출력 형식
- REQ-012: 의견 본문은 자유 형식 (페르소나 이모지 + 이름 접두사) — 우선순위: HIGH
- REQ-013: 라운드 종료 시 `**합의:**` 또는 `**미합의:**` 마커 + 결론 요약 — 우선순위: HIGH
- REQ-014: 정형 태그(찬성/반대 등) 사용하지 않음 — 우선순위: MEDIUM

### 국면별 방향성
- REQ-015: 국면별 강제 비율 없음, 방향성 가이드로 자연 조절 — 우선순위: MEDIUM
  - 국면 1: 비판보다 호기심으로
  - 국면 3: 리스크를 놓치지 말 것
- REQ-016: 자유로운 분위기 유지, brainstorming 발산적 성격 훼손하지 않음 — 우선순위: HIGH

### 기록
- REQ-017: phase 파일에 합의/미합의 결론만 기록 (토론 과정 제외) — 우선순위: HIGH
- REQ-018: 주제별 그룹핑 형식으로 기록 — 우선순위: MEDIUM

## 4. 설계 개요
- `brainstorming/SKILL.md`에 피드백 루프 섹션을 추가
- 기존 국면 2(발견)의 "충돌/보완" 규칙을 피드백 루프 메커니즘으로 확장
- 각 국면의 "포함 내용"에 "페르소나 피드백 결과" 섹션 추가
- plan-stage와 독립적으로 동작, 의존성 없음

## 6. 기술 결정 및 대안 검토
| 결정 사항 | 선택 | 근거 | 검토한 대안 | 기각 사유 |
|-----------|------|------|-------------|-----------|
| plan-stage와의 관계 | 별도 유지 | 스킬 간 의존성 방지, 독립 진화 가능 | 통일/공유 규칙 | 한쪽 수정 시 동기화 문제, 커플링 |
| 출력 형식 | 자유 형식 + 마커 | brainstorming 발산적 성격 보존 | 정형 구분선 (plan-stage 방식) | 브레인스토밍 흐름 저해 |
| 토론 리드 선정 | 사용자 선택 (없음 허용) | 사용자 주도권 보장, 규칙 단순화 | 자동 매칭 / 고정 리드 | 매칭 로직 복잡 / 관점 편중 |
| 동기화 방법 | 없음 (각자 독립) | 의존성 최소화 | 주석 표기 / 공유 파일 | 불필요한 커플링 |

## 7. 제약조건과 가정
- Claude가 세션 내 대화 컨텍스트로 라운드 수와 이전 리드를 추적할 수 있다고 가정
- 컨텍스트 압축 시 추적 정확도가 낮아질 수 있으나, 라운드 카운터 출력으로 보완

## 8. 기술 가이드라인
- 프롬프트 엔지니어링 수준의 수정 (코드 변경 없음)
- 기존 brainstorming 스킬 구조를 최대한 유지하면서 섹션 추가
- 페르소나 전문 영역 한 줄 설명 기록 권장 (필수 아님)

## 9. 구현 결과 및 일탈 사항

- 국면 2 규칙에서 "충돌/보완" → 피드백 루프 참조로 대체
- phase1/2/3 포함 내용에 "페르소나 피드백 결과" 항목 추가
