# gate-effectiveness 설계 문서

> 한 줄 요약: 게이트(사용자 확인 지점) 전체를 "제 역할을 하는가" 기준으로 심사해 — 등급이 틀린 확인은 바로잡고, 형식만 남은 확인은 안전망과 함께 걷어낸다.

## 1. 배경과 동기

로드맵 원안(G5)은 "게이트 배칭 — 불필요한 확인 절차 축소"였다. 브레인스토밍의 본질 질문과 조사로 원안의 전제가 낡았음이 확인됐다:

- 원안이 지목한 "문서 취합 확인 5회"는 이미 2곳뿐 (이전 릴리스들에서 해소된 낡은 수치)
- 사용자는 번호 응답 UI로 게이트가 많아도 크게 불편하지 않음 ("단, 적응의 결과일 수 있음")
- "묶어 보여주기(배칭)"는 명시적 불원 — 직전 릴리스(conversation-ux)의 "하나씩" 원칙과 충돌하는 방향

대신 조사에서 실제 결함 2유형이 발견됐다: ① **등급 오류** — 푸시(원격 공개, 고위험)가 커밋(로컬, 중위험)과 "커밋+푸시를 진행할까요?" 한 확인에 결합 ② **형식적 게이트** — 확인은 있으나 실효가 없는 것들(검증 재료를 주지 않는 학습 게이트, 가벼운 코멘트로도 강제되는 토론 라운드, 근거 없는 에이전트 재투입). G5는 **게이트 실효성 정비**로 재정의되었다.

## 2. 목표와 비목표

### 목표

- GOAL-001: 게이트 전체 목록(34행)을 3단 결정 트리로 심사해 등급·처분·근거를 확정 기록
- GOAL-002: 등급 오류 교정 — 무게가 다른 행동(커밋/푸시)에 각자 등급에 맞는 확인
- GOAL-003: 형식적 게이트 3건의 처분 — 검증 불가한 확인은 걷어내고 안전망으로 대체, 뻔한 라운드는 조기 종료
- GOAL-004: 재발 방지 — 게이트 신설 시 같은 기준(3단 트리)을 통과하게 하는 상시 필터를 남김

### 비목표

- 배칭 UI (사용자 명시 불원 + conversation-ux "하나씩" 규범 충돌)
- 게이트 수 줄이기 자체 (기준은 오직 "역할을 하는가" — 실제 제거 판정 0건)
- 원안의 "5회→2게이트" (낡은 전제로 폐기)
- SemVer 일반 원칙 신설 (이번 결과만 보고 — CLAUDE.md 개정으로 확대 금지)
- document-consolidation Mode 3 (폐기 예정 — 심사 배제)
- conversation-ux 문면 스켈레톤 재정의 (기존 스켈레톤 100% 내에서만 작업)
- 도메인 문서의 게이트 서술 갱신 (본 기능의 도메인 머지 시점에 수행)

## 8. 기술 가이드라인

1. **확정 문면 사전 고정**: 개정 문면(고지 1줄·판별문·분리 게이트·신설 2절)은 plan.md에 완성형 수록 — 구현은 verbatim 치환만
2. **구 독트린 전수 스윕**: "생략할 수 없다"·"학습 게이트"·"커밋+푸시" 계열 grep 목록화 → 잔존 문장 전수 정렬 (conversation-ux 교훈 — 무변경 줄이 태스크 리뷰 사각)
3. **차단성 판별문**: "이 우려가 반영되지 않으면 dry-run 승인을 진행할 수 없는가?" — Yes=차단(라운드 증가), No=권고(회계 1줄)
4. 커밋/푸시 분리와 bootstrap.md 동기화는 같은 태스크 (폴백 누락 선례)
5. 3단 트리 문면을 development-principles 신설 절에 그대로 수록 — 이번 심사와 미래 심사가 같은 기준
6. 검증: 게이트 전후 대조 목록 + 확정 문면 grep + 드라이런 1회 — 신규 인프라 금지
7. 릴리스: MINOR (docs/design 구조 무변경, 게이트 동작 변경은 하위 호환 — 확인이 줄어드는 방향이며 안전망 동반)
