# conversation-ux 설계 문서

> 한 줄 요약: dev-workflow가 직접 진행하는 구간의 대화 방식을 "하나씩 · 보이는 것만 · 쉬운 말 · 깨지지 않는 형식"으로 통일한다.

## 1. 배경과 동기

v1.13.0 실사용(G4 도그푸딩 세션)에서 사용자가 스크린샷 기반으로 6가지 대화 UX 불편을 보고했다:

| # | 이름 | 요약 |
|---|---|---|
| 1 | 결정 연극 | 번호 선택지 목록을 출력해 놓고 모델이 스스로 선택해 진행 |
| 2 | 초대-대기 불일치 | "일괄로 답하셔도 됩니다" 직후 "검토 중이니 대기" — 존재하지 않는 결정에 선답 유발 |
| 3 | 박스 깨짐 | 괘선(┌│└)이 한글 이중폭+터미널 렌더링에서 정렬 불가 |
| 4 | 용어 장벽 | 내부 작업 어휘로 압축된 문면 — 판단 곤란. 초심자용 풀이에 만족, 기본값화 요청 |
| 5 | 질문 일괄 제시 | 결정은 "하나씩" 규범이 있으나 질문은 일괄 — 비대칭 |
| 6 | ID 불투명 참조 | 문서 내부 ID(REQ-001 등)를 대화에서 단독 지칭 — 맥락 단절 |

공통 뿌리는 두 가지다: (a) 상호작용 규범이 "결정"에만 존재하고 질문·안내에는 없던 비대칭, (b) 문면이 여는 입력 창구와 실제 대기 상태의 불일치. 백로그 1건(§8 문면이 v1.13.0 분할 후 스테일)을 함께 처리한다.

## 2. 목표와 비목표

### 목표

- GOAL-001: 사용자 대면 상호작용(결정·질문·안내)의 문법을 단일 규칙집으로 통일 — decision-flow.md를 결정+질문 공통 대화 규칙집(상호작용 SSOT)으로 확장
- GOAL-002: 렌더링 환경(한글 이중폭·터미널)에서 구조적으로 깨지지 않는 고정 형식 확립
- GOAL-003: 초심자 기준 언어를 기본값으로 — 판단에 필요한 정보(결과 의미·용어 풀이·추천 사유)가 항상 문면에 있음
- GOAL-004: 성능 낮은 모델도 지킬 수 있는 규칙 형태 — 판단형 규칙을 기계적 규칙·화이트리스트·고정 문면으로 하향

### 비목표

- Superpowers 위임 단계(DEVELOP/REVIEW)의 대화 UX (실불편 관측 후 별도 확장)
- 결정 확정 절차(번호 매핑·기록)의 근본 로직 재설계 — 형식과 시점만 조정
- 질문 순차화를 위한 새 상태 기계/큐 시스템 — 기존 대기열·phase 불변 구조 재사용
- decision-flow.md 파일 분리 실행 (분리 부적합 판정 — §6)
- HANDOFF 스키마 전면 재설계
- 헤더 진행 표시(📋 확정 N/M)의 재설계 — 유지 + 별개 카운트 규정만 추가

## 3. 확정된 요구사항

- REQ-001 (공통 스켈레톤): 결정(📌)·질문(❓) 공통 출력 형식을 고정 — 괘선(┌│└)·마크다운 표 금지, 완성 예시 수록, 💡 추천 줄 분리 (§8에 확정 문면) — HIGH
- REQ-002 (응답 원칙 개정): "선택지까지 노출된 항목에만 답할 수 있다" — 하나씩 모드에서 쉼표/줄바꿈 다중 값 응답은 첫 값만 현재 항목 반영+안내, 선답의 모드 전환 자동 해석 금지 (명시적 "다 보여줘"/"알아서"는 유지). 기존 §4 일괄 수용·순서 무시 조항 재작성 — HIGH
- REQ-003 (자가 응답 금지): 선택지를 출력했으면 그 선택은 사용자의 것. 선택지 생략은 고정 표현 화이트리스트("그냥 기본값으로"/"알아서 해줘"/"당연히 A로" 류)에 해당할 때만. 오판 시 대기열 경유 없이 즉시 선택지 소급 제시. context-handling 축약 규칙에 "새 작업 의도 명확 시 목록 미출력(정보 1줄 대체), 목록 출력 시 반드시 대기" 케이스 추가 — HIGH
- REQ-004 (초대 시점): 사전 브리핑·입력 초대 문면은 실제 응답 대기 시점에만 — 브리핑은 첫 결정 박스 직전, 토론 라운드 선행 경로에서는 라운드 종료 후로 이동 — HIGH
- REQ-005 (질문 트랙): 탐색적 질문 기본 하나씩 + ❓ 질문 N/M 진행 표시(결정 카운터와 별개 집계), M은 페르소나 전원 소집 후 확정, 답은 잠정(국면 전환 시 확정), 모름·넘어가기→조사 대기열 적재, 기존 "한 번에 최대 2개" 규칙 대체. brainstorming Step A는 정제(수집·병합·정렬·힌트) 유지, 노출만 순차로 — HIGH
- REQ-006 (초심자 언어 기본값): 사용자 대면 질문·결정·결과 공유 문면의 기본 = 짧은 배경 + 선택지별 "고르면 어떻게 되는지" 1줄 + 용어 등장 시 풀이 + 💡 추천·사유 별도 줄. 비유는 도움 될 때(의무 아님). 트리거 불요 — 기본값. 본 브레인스토밍 세션의 실물 예시를 규칙 문서에 수록 — HIGH
- REQ-007 (ID 라벨 병기): 문서 내부 ID의 대화 단독 지칭 금지 — 항상 짧은 라벨 병기 (예: `REQ-001(공통 스켈레톤)`) — MEDIUM
- REQ-008 (공유 출력 형식): 요약·결과 공유는 구분 헤더(이모지 부착, 필요시 `──` 구분선) + `-` 항목 나열 + 구분 간 빈 줄. `/` 인라인 구분 금지. 페르소나 발언은 이름 줄 + 개행 후 내용 — MEDIUM
- REQ-009 (로컬 화면 통일): 스킬 고유 선택지 화면 약 10개(시드 확인·Simplifier·국면0 카테고리·미답변 조사 Pass2·plan-stage 재협의·잔존 HANDOFF 삭제·persona 저장 제안 등)를 신규 스켈레톤으로 개정. §8 "문서의 템플릿을 그대로 출력한다" 원칙은 유지 — HIGH
- REQ-010 (HANDOFF 질문 진행): 결정 마커 확장 없이 별도 줄 신설 — `❓ 질문 진행: [N]/[M] · 국면: [국면명]` + 미답변 질문 목록 — MEDIUM
- REQ-011 (폴백 동기화): workflow-orchestrator SKILL.md 결정 요청 폴백 형식 + 부트스트랩(bootstrap.md) 폴백 문면을 신규 스켈레톤으로 동기화 (이중 관리 지점 — 누락 시 SSOT와 폴백이 다른 모양) — HIGH
- REQ-012 (예외·경계 명문화): `──` 대시 구분선·디렉토리 트리·산출물 내부 표는 금지 대상 아님을 명시. 적용 범위(BRAINSTORM/PLAN/COMPLETION + 세션 시작/컨텍스트 관리, DEVELOP/REVIEW 제외)와 네거티브 스코프(merge-to-domain·design-summary 기존 출력 별개) 명시. G5 관계 1문장("G5는 확인의 묶음 단위를, 본 규칙은 묶음 내부의 제시 방식을 다룬다"). §8 "본문에" 문면을 "본문 또는 본문이 지시하는 references/ 템플릿에"로 갱신(백로그 B1) — MEDIUM
- REQ-013 (검증): G4 패턴 재사용 — 게이트 대조 인벤토리(화면 이름·선택지 수·0번 유무 전후 비교) + §4 개정 문면·스켈레톤의 사전 확정(plan에 완성형 수록, 구현은 치환만) — HIGH

## 4. 설계 개요

**decision-flow.md 내부 재편 (한 파일, 3부 구성):**

- 1부 결정 규칙 — 기존 §1~§9 개정 (박스→스켈레톤, §4 응답 원칙 재작성, F 브리핑 시점 이동)
- 2부 질문 규칙 — 신설 (하나씩·❓ N/M·M 확정 시점·잠정 답·조사 대기열 라우팅·결정과의 층위 차이)
- 3부 공통 형식·언어 — 신설 (스켈레톤 확정 문면+완성 예시, 초심자 언어 기본값+실물 예시, ID 라벨 병기, 공유 출력 형식, 예외 목록, 적용 범위, G5 관계)

**수정 파일 지도:** decision-flow.md(중심) / workflow-orchestrator SKILL.md(폴백·Ambiguous 질문) / hooks 부트스트랩(폴백 1줄) / brainstorming SKILL.md+references/templates.md(통합 정제 노출부·로컬 화면) / plan-stage(재협의·미결 확인) / context-handling(축약 규칙·HANDOFF·삭제 확인) / persona-resolution(저장 제안). 상세 충돌 지점 30곳은 phase2_discovery.md 2절이 원장.

## 7. 제약조건과 가정

- 다중 모델 하드 제약: 판단형 규칙 금지 — 기계적 트리거·화이트리스트·고정 문면만
- 부트스트랩 주입 예산(~2.5KB) 내에서 폴백 문면 교체
- 산출물(설계 문서·phase 파일) 내부의 마크다운 표는 대화 UX가 아니므로 계속 사용
- 가정: 신규 규범을 기존 decision-flow.md 절 구조에 편입 가능 (조사로 §별 개정 지점 확인 완료)
- 가정: "초심자 언어 기본값"의 절반은 스켈레톤 구조가 강제, 나머지는 실물 예시 모방 — 준수도는 도그푸딩 관측 대상

## 8. 기술 가이드라인

1. **확정 문면 사전 고정**: §4 개정 문면·스켈레톤·폴백 1줄은 plan.md에 완성형으로 수록 — 구현 태스크는 verbatim 치환만 (재작성 드리프트 방지)
2. **공통 스켈레톤 확정 문면** (결정 예 — 질문은 📌→❓, "결정 요청"→"질문 N/M"만 상이):

   ```
   📌 결정 요청 : [제목]

   **배경:** [왜 이걸 묻는지 1~3문장, 용어 풀이 포함]

   1. [선택지] — 고르면: [어떻게 되는지 1줄]
   2. [선택지] — 고르면: [어떻게 되는지 1줄]

   💡 추천: [번호]
   [사유 — 다음 줄]
   ```

3. 검증: 게이트 대조 인벤토리(화면 이름·선택지 수·0번 유무 전후 동일) + 확정 문면 grep — G4 검증 체계 재사용, 신규 인프라 금지
4. brainstorming 접합: 통합 정제의 수집·병합·힌트 로직 무변경 — 노출부만 "하나씩 + ❓ N/M + 모름 라우팅"으로 교체
5. 적용 범위 밖 무변경: REVIEW AC 박스, merge-to-domain·design-summary 출력, 디렉토리 트리, 산출물 내부 표
6. 릴리스: MINOR (docs/design 구조 무변경, 스킬 동작 하위 호환 — 기존 프로젝트 문서 그대로 동작)
