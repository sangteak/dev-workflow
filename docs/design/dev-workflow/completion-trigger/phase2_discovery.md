## 핵심 원칙 채택 (국면 2의 최대 발견)

사용자 문제 제기("여러 상황을 시스템이 알아서 판단하려는 게 이상하다")로부터 채택:

> **"완료는 시스템이 판단하는 상태가 아니라, 사용자가 선언하는 사건이다.**
> **시스템의 역할은 ① 열려 있음의 가시화 ② 선언의 확실한 라우팅 ③ 선언 수단 제공 — 세 가지로 한정한다. 판단은 어디에도 없다."**

- 작업의 진짜 마무리 경로는 다양함(리뷰 호출·셀프 리뷰·실 테스트·PR 피드백 대응) — 시스템이 완료 시점을 추론하는 설계는 전부 기각
- 확정된 4축이 모두 이 원칙과 정합함을 재검증 완료

## 영역별 결정 사항

### 영역 1: CI/PR 앵커 (OQ6) — 결정: 반영하지 않음

- 2단 Completion 분리안(작업자 마무리 / feature 닫힘) 기각 — 스킵당하는 마무리를 쪼개면 스킵 지점이 2배, "1단 완료·2단 미완" 좀비 상태 신설, 2단은 소유자 없는 시점이라 더 잘 스킵됨
- "PR 거절 불일치"는 관측 0건의 이론적 문제 — 최악의 경우에도 "닫힌 feature의 문제는 별도 feature" 규범이 기존 출구
- status=complete는 문서 시스템의 상태, PR 머지는 코드 시스템의 이벤트 — 억지 동기화는 문제 정의 오류
- PR 머지 감지·소비 시점 검증·`pr:` 참조 기록 등 일체의 기계 반영을 하지 않음 (핵심 원칙의 일관 적용 — 사용자가 "PR까지 끝났다"고 판단되면 선언하면 됨)

### 영역 2: 회색 지대 경계 — 결정: 파일 술어

- 🔔 마무리 대기 줄의 조건: **status ≠ complete ∧ plan.md에 잔여 태스크 없음 (LLM이 파일을 읽어 판단)**
- 원안(SDD 완료 보고 시점 = 사건 기반) 기각 — 대화 기억을 상태 저장소로 쓰는 설계는 컴팩션·세션 교체에 깨짐 (이번 문제의 근본 원인과 동일 결함)
- 시드 제약("plan.md 형식은 외부 산출물, 기계 파싱 의존 불가")과의 정합: 판단 주체를 LLM의 파일 읽기로 한정 — bash가 체크박스를 세지 않는다
- 채널 등급 분리: 🔔 규범 줄(LLM) = 정밀 술어 / statusline(bash) = 잔존 기반 "열림 N건" (성기지만 정직한 표시)
- 끝점: Completion Step 1이 status를 complete로 전이하는 순간 술어가 거짓 → 자동 소등
- 노출 지점: 오케스트레이터 통제 하의 일단락 출력(이슈 카드 해소 고지·Evaluator 결과·리뷰 완료 보고) 말미
- 문구(안): `🔔 마무리 대기: [feature] — /dev-workflow:finish 또는 마무리 발화` (안내 발화는 훅 테스트 케이스에 있는 것만 사용)

### 영역 3: 어휘 SSOT — 합의 통과

- SSOT = orchestrator SKILL.md의 기존 마무리 트리거 목록. 훅 스크립트는 미러 + 상호 크로스레퍼런스 주석
- 미러 드리프트 방지: **어휘 diff 검사를 회귀 테스트 0번 케이스로** (주석 희망 → 기계 보증)
- 오탐 시나리오 실재 인정 (개발 중 "이 함수 정리해줘" → 잔존 조건 참이므로 발동): 주입문을 **판정 위임형**으로 작성 — "사용자 발화가 세션 전체의 마무리 의도라면 Completion Protocol로 라우팅하라. 특정 코드·파일에 대한 정리 요청이면 해당 없음"

### 영역 4: statusline 절차 — 합의 통과

- 감지 위치 3곳 전부 검사: `~/.claude/settings.json` · 프로젝트 `.claude/settings.json` · `settings.local.json`
- `/dev-workflow:setup` 실행 시에만 제안, 자동 수정 없음 (신규 설치 제안 금지 — phase1 결정 유지)
- 병합 방식 기본 후보: "제안문 출력 + 사용자 직접 붙여넣기" (메모리의 "setup 파일 수술 위험" 회피 출구) — 상세는 PLAN
- 숨은 의존 명시: statusline이 성립하려면 영역 2가 파일 기반이어야 함 (스크립트는 대화 기억 접근 불가) — 파일 술어 채택으로 충족됨

### 영역 5: command 상세 — 결정: references 분리형

- 이름: `finish` (`/dev-workflow:finish`) — Superpowers `finishing-a-development-branch`와 네임스페이스로 구분, 커맨드 본문에 "별개" 1줄 명시
- save/resume의 ARGUMENTS 하드코딩 패턴 기각 — 그 패턴은 받는 스킬(context-handling)에 서브커맨드 분기 구조가 있어서 작동하는 것. orchestrator에는 분기 구조가 없고, 400줄 스킬 재로딩은 세션 시작 프로토콜 재실행 부작용 위험
- 채택: **Completion Protocol 절을 `skills/workflow-orchestrator/references/completion-protocol.md`로 추출**, command는 그 파일만 Read — 직행이 파일 구조로 보장 (v1.13.0 references 분할 관례 일치)
- 커맨드 호출 자체가 사용자의 완료 선언 — 트리거 감지 단계 생략
- 부대: orchestrator 본문의 해당 절을 참조로 치환 + 크로스레퍼런스 스윕 (규범 스윕 교훈 적용)

### 영역 6: 검증 방법 — 합의 통과

- ① 훅 회귀 테스트: 시나리오 6종(발동/침묵 경계) + 어휘 diff(0번) + 가짜 docs/design 픽스처 명시
- ② 성공 기록: "이번 마무리의 진입 경로 1줄 기록"을 Completion Protocol의 Step으로 편입 (프로토콜이 돌면 반드시 기록됨 — 자기 측정의 역설 회피)
- ③ 실패 기록: "다음 세션에서 사용자가 스킵을 발견해 지적했을 때"로 정직하게 한정 — 라우팅 실패는 본질적으로 자기 기록 불가(기록 시점이 증발)
