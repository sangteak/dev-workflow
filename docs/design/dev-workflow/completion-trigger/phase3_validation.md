## 생태계 비교 분석 (🔍 Ecosystem Analyst)

### 경쟁 플러그인의 마무리 스킵 대응 5류형

| 유형 | 대표 | 요지 | 확신도 |
|---|---|---|---|
| 파이프라인 내장 | feature-dev | 마무리가 단일 명령 내부 순차 단계 — 스킵을 구조로 소거, 단 이탈 시 회수 없음 | 중간 |
| 규범 + 선택지 제시 | Superpowers | verification-before-completion(Iron Law 수사) + finishing 4선택지. 훅은 SessionStart뿐 — **라우팅은 확률적, 우리가 관찰한 스킵이 정확히 이 구조의 약점** | 높음(실물 확인) |
| 상태 파일 영속화 | Deep Trilogy | "완료 유실"을 "언제든 재개 가능"으로 치환 | 낮음 |
| 훅 차단 | orchestration 계열 | blocking 강제 — 본 프로젝트 독트린과 충돌 | 낮음 |
| 인센티브 재설계 | Compound Engineering | 마무리를 교훈 자산화 가치 단계로 재정의 | 낮음~중간 |

- **"미완료 상시 가시화(경고등)"는 참조 범위 내 유례 없음** — 차별점
- 차별점: 완료=선언 사건 철학(주류는 시스템 감지 상태) · 조건부 훅의 외과적 2단 구조 · command의 references 직접 Read(라우팅 확률성에 대한 구조적 답)
- 함정 지적: ① 어휘 사전은 반복 확인된 부채(미탐=원문제 재발, 중복 방어 필수) ② 경고등의 규범 구현은 원문제와 같은 실패 계급 ③ bootstrap 강화는 "더 크게 외치기" 안티패턴 인접 — 수사 격화 금지
- 트렌드 정합: 훅 주입(advisory) 사용은 사용자 주권 면에서 우월, 파일 술어는 spec-as-state 흐름과 일치(단 문서 포맷 규율이 신뢰성 상한), 조건부 주입·references 분리는 컨텍스트 효율 트렌드 정중앙

## 재협의 항목과 최종 결정

- **Stop 훅 경고등 격상 (Analyst 제안)** vs TD 반론: ① 정밀 술어의 절반(plan 잔여 태스크)은 LLM 판단 한정 확정 — bash Stop 훅은 성긴 술어만 가능해 DEVELOP 중 소음 재등장 ② Stop 훅 등록·가시성은 저장소 근거 없는 일반 지식 영역
- **사용자 결정: 관측 후 확장 카드로 보류** — 규범 누락 실측 시 재검토 ("관측 기반 점진" 전략·"위반 실측 후 훅 도입" 전례와 일치)
