# Phase 1: 탐색 (Exploration) — completion-trigger

> 생성 후 수정 불가 (불변 스냅샷) · 생성일: 2026-07-14

## 주제

개발 종료 후 결함 처리가 이어지는 동안 Completion Protocol(마무리 시퀀스)이 스킵되는 문제.
"dev-workflow로 마무리해줘"라고 명시적으로 말하지 않으면 LLM이 그냥 지나침.

## 확정된 페르소나 구성

- 🛠️ Claude Code Expert — 플랫폼 제약, 스킬/훅/플러그인 메커니즘 (국면 1~3)
- 🎯 Workflow Designer — 워크플로우 UX, 단계 흐름 (국면 1~3)
- 🔍 Ecosystem Analyst — 경쟁 플러그인 비교 (국면 3 활성화)

## A-0 본질 질문 결과 (Ontologist)

| 관점 | 결론 |
|---|---|
| 본질 | 마무리 질문은 DEVELOP 종료 시 1회 발화되는 이벤트 — 흐름 이탈 시 재무장 계기 없음 |
| 근본 원인 | 마무리 시점 결정권은 사용자에게 있음. LLM은 코딩 완료 이후 시점을 알 수 없음 |
| 전제 조건 | (조사) "완료 기록 체계"(status·이슈 카드)는 기계 판정 가능, "마무리 시점 도래"는 기계 판정 없음. 비대칭: 미완료는 기계 감지됨(plan.md/HANDOFF 잔존), 완료 도달은 감지 안 됨 |
| 숨은 가정 | 게이트 개념("마무리 없이 feature 못 닫음")은 타당(사용자 동의). deny형 훅은 독트린 충돌로 기배제, 주입형·ask-승격형은 가용 |

## Step A 인터뷰 결과 (Socratic ×2 병합, 4문)

1. **알람 vs 경고등** → 경고등이 꺼져 있는 문제. 상태가 지속 노출됐다면 사용자가 인지했을 것 (해법 공간 = 상태의 지속 가시화)
2. **스킵 비용** → ① resume 시 미완료 오표시 ② 도메인 머지 제외 ③ 사후 완료 여부 재확인 수고·토큰 (컨텍스트 소실로 비싸지는 복구)
3. **리마인드 타이밍** → 작업자마다 다름. 공통 앵커: **최종 테스트로 요구사항 충족 확인되는 순간 = 작업 완료 시점**. CI 프로젝트는 **PR 종료 시점 = feature 마무리 순간**
4. **잊음의 주체** → **중대 재정의: 놓치는 주체는 사용자가 아니라 LLM.** "마무리하자" 발화가 있어도 Completion Protocol로 라우팅하지 않고 자체 판단으로 정리함 (라우팅 실패)

## 시드 (Step B 확정 — 사용자 승인)

```yaml
goal: "dev-workflow의 Completion Protocol이 개발 종료 후 스킵되지 않도록, 미완료 상태의 지속 가시화(경고등)와 마무리 발화 시 스킬 라우팅 신뢰성을 함께 확보한다."
constraints:
  - "deny형(무조건 차단) 훅 사용 불가 — 모든 개입은 오탐 시 사용자 우회 수단 동반"
  - "허용 개입: 컨텍스트 주입형 훅 + ask-승격형"
  - "마무리 시점의 최종 결정권은 사용자 — LLM의 자의 확정·자동 실행 금지"
  - "'완료 도달' 기계 판정 불가 — 기계 신호는 '미완료 지속'(plan.md/HANDOFF 잔존)과 '완료 기록 체계'(status·이슈 카드)뿐"
  - "문서 기반 플러그인 구조 유지"
  - "commands/ 추가 시 기존 포함 기준 준수"
non_goals:
  - "Completion Protocol 내부 단계 재설계 (문제는 내용이 아니라 진입 실패)"
  - "완전 자동 판정·무인 자동 실행"
  - "무조건 차단형 게이트"
  - "Superpowers plan.md 형식 규정화"
success_criteria:
  - "미완료 상태가 세션 중 지속 가시화 — 흐름 이탈에도 마무리 계기 소실 없음"
  - "마무리 발화 시 Completion Protocol로 라우팅 — LLM 임의 마무리 재현 안 됨"
  - "스킵 비용 3종(resume 오표시·머지 제외·사후 재확인 토큰) 제거"
  - "모든 개입에 오탐 시 우회 경로 존재"
assumptions:
  - "완료 앵커: 최종 테스트 통과 순간 / CI 프로젝트는 PR 종료 시점 — 일반화 미검증"
  - "키워드 매 세션 주입에도 라우팅 실패 — 실패 메커니즘 미규명 (후보: 묻힘/컴팩션 소실·약한 구속력·경쟁 지시)"
  - "PreToolUse/UserPromptSubmit 훅 등록 가능 — 문서 전제, 실동작 미검증"
  - "'미완료' 기계 감지(find 검사)는 신뢰 가능"
  - "command 제공은 후보 중 하나 (당시 미확정 → Step C에서 확정됨)"
open_questions:
  - "OQ1: 두 문제(경고등 부재·라우팅 실패)를 하나의 메커니즘으로? 분리? → Step C에서 해소: 분리(4축)"
  - "OQ2: 상태 가시화 채널? → Step C에서 해소: 규범 줄 + 기존 statusline 사용자 한정"
  - "OQ3: 라우팅 실패 근본 원인? → 후보 3종 식별, 조건부 주입이 3종 모두 우회 (개별 규명은 관측으로)"
  - "OQ4: ask-승격형 개입 지점? → 채택 설계에 불필요해짐 (기각)"

  - "OQ7: 게이트를 deny 없이 구현하는 방법? → 4축 방어가 소프트 게이트 역할 (완전 해소 아님, 국면 2 검토 대상)"
```

## 명확도 체크리스트 (전환 시점)

- ✅ Goal Clarity: 두 겹 문제(가시화+라우팅)로 명확
- ✅ Constraint Clarity: 기존 독트린에서 도출된 경계 구체적
- ⚠️ Success Criteria: "라우팅 실패 재현 안 됨"의 검증 방법 미정 — 관측 기반 검증으로 위임 (기존 프로젝트 전략)
- → 다음 국면 집중 영역: OQ6(CI/PR 앵커), OQ7(소프트 게이트 충분성), 검증 방법 구체화

## Step C 확장 토론 — 페르소나 피드백 결과

[라운드 1/3, 비판: 🛠️ Claude Code Expert (Contrarian)]

- **합의:** 매 턴 무조건 주입 기각(훅 지연·banner blindness·"정보 부재" 가설은 bootstrap 실패로 이미 반증). 라우팅 축 기본 방향 = "조건부·시점 정합 주입 + bootstrap 수리". command는 "기계적 보장"이 아닌 "저실패율 경로"로 병행.
- **핵심 논거 (🛠️):** 수신자 분리 — 훅 주입 텍스트는 모델 전용(사용자 화면 비표시), 사람용 상시 채널은 statusline. 라우팅 실패 원인 후보 3종: ① 주입문 묻힘/컴팩션 소실 ② 표 형식의 약한 구속력 ③ 경쟁 지시(superpowers 마무리 계열 스킬). 마무리 "발화 감지"는 훅 스크립트의 기계 판정 가능 (완료 "시점 판정" 불가와 별개 문제).
- **수용 조건 (🎯):** 기계 어휘 매칭은 안전망이지 유일 경로 아님(어휘 밖 발화 미탐 존재) · statusline 미설치 사용자를 위한 차선 경고등 필요.
- **미합의였다가 결정으로 해소:** statusline 도입 범위 → 사용자 결정으로 "기존 사용자 한정" 확정.

## 확정된 결정 (4건, 사용자 승인)

| # | 결정 | 선택 | 부대 요구사항 |
|---|---|---|---|
| 1 | 라우팅 보강 방식 | **조건부 정밀 주입** (UserPromptSubmit 훅: 미완료 잔존 ∧ 마무리 어휘 → 명령형 지시 주입) | 어휘 목록 SSOT+크로스레퍼런스 · 잔존 감지 로직 session-start와 공용화 · 견고한 JSON 파싱(jq/폴백) |
| 2 | 경고등 채널 | **규범 + statusline 병행 (기존 사용자 한정)** — 회색 지대 일단락마다 "🔔 마무리 대기" 줄, statusline은 이미 운영 중인 사용자에게만 opt-in 통합 제안 (신규 설치 제안·강요 금지 — 사용자 명시 제외) | 미설치 사용자는 조용히 스킵 |
| 3 | command 추가 | **추가** — Completion Protocol 직행 명령 (이름 PLAN에서 확정, 예: finish) | 경고등 문구가 명령 안내 · commands/ 포함 기준 부합 확인 |
| 4 | bootstrap 수리 | **명령형 강화** — COMPLETION 매핑을 명령문+금지문("자체 정리 금지, orchestrator invoke")으로 교체 | compact 재주입은 hooks.json matcher에 이미 존재 (수리 불요 확인됨) |

## 시뮬레이션 결과 (프로토타입 실증)

- **라우팅 훅 6시나리오:** 발동(잔존∧어휘, 한/영) · 침묵(무관 발화, 잔존 없음, _archive만 잔존) 모두 의도대로. 미탐 확인: 어휘 밖 발화("여기까지 하고 넘어가자") — 2·4축이 보완할 구멍으로 기록
- **statusline 3상태:** 미완료 N건 경고 표시 → 마무리 직후 ✓ 즉시 전환 (파일 기반 실시간)
- **한계:** 주입 후 LLM의 실제 라우팅 성공 여부는 오프라인 검증 불가 — 도입 후 실사용 관측으로 검증 (기존 "관측 기반 점진" 전략)
- 유지보수성 평가: 기존 bash 훅과 동일 기술 표면, 시나리오는 회귀 테스트로 보존 가능 — Markdown 규범보다 기계 검증성 우위
