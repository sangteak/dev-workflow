## 각 항목에 대한 Q&A 결정 사항

### 1. 요약의 출력 형태

**질문:** 화면 출력만 할 것인가, 파일로 생성할 것인가?

**결정:** 화면 출력 전용. 파일 생성 없음.

**근거:**
- 파일로 저장하면 원본 문서 변경 시 동기화(outdated) 문제 발생
- 저장 위치 결정도 복잡 (기능 디렉토리와 혼재 우려)
- 사용 빈도가 "가끔"이므로 매번 최신 상태로 재생성하는 것이 깔끔
- 파일 관리 부담 제로

### 2. 요약의 깊이

**질문:** 각 문서에서 어떤 정보를 추출하고, 어떤 형태로 제시할 것인가?

**결정:** 서사적 재구성 (샘플 B 방식)

**검토한 대안:**
- 템플릿 기반 추출: 각 문서에서 정해진 섹션을 기계적으로 뽑아 정형 포맷에 배치. 일관성 높지만 "기획서" 느낌 약함
- 서사적 재구성: Claude가 전체를 읽고 시스템 관점에서 자연어로 재서술. 읽는 사람이 시스템을 파악하기에 적합

**서사적 재구성 포함 요소:**
- 시스템 전체 개요 (이 시스템은 무엇인가)
- 기능별 서술 (배경, 동작 방식, 핵심 설계 결정을 자연어로)
- 기능 간 관계 (의존성 맵을 서사적으로)
- 현재 상태 (각 기능의 완료/미완료)
- 제약조건 및 전제

**컨텍스트 전략:** 서브에이전트 2단계
- 1단계: 서브에이전트가 각 문서를 병렬로 읽고 핵심 추출
- 2단계: 메인 에이전트가 압축된 요약들을 조합하여 기획서 형태로 재구성
- 문서 2~3개면 직접 로드, 4개 이상이면 서브에이전트 분할

### 3. 구현 위치

**질문:** 기존 design-doc-index 스킬을 확장할 것인가, 별도 스킬을 만들 것인가?

**결정:** 별도 스킬로 분리

**근거:**
- design-doc-index: "찾기" (개별 문서 탐색/로드)
- 새 스킬: "이해하기" (여러 문서를 읽고 서사적으로 재구성)
- 단일 책임 원칙 — 하나의 스킬이 두 가지 성격의 작업을 갖지 않음
- 독립적으로 개선/교체 가능
- 사용자 명시 호출이므로 오케스트레이터의 자동 감지 부담 증가 없음

### 4. 트리거 방식

**질문:** 사용자 명시 호출만인가, 세션 시작 시 자동 제안도 포함인가?

**결정:** 사용자 명시 호출만. 자동 제안 없음.

**근거:**
- 사용 빈도가 "가끔"이므로 매 세션 자동 제안은 불필요한 노이즈
- 세션 시작 프로토콜이 이미 6단계로 무거움
- 대신 design-doc-index의 색인 모드 결과에서 관련 문서 다수 감지 시 힌트 제공 가능
  (예: "관련 문서가 N개 발견되었습니다. 통합 요약이 필요하시면 [명령어]를 사용해주세요")

- [라운드 1/3] — **합의:** 화면 출력 전용. 파일 저장 시 동기화 문제와 저장 위치 문제가 발생하며, 매번 재생성이 항상 최신 상태를 보장

### 주제 2: 요약의 깊이
- [라운드 1/3] — **미합의:** 템플릿 기반 vs 서사적 재구성, 컨텍스트 전략(일괄 vs 서브에이전트)
- [라운드 2/3] — 서브에이전트 2단계 전략 설명 및 합의. 요약 스타일은 샘플 비교로 결정
- 샘플 A(템플릿) / 샘플 B(서사적) 실제 생성 → 사용자 판단: **샘플 B 선택**

### 주제 3: 구현 위치
- [라운드 1/3] — **합의:** 별도 스킬로 분리. "찾기"(design-doc-index) ↔ "이해하기"(새 스킬) 역할 구분

### 주제 4: 트리거 방식
- [라운드 1/3] — **합의:** 사용자 명시 호출만. design-doc-index에서 힌트 제공으로 보완
