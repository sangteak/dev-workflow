## 탐색된 요구사항

### 배경
- dev-workflow를 사용하는 프로젝트에서 하나의 큰 기능이 `peek-overview`, `peek-detail`, `peek-interaction`처럼 여러 하위 작업으로 분할되어 진행되는 경우가 있음
- 이는 git feature branch 전략과 유사한 패턴
- 문서가 증식하면서 전체 그림을 파악하기 어려워질 수 있다는 우려

### 논의된 방향들

1. **물리적 병합 (git merge 대응)** — 분산된 설계 문서를 하나로 통합
   - 결론: **도입하지 않음**
   - 사유: REQ 번호 재정렬, 의존성 맵 합산, 기술 결정 테이블 중복 제거 등 복잡도가 높고 Markdown 스킬 지시문만으로 품질 보장이 어려움

2. **논리적 그루핑 + 일관성 검증** — 그룹 개념만 도입하고 일관성 체크
   - 결론: 부분 채택 가능하나 단독으로는 불충분

3. **읽기 시점 취합 (최종 합의 방향)** — 사용자 요청 시 관련 문서 그룹의 통합 요약 생성
   - 물리적 파일 변경 없이 요약 뷰를 제공
   - 기존 `design-doc-index`의 검색/로드 오버헤드가 낮다는 결론을 활용

### 그룹 감지 방식 논의
- 네이밍 패턴 기반 (`peek-*`): 간단하나 사용자가 일관된 접두사를 안 쓸 수 있음
- 명시적 그룹 선언 (프론트매터 `group:`): 확실하나 사용자 부담 증가
- 사용자 직접 지정: 가장 유연하나 자동화 수준 낮음
- **합의:** 사용자 지정 + 네이밍 패턴 보조 (사용 빈도가 낮으므로 무거운 자동화 불필요)

### 명시적으로 제외한 항목
- 워크플로우에 강제 머지 단계 추가 — 실제 불편함이 없으므로 불필요
- 물리적 문서 병합/재구성 — 복잡도 대비 실익 없음
- dev-workflow 플러그인 자체의 문서 구조 변경 — 대상은 사용 프로젝트
