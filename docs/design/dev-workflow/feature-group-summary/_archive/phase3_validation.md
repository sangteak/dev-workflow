## 재협의 항목 및 최종 결정

### 구현 위치 재검토
- **쟁점:** 별도 스킬 vs design-doc-index의 세 번째 모드 (관리 비용 관점)
- **Ecosystem Analyst 우려:** "가끔" 쓰는 기능에 스킬 하나를 정당화할 수 있는가
- **사용자 결정:** 역할이 맞지 않으므로 별도 스킬로 분리. 관리 이슈 감수
- **최종:** 별도 스킬 (design-summary) 확정

### 스킬 네이밍
- **후보:** design-summary (추천) / feature-overview / feature-digest
- **선정 근거:** "요약", "정리", "summary" 발화에 자연스럽게 매칭. design- 접두사로 design-doc-index와 도메인 일관성
- **최종:** design-summary 확정

### 트리거 방식 확장
- **쟁점:** 자연어 감지 병행 vs 명령 전용
- **Workflow Designer 우려:** "요약해줘" 발화의 오탐 리스크 (브레인스토밍 중 토론 요약과 혼동)
- **Ecosystem Analyst 분석:** 유틸리티 성격의 스킬은 명시적 호출이 자연스러움. 워크플로우 단계가 아닌 독립 유틸리티
- **사용자 결정:** 스킬이 명령으로 이용 가능하므로 자연어 감지 불필요
- **최종:** 명령 호출 전용 (`/dev-workflow:design-summary`)
