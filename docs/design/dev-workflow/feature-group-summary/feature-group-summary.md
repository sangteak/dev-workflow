# feature-group-summary 설계 문서

> 한 줄 요약: 관련 설계 문서 그룹을 기획서 수준의 통합 요약으로 재구성하여, 시스템 전체를 한 번에 파악할 수 있게 한다.

## 1. 배경과 동기

- dev-workflow를 사용하는 프로젝트에서 하나의 큰 기능이 여러 하위 작업(`peek-overview`, `peek-detail`, `peek-interaction` 등)으로 분할되어 각각 별도 설계 문서로 진행되는 경우가 있다
- 문서가 증식하면서 "이 시스템 전체가 무엇인가"를 파악하려면 모든 문서를 개별로 열어야 하는 부담이 생긴다
- git의 feature branch → main merge 전략에서 착안했으나, 물리적 문서 병합은 복잡도 대비 실익이 없다고 판단
- 대신 **읽기 시점에 통합 요약을 생성**하는 가벼운 접근을 채택

## 2. 목표와 비목표

### 목표
- GOAL-001: 관련 설계 문서 그룹의 통합 요약을 기획서 수준으로 생성한다
- GOAL-002: 서사적 재구성으로 읽는 사람이 시스템을 한 번에 파악할 수 있게 한다
- GOAL-003: 물리적 파일 변경 없이 화면 출력만으로 최신 상태를 보장한다

### 비목표
- 물리적 문서 병합/재구성 — 복잡도 대비 실익 없음 (REQ 재정렬, 의존성 합산, 서사 재구성 등)
- 워크플로우에 강제 머지 단계 추가 — 실제 불편함이 아닌 예방적 우려였으므로 불필요
- 요약 결과의 파일 저장 — 원본 변경 시 동기화 문제, 저장 위치 문제 발생
- 세션 시작 시 자동 제안 — 사용 빈도가 "가끔"이므로 불필요한 노이즈
- 범용 자연어 감지 ("요약해줘") — 브레인스토밍 중 토론 요약과 오탐 위험

## 3. 확정된 요구사항

- REQ-001: 별도 스킬(`design-summary`)로 구현한다 — 우선순위: HIGH
- REQ-002: 명령 호출 전용(`/dev-workflow:design-summary`)으로 트리거한다. 자연어 감지 없음 — 우선순위: HIGH
- REQ-003: 출력은 화면 전용이다. 파일을 생성하지 않는다 — 우선순위: HIGH
- REQ-004: 사용자가 대상 범위를 지정한다 (카테고리명, 키워드, 기능명 목록 등) — 우선순위: HIGH
- REQ-005: 네이밍 패턴 보조로 관련 문서를 자동 제안한다 (예: `peek-*` → 3개 매칭) — 우선순위: MEDIUM
- REQ-006: 서사적 재구성 방식으로 기획서 수준의 통합 요약을 생성한다 — 우선순위: HIGH
- REQ-007: 출력 구조는 고정 가이드를 따른다 (시스템 개요 → 기능별 서술 → 기능 간 관계 → 제약조건) — 우선순위: HIGH
- REQ-008: 문서 4개 이상이면 서브에이전트 2단계 전략을 사용한다 (1단계: 병렬 추출, 2단계: 통합 재구성) — 우선순위: HIGH
- REQ-009: 문서 3개 이하면 직접 로드하여 재구성한다 — 우선순위: MEDIUM
- REQ-010: `design-doc-index`의 색인 모드에서 같은 접두사 문서가 3개 이상이면 `design-summary` 스킬 힌트를 제공한다 — 우선순위: LOW

## 4. 설계 개요

### 스킬 구조

```
skills/design-summary/
└── SKILL.md              ← 스킬 정의 (프론트매터 + 지시문)
```

### 실행 흐름

```
사용자: /dev-workflow:design-summary
  │
  ├─ 범위 지정 확인
  │   ├─ 인자 있음 (키워드/카테고리) → 매칭 탐색
  │   └─ 인자 없음 → 범위 질문
  │
  ├─ 대상 문서 탐색
  │   ├─ docs/design/[카테고리]/[키워드]*/ glob 매칭
  │   ├─ status: complete 필터링
  │   └─ 매칭 결과 제시 → 사용자 확인
  │
  ├─ 문서 수에 따른 분기
  │   ├─ 1~3개 → 직접 로드
  │   └─ 4개+ → 서브에이전트 2단계
  │       ├─ 1단계: 서브에이전트가 각 문서를 병렬 읽기 + 핵심 추출
  │       └─ 2단계: 메인이 추출 결과를 조합하여 서사적 재구성
  │
  └─ 출력 (화면 전용)
      ├─ 시스템/기능 그룹 개요
      ├─ 기능별 서술 (배경 → 동작 → 핵심 결정 → 상태)
      ├─ 기능 간 관계
      └─ 제약조건 및 전제
```

### 서브에이전트 추출 항목 (표준화)

서브에이전트에 전달하는 추출 지시:
1. 한 줄 요약
2. 배경과 동기 (핵심 2~3줄)
3. 목표 (GOAL 목록)
4. 확정된 요구사항 (REQ 번호 + 한 줄 설명)
5. 설계 개요 (핵심 구조/흐름)
6. 의존성 (의존 대상 + 영향받는 컴포넌트)
7. 제약조건과 가정
8. 현재 상태 (status, 구현 결과)

## 6. 기술 결정 및 대안 검토

| 결정 사항 | 선택 | 근거 | 검토한 대안 | 기각 사유 |
|-----------|------|------|-------------|-----------|
| 통합 방식 | 읽기 시점 서사적 재구성 | 물리적 변경 없이 항상 최신, 기획서 수준 가독성 | 물리적 문서 병합 | REQ 재정렬, 의존성 합산 등 복잡도 높고 품질 보장 어려움 |
| 출력 형태 | 화면 전용 | 파일 관리 부담 제로, 매번 최신 보장 | 파일 생성 | 원본 변경 시 outdated, 저장 위치 문제 |
| 요약 스타일 | 서사적 재구성 (샘플 B) | 읽는 사람이 시스템을 자연스럽게 이해 | 템플릿 기반 추출 (샘플 A) | 정보 나열에 가까워 "기획서" 느낌 부족 |
| 구현 위치 | 별도 스킬 (design-summary) | 단일 책임 원칙, 독립 개선 가능 | design-doc-index 3번째 모드 | 역할 불일치 (찾기 vs 이해하기) |
| 트리거 방식 | 명령 전용 (/dev-workflow:design-summary) | 오탐 제로, 유틸리티 성격에 적합 | 자연어 감지 병행 | 브레인스토밍 중 "요약해줘"와 오탐, 워크플로우 단계와 충돌 가능 |
| 컨텍스트 전략 | 서브에이전트 2단계 (4개+ 문서) | 메인 컨텍스트 압박 방지, 병렬 처리 효율 | 일괄 로드 | 문서 5개 이상이면 1,000~2,000줄 로드로 컨텍스트 압박 |
| 그룹 감지 | 사용자 지정 + 네이밍 패턴 보조 | 사용 빈도 대비 적정 복잡도 | 프론트매터 group 필드 | 사용자 부담 증가, 사용 빈도 대비 과잉 |

## 7. 제약조건과 가정

- 설계 문서는 10섹션 표준 포맷(프론트매터 포함)을 따른다고 가정
- `status: complete` 문서만 대상 (미완성 설계 포함 시 정보 신뢰도 저하)
- 서사적 재구성 결과는 매번 동일하지 않을 수 있음 — 출력 구조 가이드로 정보 누락만 방지
- 서브에이전트 사용 시 Claude Code의 Agent 도구 사용 가능을 전제
- 프로젝트당 하나의 그룹에 포함되는 문서는 수십 개 이내 가정
