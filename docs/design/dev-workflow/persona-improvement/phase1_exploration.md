## 1. 탐색된 요구사항 전체 목록 (확정/미확정 구분 없이 언급된 모든 것)

| # | 요구사항 | 상태 |
|---|---|---|
| R1 | 공용 문체 블록 1개 신설 — 한국어 · 초보자도 읽는 일상어 · 전문 용어 즉석 풀이 · 줄임말 금지 | 확정 |
| R2 | 문체 블록 원본은 brainstorming 스킬 본문 공통 토론 규칙에, 서브에이전트 프롬프트에는 복제 삽입 (본체 발언 + 서브에이전트 발언 전부 커버) | 확정 |
| R3 | 문체 블록을 페르소나 발언이 만들어지는 모든 경로에 적용 — plan-stage 페르소나 토론 경로 포함 | 확정 |
| R4 | 페르소나 발언 실질성 규칙(맞장구 금지) — 발언은 도메인 근거 필수, 동의도 근거 병기, 보탤 것 없으면 발언 생략 | 확정 |
| R5 | 블라인드 첫 라운드 — 토론 첫 라운드는 전원 페르소나를 서로 안 보이게 병렬 독립 실행(서브에이전트), 2라운드부터 현행 대화형 | 확정 |
| R6 | 독립 실행 페르소나에게 본체 요약 대신 토론 원문 발췌 전달 (기울어진 요약 통로 차단) | 확정 |
| R7 | 비판 비율 상충 해소 — 도메인 문서(ux-consistency.md:13)의 "비판 60%:건설 40%" 수치 제거, "강제 비율 없음 + 방향성 안내"로 통일 | 확정 |
| R8 | 토론 리드 질문 규칙(ux-consistency.md:15) 문서에서 제거 — 스킬에 구현하지 않음 | 확정 |
| R9 | 기본 페르소나 표 단일화 — 모델용 원본은 persona-resolution 한 곳, brainstorming은 참조, README는 표 대신 안내 한 줄 | 확정 |
| R10 | HANDOFF 템플릿 "확정된 페르소나" 섹션에 PLAN 페르소나 자리 추가 (현재 브레인스토밍 목록만 예시) | 확정 |
| R11 | 이슈 카드 — "수정 방향대로 진행할까요?" 확인 지점에 페르소나별 관점 1줄씩 병기 (토론·별도 실행 없음, 할 말 없는 페르소나는 줄 생략) | 확정 |
| R12 | 국면별 비평 강도 — 현행 "방향성 안내" 유지 (추가 구체화 안 함) | 확정 (무변경 결정) |

언급되었으나 채택되지 않은 대안: 만장일치 경보(반대 0 라운드 발생 시 다음 라운드만 전원 독립 승격 — 블라인드 첫 라운드에 밀려 미채택), 전원·전라운드 독립화(비용·경험 훼손으로 기각), 이모지 선택 가이드 문서화(감사 P3 후보 — 이번 논의에서 다루지 않음).

## 2. 사용자가 명시적으로 제외한 항목과 이유

- 형식(템플릿 준수)만 기계 검사하는 훅/MCP — "MCP가 페르소나를 관리하고 발전시키는 용도라면 모를까 형식만 관리한다면 의미가 없어"
- MCP 기반 페르소나 관리·발전 기능 — 흥미로운 미래 방향으로 기록만, 이번 범위 아님
- DEVELOP/REVIEW 본 흐름에 페르소나 추가 — 의도된 경계로 유지 (예외: 이슈 카드 확인 지점 R11)
- 사고 제약 원문(영어 verbatim)의 수정·번역·버전 교체 — 고정 유지 (조사로 확정: 드리프트 0, REQ-005 정합)
- 비유 기반 설명 — 사용자가 원하는 것은 비유가 아니라 일상어 전환

### 라운드 요약

- **라운드 1** (비판: 🛠️ Claude Code Expert [Contrarian]): ① 문체 블록을 서브에이전트 프롬프트에만 넣으면 발언 대부분(본체 연기분)이 적용 밖 — 주입 지점 역전 필요 ② "60:40 유지"는 모델이 측정 불가한 지시라 문서 수치 제거만이 성립 ③ 페르소나 표 3중 중복은 독자(사람/모델) 분리로 재정의 ④ 이슈 카드 페르소나는 시드 내부 충돌 — 라벨만은 장식, 토론은 경량 철학 훼손, 최소 접점(관점 1줄)만 권장 ⑤ 리드 질문 규칙 미구현은 자연도태. → 전부 합의로 수렴.
- **라운드 2** (비판: 🎯 Workflow Designer [Contrarian]): 사용자 제안(전원 독립화) 재론. ① 독립 실행 ≠ 독립 사고 — 요약 작성자(본체)가 정보 입구를 쥐면 편향 잔존 ② 편향 최대 지점은 첫 발언 앵커링 ③ 전원·전라운드 독립화는 주제당 최대 9회 호출로 "읽으며 끼어들기" 경험 훼손 ④ 대안: 블라인드 첫 라운드(병렬, 대기 1회분)·만장일치 경보·원문 발췌. 🛠️ CCE 실측 보강: 호출당 30초~2분·3~4만 토큰, 블라인드 라운드는 Step A 기존 패턴과 일관. → 블라인드 첫 라운드 + 원문 발췌로 확정.

## 5. 시드 (Step B 사용자 확인 완료 버전)

constraints:
  - "고치는 범위는 지시문과 문서 정비까지만 한다 — 프로그램 코드로 형식을 자동 검사하는 장치(훅/MCP)는 만들지 않는다."
  - "페르소나에 섞어 쓰는 사고 제약 원문(영어)은 그대로 둔다 — 번역하거나 고치지 않는다 (현재 설치본 0.39.0과 차이 없음 확인됨)."
  - "말투 개선은 공용 문체 블록 1개(한국어 사용, 일상어로 쓰기, 전문 용어는 나오는 자리에서 바로 풀이, 줄임말 금지)를 모든 페르소나 서브에이전트 프롬프트를 조립할 때 끼워 넣는 방식으로 한다."
  - "이 방식은 기존 설계 원칙(REQ-005: 완충 지시는 원문 블록이 아니라 내보내는 프롬프트 쪽에 감싸기, 출력 형식은 전부 자체 템플릿이 소유)과 어긋나면 안 된다."
  - "쉽게 쓴다는 것은 비유를 들어 설명하라는 뜻이 아니다 — 어려운 한자어와 줄임말을 일상어로 바꾸라는 뜻이다."

  - "DEVELOP/REVIEW 단계 본 흐름에 페르소나를 넣지 않는 현재 경계는 의도된 것이므로 유지한다."

  - "MCP가 페르소나를 관리하고 발전시키는 기능 — 흥미로운 미래 방향으로 기록만 하고 이번에는 만들지 않는다."
  - "비판 담당 외의 페르소나까지 독립 서브에이전트로 돌리는 구조 확장 — 개선 후보로 기록만 한다."
  - "DEVELOP/REVIEW 단계 본 흐름에 페르소나 토론을 추가하는 것 (이슈 카드 흐름은 예외로, 신규 요구에 포함)."
  - "사고 제약 원문(영어)의 수정, 번역, 버전 교체."

> 참고: 시드의 open_questions는 Step B 확정 시점 기준 6건이었으며, Step C 토론에서 전부 해소되었다 (해소 결과는 §4 결정 원장이 원본). R4·R5·R6은 Step C에서 추가 확정된 요구사항으로 Step B 시드에는 없다.
