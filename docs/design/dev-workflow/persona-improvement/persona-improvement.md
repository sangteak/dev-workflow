# persona-improvement 설계 문서

> 한 줄 요약: 페르소나 발언을 초보자도 읽는 일상어로 바꾸고, 토론의 실질성·독립성을 지시문 수준에서 강화하며, 페르소나 관련 문서-스킬 불일치를 정비한다.

## 1. 배경과 동기

- 사용자 실사용 피드백: 역할별 의견 분리는 문제 파악에 실질적으로 유효하나, 세 가지 불만이 확인됨 — ① 발언의 언어 난이도(전문 용어 과다·평소 안 쓰는 한자어·줄임말) ② 개행 없는 벽 텍스트 가독성 ③ "한 AI의 역할 놀이" 의문(본체가 연기하는 비판 담당 외 페르소나의 동조 편향)
- 전수 감사에서 문서-스킬 불일치 실재 확인: ux-consistency.md의 "비판 60:40" 수치(스킬은 "강제 비율 없음"), 미구현 "토론 리드 질문" 규칙, 기본 페르소나 표 중복(persona-resolution + README 2곳), HANDOFF 템플릿의 PLAN 페르소나 슬롯 부재
- 근본 원인 확정(조사): 발언이 어려운 이유는 사고 제약 원문(영어)이 아니라, 서브에이전트 프롬프트에 출력 언어·문체 지시 층이 통째로 없기 때문. 사고 제약 원문은 설치본 Ouroboros 0.39.0과 드리프트 0 확인 — verbatim 고정의 원 목적(REQ-005 의미 드리프트 기계 검증)은 유효하며, 출력 문체는 처음부터 자체 소유 영역

## 2. 목표와 비목표

### 목표
- GOAL-001: 페르소나 발언이 만들어지는 전 경로에 언어 규범(일상어·문단 개행)을 적용한다
- GOAL-002: 토론의 실질성(맞장구 차단)과 독립성(앵커링 차단)을 지시문 수준에서 강화한다
- GOAL-003: 페르소나 관련 문서-스킬 불일치를 0으로 만든다

### 비목표
- 형식(템플릿 준수)만 기계 검사하는 훅/MCP 도입 — "형식만 관리한다면 의미가 없다"(사용자)
- MCP 기반 페르소나 관리·발전 기능 — 흥미로운 미래 방향으로 기록만
- 전원·전라운드 페르소나 독립 서브에이전트화 — 비용·"읽으며 끼어들기" 경험 훼손으로 기각, 블라인드 첫 라운드로 대체
- DEVELOP/REVIEW 본 흐름에 페르소나 토론 추가 — 의도된 경계 유지 (이슈 카드 확인 지점 관점 1줄만 예외)
- 사고 제약 원문(영어 verbatim)의 수정·번역·버전 교체
- 비유 기반 설명 강제 — 목적은 일상어 전환이지 비유가 아님
- 만장일치 경보(대화형 라운드 동조 감지 → 전원 독립 승격) — 관측 후 재고 후보로만 기록

## 4. 설계 개요

- **언어 규범 아키텍처:** 원본 1곳(decision-flow 3부 「언어 규범」 펜스) + 실행 시점 복제(각 스킬의 서브에이전트 프롬프트 조립 시 펜스를 그대로 삽입). 정적 복제(사본이 따로 낡음)와 참조만(서브에이전트가 안 읽음)의 두 실패를 모두 회피. 본체 발언은 decision-flow가 상시 규범이므로 원본이 직접 커버
- **블라인드 라운드 구조:** Step A(인터뷰 전원 서브에이전트)의 기존 패턴을 판정·첫 루프로 확장하는 것으로 서술 — 새 발명이 아님. 페르소나 수만큼 병렬 호출(대기 1회분), 프롬프트 = 사고 제약(해당 시) + 도메인 정의 + 발췌(REQ-006 규격) + 언어 펜스 + 정형 형식(결론 1줄 + 근거 최대 3개)
- **발언 실질성:** 금지형이 아니라 의무형 슬롯(새 기여 요소) + 기본값 반전. 대화형 라운드는 자유 문장 유지(형식 라벨 강제 없음)
- **이슈 카드 접점:** orchestrator Issue Lifecycle 경량 사이클 1단계 확인 문구에 통합. "DEVELOP 중 계획 성격 활동에 페르소나"라는 크리티컬 경로(재계획 → plan-stage) 선례의 확장
- **도메인 교정:** 본 문서 §6의 supersede 지정 → 관리자의 merge-to-domain 시점에 원자 반영 (직접 수정은 이중 반영 위험으로 기각)
- **(구현 확정 구조)** 원본 2곳: decision-flow 3부 `### 언어 규범 (원본)` 펜스 · §7 `### 블라인드 라운드 공통 절차 (원본)` 하위절 (절 번호 무변경 — 외부 참조 보호). 복제·포인터 소비 지점 실측: brainstorming templates 프롬프트 7 + 신설 블라인드 프롬프트 1 + plan-stage 프롬프트 2 + 판정 구성 문구 + orchestrator 축약 2곳(이슈 카드·폴백) + bootstrap 불변식 1줄. 구현 순서는 원본 선행 → 소비자 반영 → 스킬 밖+스윕 3단계로 수행

## 7. 제약조건과 가정

- 지시문·문서 정비 한정 — 프로그램 코드(훅/MCP) 변경 없음, 전 변경이 git 되돌리기 가능(비가역 지점 없음)
- 사고 제약 원문(agent-roles.md 3개 파일) 바이트 불변 — 완충·문체 지시는 프롬프트 조립 층에서만
- 지시문 준수율은 보장이 아니라 관측 대상 (REQ-013 관측 고리)
- 블라인드 비용 실측 전제: 호출당 30초~2분·3~4만 토큰, 브레인스토밍 1건당 9~15회 호출 증가, 병렬이라 체감 대기는 라운드당 1회분
- 발췌 조립 주체가 본체 자신이라는 한계는 잔존 — 결정적 규칙(원문 인용)으로 재량 축소, 사후 대화 기록으로 확인 가능

## 8. 기술 가이드라인

1. decision-flow 3부 분리는 항목 단위 재배치표를 먼저 만들고 실행 — "초심자 언어 기본값" 5항목 중 언어 규범은 용어 풀이 1개뿐, 문단 통째 이동 금지
2. 언어 규범 펜스 머리에 적용 범위 선언 1줄 — "본 파일의 결정 흐름 적용 범위와 별개로 모든 사용자 대면 문면에 적용"
3. §7 예외는 "블라인드 라운드의 원문 나열"로만 한정 — "결정 요청 문구 노출 금지"는 계속 유효함을 병기
4. brainstorming·plan-stage의 decision-flow Read 의무 시점을 "첫 서브에이전트 투입 전"으로 앞당김 (현행은 Step A·Step 2가 구멍)
5. 문체 펜스 적용 경로 목록을 명시적으로 확정 — 미답변 조사 프롬프트 포함 여부까지 못박아 성공 기준 검증 가능하게
6. 발언 형식은 기존 「출력 형식」 절("자유 형식·정형 태그 금지")의 라운드 유형별 재서술로 반영 — 문장 추가만 하면 상충 규범 공존
7. "새 기여를 만들 수 없으면 지어내지 말고 생략하라"를 의무·생략 규칙의 접합부에 한 자리 배치 (억지 기여 방지 — TD·EA 독립 수렴 지점)
8. plan-stage 블라인드 판정 프롬프트 구성: "도메인 페르소나 정의 + 판정 기준(✅/⚠️/🚫) + 언어 펜스 + 정형 형식" — 사고 제약 원문 불포함을 명문화 (판정용 원문이 존재하지 않음을 설계로 확정)
9. 나열 순서 회전 상태 유실 시 "임의 순서로 재시작" 허용 — 유실을 실패로 만들지 않음
10. supersede 대상은 정책 ID가 아니라 원문 문장 인용으로 지정 — ID 없는 레거시 문서에서 LLM 판정의 유일한 앵커. 첫 머지의 ID 부여 질문은 정상 동작
11. context-handling과 persona-resolution의 HANDOFF 복구 문장을 같은 표현으로 정합 (한쪽 원본·한쪽 참조 구조)
12. 규범 스윕 경로 분리 명시 — 스킬·README·CLAUDE.md는 직접 수정, 도메인 문서 재서술은 supersede 경유 (이중 반영 방지)
13. 이슈 카드 확인 문구에 빈 슬롯 예시를 그리지 않음 — 보이는 칸은 채워진다 (HANDOFF 빈 자리 표시자 함정과 동일 원리)
14. "블라인드 일치도 강한 증거가 아니라 약한 신호" 문장을 블라인드 규칙 문면에 포함
15. 언어 펜스 위반 발언 발견 시 tasks/lessons.md 기록 — 기존 자기개선 루프에 관측 고리 연결
16. 규범 의미 변경("DEVELOP 페르소나 없음" 예외 신설, "자유 형식" 재서술, §7 예외)은 재서술 문장 grep 인벤토리 전수 스윕 — 범위는 skills/ 밖(README·CLAUDE.md)까지 (development-principles 「규범 변경 스윕」)
