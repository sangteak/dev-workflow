## 0. 검증 방식 (블라인드 도그푸딩)

국면 2에서 확정한 "판정·평가 라운드 상시 블라인드" 원칙을 본 검증에 선행 적용했다 — 🛠️ TD 기술 검토와 🔍 Ecosystem Analyst 검토를 상호 미노출 병렬 서브에이전트로 실행하고, 취합은 원문 우선(재요약 최소화)으로 제시. 두 검토가 독립적으로 같은 지점("새 기여 의무 ↔ 생략 허용"의 접합 명시)을 짚어 수렴을 확인했다.

## 4. 기술 가이드라인 (설계 문서 §8로 승계)

1. decision-flow 3부 분리는 항목 단위 재배치표를 먼저 만들고 실행 (문단 통째 이동 금지)
2. 언어 규범 펜스 머리에 적용 범위 선언 1줄: "본 파일의 결정 흐름 적용 범위와 별개로 모든 사용자 대면 문면에 적용"
3. §7 예외는 "블라인드 라운드의 원문 나열"로만 한정 — "결정 요청 문구 노출 금지"는 계속 유효함을 병기
4. brainstorming·plan-stage의 decision-flow Read 의무 시점을 "첫 서브에이전트 투입 전"으로 앞당김
5. 문체 펜스 적용 경로 목록을 명시적으로 확정 (미답변 조사 프롬프트 포함 여부 포함) — 성공 기준 검증 가능성 확보
6. 발언 형식은 기존 "자유 형식" 절의 재서술로 반영 (문장 추가 금지 — 상충 규범 공존 방지)
7. "새 기여를 만들 수 없으면 지어내지 말고 생략하라"를 의무·생략 규칙의 접합부에 명시 (한 자리 배치)
8. plan-stage 블라인드 판정 프롬프트는 "도메인 페르소나 정의 + 판정 기준(✅/⚠️/🚫) + 언어 펜스 + 정형 형식"으로 구성 — 사고 제약 원문 불포함을 명문화
9. 나열 순서 회전 상태 유실 시 "임의 순서로 재시작" 허용 (유실을 실패로 만들지 않음)
10. supersede 대상은 정책 ID가 아니라 원문 문장 인용으로 지정
11. context-handling과 persona-resolution의 HANDOFF 복구 문장을 같은 표현으로 정합 (한쪽 원본·한쪽 참조)
12. 규범 스윕은 "스킬·README·CLAUDE.md 직접 수정 / 도메인 문서는 supersede 경유" 분리 명시
13. 이슈 카드 확인 문구에 빈 슬롯 예시를 그리지 않음 (보이는 칸은 채워진다)
14. "블라인드 일치도 강한 증거가 아니라 약한 신호" 문장을 스킬 문면에 포함 (EA 조건)
15. 언어 펜스 위반 발언 발견 시 lessons.md 기록 — 기존 자기개선 루프에 관측 고리 연결 (EA 조건)
