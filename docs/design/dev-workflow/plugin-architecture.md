> 플러그인 내부 구조 — 스킬 파일의 구성·로드 비용·분할 규범과 그 검증 체계를 관리한다. 스킬을 수정하거나 신설할 때 지켜야 할 상시 규범의 SSOT다.

## 시스템 개요

dev-workflow는 코드 없는 문서 기반 플러그인으로, 모든 로직이 Markdown 스킬 파일에 정의된다. 따라서 스킬 파일의 크기와 구조가 곧 실행 비용이다. 성능 지표는 디스크 KB가 아니라 **invoke당 실질 로드 비용**이다 — v1.12.0 세션 시작 재설계(지연 활성화) 이후 상시 주입은 약 2.5KB로 억제되어 있고, 나머지 비용은 스킬이 invoke될 때 본문이 로드되는 순간 발생한다.

2026-07-07 Fable5 적합성 리뷰가 스킬 본문 슬림화를 권고했고, G4(v1.13.0)에서 방법론이 확정됐다: 수단은 재작성 압축이 아니라 **삭제(증명된 축자 중복)와 verbatim 분할(대형 스킬의 참조 자료)**이다. 재작성형 압축은 기계 검증이 불가능한 의미 드리프트를 낳으므로 전량 배제한다. 이 원칙에 따라 brainstorming과 merge-to-domain의 참조 자료가 `references/templates.md`로 분할됐다 (각 10블록, 행동 등가성 유지).

## 스킬 본문 슬림화 규범

**절감 수단은 두 가지만 허용한다:**

- 삭제 — 실측 근거가 있는 축자 중복 블록만 제거한다
- verbatim 분할 — 참조 자료를 분리 파일로 문장 수정 없이 이동한다. 이동 중 수정이 필요해지면 별도 커밋으로 분리한다

**분할 기준:** 본문 15KB 이상 **그리고** 참조 자료가 분리 가능한 구조일 때만 분할한다. 크기 단독 기준은 일체형 파일(안무·규범이 본문과 얽힌 workflow-orchestrator, decision-flow.md 등)의 오분할을 유발하므로, 일체형 파일은 크기와 무관하게 분할 대상이 아니다 — 분할 시 다중 Read 역효과가 난다.

**분할 구조:**

- SKILL.md 본문에는 흐름 제어·판단 기준·게이트·안전 크리티컬 앵커만 잔존한다
- 출력 템플릿·서브에이전트 프롬프트 전문·긴 예시는 `skills/[스킬]/references/[주제].md`로 verbatim 이동한다 (Superpowers 관행 준용)
- 각 디스패치 지점에 명시 Read 지시("references/[주제].md의 [블록명]을 Read하여 사용하라")와 Read 실패 시 최소 인라인 폴백 1~2줄을 둔다

**도달성:** 분리 파일 참조는 Skill invoke 시 하네스가 제공하는 "Base directory for this skill" 상대 경로를 사용한다. 리포 상대 경로는 배포 환경(플러그인 캐시)에서 도달 불가가 실증되어 금지한다 (v1.10.0 교훈).

**다중 모델 제약:** Fable 5 전용 최적화를 금지한다. 분리 파일 Read 지시는 약한 모델의 참조 추적 한계에 대응해 디스패치 지점마다 명시한다. 사용자 노출 출력 템플릿은 명시적 화이트리스트에 오른 항목 외에는 바이트 보존한다.

## 검증 규범

슬림화·분할 변경의 검증은 이원 체계다 — "행동 등가성 증명"은 불가능한 속성이므로 검증 연극을 피하고, 기계로 확인 가능한 속성과 판단이 필요한 속성을 분리한다:

- **기계 체크 (3튜플 전수):** 베이스라인 불변식에 대한 정확 앵커(인용문/정규식/해시)를 변경 전에 확정하고, 변경 후 존재 grep · 해시 동일 · 개수 일치를 전수 확인한다
- **판단 리뷰:** 신선한 눈 1회가 판단 리뷰 목록(음성 불변식 감사 포함)을 전담한다

**작업 순서 강제:** 베이스라인 추출 → 압축/분할 → 기계 체크 → 판단 리뷰. 역전을 금지한다.

**음성 불변식:** diff에는 삭제·verbatim 이동·참조 삽입만 허용한다. 신규 규범 문장은 승인 목록을 경유해야 한다.

**수치 프록시 금지:** 전역 KB 감소율 같은 수치 목표는 굿하트 지표라 설정하지 않는다 — 수치를 맞추려 load-bearing 중복까지 깎는 유인이 생긴다. 합격선은 항목 전수 완수 + 불변식 통과다. 수치 프록시를 계획에 넣을 때는 이동 가능 바이트의 구조적 상한을 먼저 실측하고, 프록시가 깨지면 원래 증명하려던 직접 속성(byte-identical 이동 + 순증 전량 분류)으로 대체 판정한다.

**인벤토리 수명:** 검증용 불변식 인벤토리는 해당 작업 전용 일회용이다. 상시 린트로 승격하지 않는다 — 머지 후 스테일해진 체크리스트는 오탐/무시를 낳아 신뢰를 오염시키고, 코드 없는 플러그인 정체성과도 긴장을 만든다.

## 관련 파일

| 파일 | 용도 |
|------|------|
| `skills/brainstorming/references/templates.md` | brainstorming 분할 산출물 (템플릿·프롬프트 전문 10블록) |
| `skills/merge-to-domain/references/templates.md` | merge-to-domain 분할 산출물 (digest 형식·게이트 출력 10블록) |
| `skills/brainstorming/SKILL.md` | 분할 적용 본문 (디스패치 지점 Read 지시 + 폴백) |
| `skills/merge-to-domain/SKILL.md` | 분할 적용 본문 (동일 구조) |
| `hooks/` | 세션 시작 주입 (상시 주입 예산 ~2.5KB의 관리 지점) |

## 핵심 결정 사항

| 결정 사항 | 선택 | 근거 |
|-----------|------|------|
| 절감 수단 | 삭제 + verbatim 분할 | 드리프트 리스크 구조적 0 근접. 재작성 압축(공통부 통합·요약화)은 기계 검증 불가능한 의미 손실로 전량 기각 |
| 분할 기준 | 본문 15KB 이상 AND 참조 자료 분리 가능 구조 | 크기 단독 기준은 일체형 파일 오분할 유발. 안무·규범 일체형은 분할 시 다중 Read 역효과 |
| 분할 파일 경로 | 스킬 base dir 상대 `references/` | invoke 시 하네스가 base dir 제공 — 도달성 구조 보장. 리포 상대 경로는 배포 환경 도달 불가 실증 |
| 검증 방식 | 3튜플 기계 체크 + 판단 리뷰 이원 | "행동 등가성 증명"은 불가능한 속성 — 검증 연극 회피. 검증 에이전트 패널 단독은 동의 편향·순환 논증 |
| 인벤토리 수명 | 작업 전용 일회용 | 머지 후 스테일 → 오탐/무시. 죽은 체크리스트의 신뢰 오염 방지. 상시 린트 승격은 새 유지보수 표면 |
| 합격선 | 미설정 (항목 전수 완수 + 불변식 통과) | 전역 KB는 굿하트 지표. 수치 맞추려 load-bearing 중복까지 깎는 유인 차단 |
| 내용 정정의 커밋 분리 | 슬림화 커밋과 별도 커밋 | 내용 변경은 음성 불변식 위반 — diff 감사 순수성 유지 |
