## 배경 — 사용자 문제 제기 (원문 요지)

실사용 세션(소비 프로젝트의 게임 매칭 서버 설계 브레인스토밍)에서 두 문제가 제기됨:

1. **맥락 유실**: "내용이 길어져서 실제 어떤 주제에 대한 답변인지 알 수가 없다. 어떤 문제가 있고 어떻게 처리되어야 할지, 진짜 문제인지, 토론 내용의 전체 맥락을 이해할 수 있게 해달라."
2. **미검증 가정의 팩트 행세**: "가정·반사실은 팩트가 아니라 모델의 고민일 뿐이다. 정말 중요한 게 아니면 말할 필요가 없고, 가정을 했으면 팩트 체크를 해서 그 결과를 달라."

문제 사례: 대화형 라운드의 Contrarian 발언이 "가정 A/B/C, 반사실 1/2" 나열 벽 텍스트로 원문 중계된 화면 (블라인드 라운드는 정형 강제로 해당 없음).

## 사용자가 명시적으로 제외한 항목과 이유

- **상시 진행 지도** (매 라운드 반복 출력되는 전체 현황판) — 길이 문제를 오히려 키움 (시뮬레이션 비교 후 기각)
- **확인 불가 가정의 전부 침묵** — 결정을 좌우하는 경고(예: "유예 규칙 미정 → 강행형 평가 불가")까지 지워 조용히 틀리는 화면이 됨 (시뮬레이션 비교 후 기각)
- **본체가 발언을 압축 정형으로 고쳐 쓰기** (최초 확정안) — 뭉개짐 사고의 원인 지점(본체 고쳐 쓰기)을 재도입하는 약점. 사용자 재논의 제안으로 "원문 유지 + 주석"으로 대체
- 기존 페르소나 발언 실질성 규칙 재설계, phase 파일 기록 방식 변경 — 스코프 외

## 페르소나 피드백 결과 (주제별)

### 주제 1: §7 블라인드 규범과 압축 정형의 충돌 [블라인드 라운드]
- **합의 (블라인드 규범 공존)**: 충돌은 허구 — 압축은 생산 규격(프롬프트), 원문 나열은 전달 규격으로 층이 다름. 블라인드 발언은 이미 정형 강제(결론 1줄+근거 3)라 벽 텍스트 통로가 없음. 실제 사고 지점은 대화형 라운드의 비판 담당 원문 중계. §7 유지, 블라인드 프롬프트에 출처 의무만 추가.
- 두 페르소나가 독립적으로 동일 결론 도달. 본체가 인용 행번호(decision-flow.md 151~153·160·162행) 재확인 완료.

### 주제 2: 적용 방식 3논점 [대화형 라운드 · 비판: 🛠️ Contrarian]
- 논점 1(적용 범위): 별도 범위 지도는 §8과 새 충돌면 — decision-flow 원본 편입으로 자동 상속하자는 대안 채택. 🎯 보강 근거: 화면 형식이 단계마다 다르면 사용자가 읽는 법을 다시 배워야 함.
- 논점 2(팩트체크 주체): "본체 직접 vs 워커 위임" 이분법 기각 — 1차 검증자는 이미 서브에이전트인 페르소나 자신. 출처 유형 분리안 채택.
- 논점 3(선별 기준): "세 후보 기준이 겹친다"는 Contrarian 스스로 [추측] 표시 — 표시 기반 자동 규칙 + 관측 조정안 채택.
