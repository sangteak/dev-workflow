# grounded-output 설계 문서

> 한 줄 요약: 결정 요청이 그 화면만으로 판단 가능하고, 화면의 모든 주장이 "확인됐거나 확인 불가라고 자백한" 상태가 되도록 출력 규범을 개편한다.

## 1. 배경과 동기

- 실사용 세션(소비 프로젝트의 매칭 서버 설계)에서 두 문제 확인:
  - **맥락 유실**: 긴 토론 뒤 결정 요청이 도착하면 어떤 문제에 대한 것인지, 진짜 문제인지, 토론에서 무슨 근거가 오갔는지 파악 불가 — "전체 내용을 파악할 수 있게 해달라"
  - **미검증 가정의 팩트 행세**: 비판 페르소나가 "가정 A/B/C, 반사실 1/2" 같은 머릿속 시나리오를 확인 없이 나열 — "중요하지 않으면 말하지 말고, 중요하면 팩트 체크 결과와 함께 달라"
- 벽 텍스트의 발원지는 블라인드 라운드가 아니라 대화형 라운드의 비판 담당(Contrarian) 원문 중계였음 (블라인드는 정형이 이미 강제됨)

## 2. 목표와 비목표

### 목표
- GOAL-001 (자기완결): 결정 요청 화면 하나만 읽고 추가 스크롤 없이 판단할 수 있다
- GOAL-002 (근거 있는 출력): 팩트와 추측의 경계가 표시로 드러나고, 화면에 올라가는 주장의 검증 책임을 본체(화면을 조립하는 메인 컨텍스트)가 진다
- GOAL-003 (피드백 루프): 소비 프로젝트가 관측 기록으로 로컬에서 발전하고, 수동 제보로 본체 규범에 환류된다

### 비목표
- 상시 진행 지도 (매 라운드 반복 출력 현황판) — 길이 문제를 키움
- 자동 검사·수집 장치 (런타임 훅·자동 로그) — 지연·굿하트 위험, "자동 루프성 기능은 플러그인 밖" 기존 결론 유지
- 확인 불가 가정의 전부 침묵 — 결정을 좌우하는 경고까지 지움 (선별 침묵이다)
- 본체의 발언 고쳐 쓰기 — 뭉개짐 사고의 원인 지점 (재논의로 명시 기각)
- 기존 발언 실질성 규칙 재설계, phase 파일 기록 방식 변경

## 3. 확정된 요구사항

- REQ-001 (4요소 결정 박스): 결정 박스에 ① 다루는 문제(앞선 결정과의 연결) ② 확인된 사실 ③ 확인 못 한 것 ④ 토론 요지. 출신 2등급 — 토론·조사 선행 결정=4칸 상시(빈 칸은 "없음" 명시, 생략 금지) / 표준 결정=문제 칸 필수+이력 있는 칸 / 로컬 템플릿·승인 게이트=기존 형식 유지. 등급 판정은 기계적 술어만("토론·조사가 선행됐는가") — 우선순위: HIGH
- REQ-002 (생산 규격): 페르소나 발언은 결론 1줄 + 근거 불릿 3개 이내 + 전제 반박 1개. 근거마다 전치 태그(출처 또는 [추측] 자백). 근거가 3개 미만이면 부족 사유 1줄로 대체 가능 (채움 목표 아님) — 우선순위: HIGH
- REQ-003 (원문 전달): 본체는 발언을 고쳐 쓰지 않고 원문 그대로 나열. 규격 위반 시 재실행 1회(화면에 "발언 1건 형식 위반 — 다시 받는 중" 표시), 재위반 시 발언 제외 + 고지(문구 실물 규정) — 우선순위: HIGH
- REQ-004 (검증 표기): 검증 출처 3종 — 계산·논리 / 프로젝트 내 코드·문서 / 외부 공식 문서. 전치 태그 [✔ 출처]·[⚠️ 확인 불가]·[✘ 반증 — 출처]·[추측]을 항목 첫머리에. 세션 최초 [✔] 출현 시 태그의 뜻 1회 고지 — 우선순위: HIGH
- REQ-005 (본체 관문): 재확인 대상 = "미확인([추측]/무표시)인데 추천 사유·선택지 성패 근거로 쓰인 주장". [✔] 주장은 첨부 증거 대조만. 규약: "화면에 도달한 [✔] = 본체 관문 통과". 본체 주석은 ↳ 단독 마커(들여쓰기, 불릿 병용 금지)로 반증·확인 불가 승격·새 사실 추가 시에만. 관문 수행 주체는 본체 고정 (워커 하향 금지) — 우선순위: HIGH
- REQ-006 (근거 첨부 의무): 프로젝트 내 출처는 파일 경로+행 번호 또는 한 줄 인용, 외부 출처는 워커가 판정+출처 주소+원문 인용 반환. 인용 없는 보고는 미확인 강등(강등 사실 가시화) — 우선순위: HIGH
- REQ-007 (선별 침묵): 확인 불가 가정은 결정에 영향을 주는 것만 ⚠️로 승격 표시, 영향 없는 추측은 미출력 — 우선순위: MEDIUM
- REQ-008 (소비자 피드백 루프): 소비 프로젝트당 관측 파일 1개 (플러그인 UX 사고 전용 — 되묻기·정정·재확인 누락, 항목 1줄 서식, 파일 머리 용도 1줄, 수동 기록·같은 유형 고지는 첫 회만). 세션 시작 교훈 검토에 포함 (파일 부재 시 조용히 스킵). `/dev-workflow:feedback` 명령이 Git 이슈 제보 초안을 텍스트로 생성 (자동 업로드·gh 실행 없음) — 우선순위: MEDIUM — **구현 범위: 후속 MINOR로 분리 (PLAN 재협의 확정 — 출력 규범 릴리스 후 첫 작업)**
- REQ-009 (언어 규범 보강): 사용자 대면 문면에서 절 기호(§) 등 법조문식 표기 금지 — 문서 이름+절 이름을 말로 풀어 쓴다. 가이드라인·목록의 슬래시 압축 금지를 산출물만이 아니라 중계·요약 문면까지 확대 명시 — 우선순위: MEDIUM
- REQ-010 (압축 생존): 등급 판정 근거·재실행 이력·관문 통과 이력을 결정 규범 문서의 상태 보존 절(HANDOFF 기록·복원)에 등재 — 우선순위: MEDIUM
- REQ-011 (규범 배치와 스윕): 새 규범은 decision-flow.md 원본에 편입 (적용 범위 자동 상속, 별도 범위 조항 없음). 신설 상세는 references/ 분리 + 원본에 포인터 (상시 주입량 억제). 스윕 대상 구 문구 목록을 착수 전 확정 — 「서브에이전트 출력 재구성 책임」 절 필수 포함 — 우선순위: HIGH

## 4. 설계 개요

3계층 구조 — 길이·보존·신뢰를 서로 다른 장치가 맡는다:

1. **생산 계층** (서브에이전트 프롬프트): 정형·분량 상한·전치 태그·근거 첨부를 발언이 태어나는 지점에서 강제 → 벽 텍스트 원천 차단
2. **전달 계층** (본체): 원문 그대로 나열 + ↳ 검증 주석만 부착 → 뭉개짐 원천 차단 (고쳐 쓰기 금지, 위반은 재실행으로)
3. **관문 계층** (본체): 표시 기반 선별 재확인 → 화면 도달 = 검증 통과라는 신뢰 규약 성립

소비 지점은 4요소 결정 박스 (자기완결 — 되묻지 않아도 되는 화면). 전체 위치는 기존 한 줄 진행 헤더가 유지. 피드백 루프(관측 파일→세션 적용→수동 제보)가 규범 조정의 관측을 닫는다.

## 7. 제약조건과 가정

- 자동 형식 검사기 없음 — 준수는 모델의 지시 추종에 의존. 따라서 규격은 지시문 한 문단으로 요약 가능해야 하며, 등급·예외 판정에 주관 술어 금지
- 소비자 관측의 자동 회수 통로 없음 — 환류는 수동 제보(Git 이슈)만 성립
- 네트워크 차단 환경에서는 외부 출처 축 전체가 미확인으로 강등됨 (규범에 명시)
- 생산 규격은 페르소나가 대체로 준수한다고 가정 — 재실행 1회가 안전망 (이번 세션 라운드 4회 실측으로 지지)
- 컨텍스트 압축 후 상태 소실 가능 — REQ-010(압축 생존)이 대응

## 8. 기술 가이드라인

- 규범 배치: 신설 상세는 references/ 분리, decision-flow.md에는 포인터만 (본문/참조 분할 전례 준용)
- 스윕: 대상 구 문구 목록을 착수 전 확정하고 커밋에 남김. 「서브에이전트 출력 재구성 책임」 절 필수 포함. sweep + bump-version --check를 릴리스 게이트로
- Contrarian 교체: templates.md 래퍼에서만 수정 — agent-roles.md verbatim 계약("문장 수정 금지") 유지
- 등급 판정: 단일 질문("토론·조사가 선행됐는가") + 경계 사례 예시 2~3개 수록 — 낮은 등급 도주 방지
- 길이 역화 방지: 4칸 각 줄 1줄 상한 / 근거 3은 상한이지 채움 목표 아님 / 부족 사유 1줄 탈출구
- 가시화: 재실행 진행 표시 / 발언 제외 고지 문구 실물 규정 / 워커 인용 미첨부 강등 고지 / 소급 반증 시 대상 발언을 이름으로 지칭하는 형식 규정
- [✔] 태그 최초 고지: 세션 첫 출현 시 "이 표시는 본체가 증거를 대조해 통과시켰다는 뜻" 1줄 (블라인드 최초 고지 패턴 준용)
- 관측 파일: 부재가 기본 상태 (없으면 조용히 스킵) / 항목 1줄 서식 / 파일 머리 용도 1줄 / "없음" 칸 3연속 빈발 시 등급 술어 과잉 신호로 기록
- 관측 후 조정 후보 (1차 릴리스 미포함): 결정 근거로 쓰인 [✔] 주장의 표본 재확인 — 잘못 붙은 [✔] 통과 사례가 관측되면 추가
- 도입 후 실측: 관측 파일 기록 건수 확인 — 0에 가까우면 서식이 아니라 기록 트리거부터 보강
