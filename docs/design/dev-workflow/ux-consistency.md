> 워크플로우 전반의 사용자 경험 일관성을 보장하기 위한 페르소나 피드백 구조, 상태 메타데이터 표준, 사용자 입력 형식 표준을 정의한다.

## 시스템 개요

dev-workflow는 브레인스토밍부터 리뷰까지 여러 단계를 거치며, 각 단계에서 페르소나가 참여하고 설계 문서가 생성된다. 이 과정에서 세 가지 일관성 문제가 발생한다. 첫째, 페르소나의 피드백이 체계적 구조 없이 흩어지면 다관점 검증의 품질이 들쭉날쭉해진다. 둘째, 설계 문서의 상태 메타데이터가 통일되지 않으면 인덱싱과 참조에서 누락이 발생한다. 셋째, 사용자 입력을 받는 지점마다 형식이 다르면 환경과 세션에 따라 경험이 달라진다. 이 domain은 페르소나 피드백의 구조화된 루프, 상태값 표준화, 입력 형식 통일이라는 세 축으로 워크플로우 전반의 UX 일관성을 확보한다.

## 페르소나 피드백 루프

brainstorming 스킬은 원래 국면 2(발견)에서 "충돌/보완" 규칙만 존재했으며, 정형화된 피드백 루프가 없었다. workflow-orchestrator가 "persona-based feedback loops"를 선언하지만 실제 구현은 plan-stage에만 있었고, brainstorming의 발산적 성격에 맞는 별도 메커니즘이 필요했다.

피드백 루프는 매 주제/질문마다 발동되며, 최대 3회로 제한된다. 초과 시 사용자 승인을 받아야 추가 토론이 가능하다. 각 라운드는 `[라운드 N/3]` 카운터로 표기하여 진행 상황을 명시한다.

비판적 관점은 라운드 로빈으로 할당하며, 라운드 시작 시 비판적 역할을 맡는 페르소나를 명시한다. 비판 60%, 건설적 의견 40%의 비율을 유지하되, 비판 시에는 반드시 대안 또는 출구를 함께 제시해야 한다.

매 주제 시작 시 사용자에게 토론 리드를 묻는다. 선택지에 "없음"을 포함하여 사용자 주도권을 보장한다. 리드 선택 시 선정 이유를 한 줄로 명시하고 비판적 비중을 약간 높이며, "없음" 선택 시 모든 페르소나가 동등하게 참여한다. 동일 페르소나가 연속 2회 리드를 맡는 것은 제한된다.

의견 본문은 페르소나 이모지와 이름 접두사를 붙인 자유 형식으로 작성한다. 정형 태그(찬성/반대 등)는 사용하지 않는다. 라운드 종료 시 `**합의:**` 또는 `**미합의:**` 마커와 결론 요약을 붙인다. 이는 brainstorming의 발산적 성격을 보존하면서도 결론을 구조화하기 위한 선택이다.

국면별로 강제 비율은 두지 않되 방향성 가이드를 제공한다. 국면 1에서는 비판보다 호기심을, 국면 3에서는 리스크를 놓치지 않도록 안내한다. phase 파일에는 토론 과정을 제외하고 합의/미합의 결론만 주제별로 그룹핑하여 기록한다.

plan-stage의 피드백 루프와는 의도적으로 별도 유지한다. 스킬 간 의존성을 방지하고 각자 독립적으로 진화할 수 있도록 하기 위함이다.

## 상태 메타데이터 표준

설계 문서 프론트매터의 `status` 필드에 `complete`와 `completed`가 혼용되면서, `design-doc-index`와 `design-summary` 스킬이 `status: complete`만 필터링하여 `completed`로 표기된 문서가 인덱싱에서 누락되는 문제가 발생했다. 근본 원인은 plan.md가 최종 상태를 `completed`로 명시했고, Claude가 `document-consolidation` 스킬의 `complete` 지시보다 plan의 값을 우선 적용한 것이다.

정규 상태값은 `ready-for-plan`과 `complete` 2개로 확정한다. 중간 상태(`in-progress`, `in-development`, `archived`)는 도입하지 않는다. phase 파일(phase1, phase2, phase3, plan.md)이 진행 추적 역할을 대체하므로 상태 라이프사이클은 2단계로 충분하다.

```
ready-for-plan (브레인스토밍 완료) → complete (개발+리뷰+문서통합 완료)
```

이 문제는 쓰기 쪽과 읽기 쪽 양쪽에서 방어한다.

**쓰기 쪽** — `document-consolidation` 스킬의 프론트매터 업데이트 규칙에 우선순위 규칙을 추가한다. plan.md 등 다른 소스에 명시된 값에 관계없이, 프론트매터 status는 반드시 정규값 `complete`를 사용한다.

**읽기 쪽** — `design-doc-index`와 `design-summary` 스킬에 인덱싱 본 로직 전 상태 정규화 전처리 단계를 분리한다. 프론트매터를 순회하여 `completed`를 `complete`로 자동 수정한 뒤 기존 필터링 로직을 그대로 실행한다. 허용 변형은 `completed` 1개만이며, 한국어(`완료`)는 제외한다. 시스템 메타데이터에 한국어 값이 들어갈 가능성이 없기 때문이다.

Markdown 스킬은 코드처럼 함수를 공유할 수 없으므로, 전처리 절차를 `design-doc-index`와 `design-summary` 양쪽에 동일한 표현으로 명시하여 유지보수 시 불일치를 방지한다.

## 입력 형식 표준

dev-workflow 플러그인에서 사용자 입력을 받는 36개 지점이 4가지 서로 다른 패턴(Yes/No 텍스트, 번호/문자 선택, 자유 텍스트, AskUserQuestion 도구)으로 혼재되어 있었다. 스킬 문서에 "무엇을 물어볼지"만 정의되어 있고, "어떤 도구/형식으로 물어볼지"는 미지정이었다. Claude가 비결정적으로 AskUserQuestion 도구 사용 여부를 결정하여 동일 사용자가 데스크톱과 노트북에서 다른 경험을 하는 문제가 발생했다.

모든 스킬에서 사용자 입력 형식을 텍스트 기반 번호 목록으로 통일하고, 스킬 문서에 입력 형식을 명시하여 Claude의 재량 여지를 제거한다. 규칙은 Claude의 출력 형식(질문 제시 방식)에만 적용되며, 사용자는 번호 또는 자연어로 자유롭게 응답할 수 있다.

### 입력 유형별 형식

**닫힌 선택 (선택지가 있는 질문)**
```
[질문 텍스트]

1. 선택지A
2. 선택지B
3. 선택지C
0. 직접 입력          ← 자유 입력 탈출구 필요 시만 추가
```

**Yes/No 질문**
```
[질문 텍스트]

1. Yes
2. No
```

**열린 입력 (선택지 없는 자유 텍스트)**
```
[질문 텍스트]
> 예: "[예시 입력]"
```

### 적용 규칙

- AskUserQuestion 도구를 사용하지 않는다. workflow-orchestrator의 CRITICAL 섹션에 명시적 금지로 배치하여 우선순위를 최대로 높인다.
- 모든 닫힌 선택은 텍스트 기반 번호 목록 형식으로 제시한다.
- Yes/No 질문도 번호 목록으로 통일한다 (`1. Yes  2. No`).
- 열린 입력은 번호 없이 예시 안내 형식으로 제시한다.
- 닫힌 선택에서 직접 입력이 필요한 경우, 0번을 "직접 입력" 예약 번호로 사용한다. 위치가 매번 달라지는 마지막 선택지 방식 대신 고정 번호를 사용하여 일관성을 확보한다.
- 문자 선택(A/B/C/D), 슬래시 구분(Yes / No), 괄호 안 선택지 나열을 금지한다.
- 사용자가 번호, "응", "Yes", "ㅇㅇ", "첫번째" 등으로 답해도 의미가 명확하면 수용한다. 의미가 모호한 경우에만 재확인한다.

### 적용 구조

규칙은 workflow-orchestrator에 Input Format Rules 섹션으로 추가하고, 9개 스킬의 기존 프롬프트 텍스트를 새 형식으로 일괄 수정한다. 규칙 분량이 10~15줄이므로 별도 스킬을 생성하지 않고 orchestrator를 확장하는 것이 적절하다.

```
workflow-orchestrator (SKILL.md)
├── CRITICAL 섹션: AskUserQuestion 도구 사용 금지
├── Input Format Rules 섹션: 입력 형식 표준 정의
└── Output Format Rules 섹션: (기존)

각 스킬 (SKILL.md)
└── 기존 프롬프트 텍스트 → 새 형식으로 교체
```

## 결정 요청 흐름

dev-workflow는 BRAINSTORM/PLAN 단계에서 페르소나 토론 후 사용자에게 결정을 요청하는 빈번한 패턴을 갖는다. 결정 항목이 한 번에 여러 개 나열되면 사용자 인지 부담과 라운드트립 중 맥락 손실이 발생한다. 본질 문제는 출력 형식의 단계적 흐름 부재와 결정 분할/통합 메타 규칙 부재 두 가지다. 이를 해결하기 위해 결정 흐름의 메타 원칙을 단일 SSOT로 정의한다.

규범은 `skills/workflow-orchestrator/decision-flow.md` 단일 파일에 집중되며, 4개 핵심 스킬(workflow-orchestrator, brainstorming, plan-stage, document-consolidation)이 SSOT를 가리킨다. orchestrator/brainstorming/development-principles에 분산 삽입하는 방식은 추상화 레벨 부정합(런타임 행위 원칙 vs 메타-가이드)과 컨텍스트 비용 비대칭(DEVELOP/REVIEW 세션에도 매번 결정 흐름 텍스트 영구 로딩)을 발생시키므로 채택하지 않는다. development-principles는 변경 없이 런타임 행위 원칙의 추상화 순수성을 유지한다.

**v1.10.0 도달성 강화:** v1.9.0 배포 리뷰에서 "1줄 수동 참조 + 리포 상대경로"는 배포 환경(사용자 프로젝트 + 플러그인 캐시)에서 SSOT가 도달 불가하다는 결함이 확정되었다. v1.10.0부터 참조는 능동 Read 지시 + 경로 해소 절차(플러그인 루트 → 스킬 디렉토리 형제 경로 → `find "$HOME/.claude/plugins"` 동적 탐색)로 교체하고, 매 세션 주입되는 orchestrator SKILL.md에 '결정 요청 폴백 형식'(하나씩 + 오픈 박스 + 헤더)을, brainstorming에 토론 출력 최소 규칙(에이전트 태그·합의 마커)을 인라인하여 Read 실패 시에도 핵심 형식이 퇴행하지 않게 한다. 훅에서 decision-flow.md 전문을 주입하는 방식은 컨텍스트 비용 비대칭 논거를 유지하여 보류하고, Read 준수율 문제가 관측되면 재검토한다.

메타 원칙(v1.10.0 재설계)은 2층 구조다. **1층 — 모델 출력 기본값:** 결정('산출물에 기록될 선택지 2개 이상 중의 선택'으로 조작적 정의, 진행/승인 게이트는 Input Format Rules 담당)은 의존성 루트부터 한 번에 하나씩 묻는다. 상호 순환 의존으로 분리 결정이 불가능한 경우에만 종합 질문으로 묶고, 주제 유사성 기반 클러스터 그룹핑은 금지한다. **2층 — 응답 주권:** 사용자 응답이 형식을 이긴다. 일괄 응답·순서 무시·위임·즉시 번복·역질문·인터럽트를 전부 수용하며 이미 답한 항목을 재질문하지 않는다. 그 위에 진행 모드 3종(하나씩[기본]/한꺼번에/알아서)을 두되, 모드 선택은 별도 메타 질문 없이 사전 브리핑 푸터에 접어 넣는다. 모드는 국면 한정이며 세션 복구 시 기본값으로 리셋된다. 의존성 분석은 모델 자율 추론으로 처리하며, 100% 정확도 대신 사후 파기 대기열로 보완한다. 명시적 산출물(의존성 그래프 파일 등)은 생성하지 않는다.

출력 형식은 B+D 하이브리드를 따르되, v1.10.0부터 헤더(B)는 '매 응답'이 아닌 **이벤트 기반**으로 표시한다(결정 요청/카운터 변경/흐름 복귀 시에만 — '매 응답' 규칙은 정보 제공 예외와 자기모순이었고 노이즈가 컸다). 헤더 값(N/M/K)은 대화 기억이 아니라 사전 브리핑 목록의 재집계로 산출한다. 결정 박스(D)는 **우측 개방 오픈형**으로 확정한다 — 우측 닫힘 박스는 한글/이모지 전각 폭으로 정렬 유지가 구조적으로 불가능했다.

동작은 F+E 하이브리드를 따른다. 사전 브리핑(F)의 발동은 시점이 아니라 이벤트다 — 미결 결정이 2개 이상 식별되면 그 시점에 출력하여, 사용자가 전체 그림을 닻으로 삼게 한다. 진행 중 새 결정이 발견되면 의존성을 한 줄로 기술하고 갱신된 순서를 통보하되 별도 응답을 기다리지 않는다(경량 이의 게이트). 앞선 결정으로 무의미해진 항목은 고지 후 제거한다(조용한 제거 금지). 우선순위 라벨(상류/확신도/되돌리기 비용/위험도)을 직접 박는 방식은 마크다운 환경에서 모델의 사후 합리화 위험을 키우므로 채택하지 않는다.

**임계값 8 가드의 폐기 (v1.10.0):** v1.9.0의 "8개 도달 시 분할 제안 → 거절 시 압축 헤더" 메커니즘은 배포 리뷰에서 총체적 결함(분할 제안 자체가 메타 질문 역설, 압축 헤더의 기호 의미 역전, '풀 박스 3개 표시'와 D 템플릿의 구조 모순)이 확정되어 수리 대신 삭제한다. Miller's Law 인지 한계 보호라는 원 취지는 "미결 8개 이상 시 브리핑 푸터에 일괄/위임 모드 권장 한 줄"로 대체한다 — 메타 질문 없이 인지 부하 탈출구를 제공한다. 헤더는 단일 형식만 유지한다.

충돌 처리는 발견 주체로 구분한다. 모델/페르소나가 자체 감지한 충돌은 대기열에 추가(추가 시점 1회 고지)하고 현재 국면의 모든 결정 완료 후 일괄 노출하여 [모두 재논의/일부만/모두 기각] 옵션을 제공하며, phase 파일 작성 직전이 최종 안전망이다. 노출 박스에는 항목 전문 목록을 포함한다(건수만 표시 금지). 사용자가 명시적으로 번복을 요청하면 대기열을 거치지 않고 즉시 처리한다. 결정 박스 없이 통과된 페르소나 합의에도 식별 가능한 이름을 부여해 번복 시 참조를 가능하게 한다.

**상태 보존 (v1.10.0):** 결정 흐름 상태는 대화 컨텍스트에만 존재해 compact/세션 이관 시 유실된다는 결함에 대해, HANDOFF.md에 조건부 '결정 흐름 상태' 섹션(상태 마커 [x]/[>]/[ ]/[-]/[=], 재논의 대기열 전문)을 추가한다. 기록 시점은 save·compact 직전·국면 완료 시로 한정한다 — 결정마다 파일에 기록하는 상시 원장 방식은 대화 템포에 파일쓰기 툴콜이 개입하는 노이즈로 기각했다.

기존 페르소나 피드백 루프와의 관계는 보완적이다. 라운드 카운터·합의/미합의 마커·비판 라운드 로빈은 루프 *진행 중* 메커니즘이며, 결정 요청 흐름은 토론 *후* 사용자 결정 단계에 적용된다. 에이전트 태그(`[Socratic]`, `[Contrarian]`, Standalone Mode 태그 없음)와 결정 요청 분리 규칙(합의 도출 시 박스 생략, "이 두 가지에 대해" 같은 대명사 참조 금지)은 SSOT 내부에 통합되어 있다.

적용 경계는 '로컬 템플릿 우선' 규칙으로 결정론화한다. 스킬 본문에 구체 출력 템플릿이 있는 결정 지점(시드 확인·Simplifier·국면 0·plan-stage 재협의·document-consolidation 단계 확인)은 해당 템플릿을 그대로 출력하고 D 박스로 재조립하지 않는다. document-consolidation은 이미 순차 구조이므로 헤더·브리핑·대기열을 적용하지 않는 명시적 예외로 선언한다(v1.9.0의 "자연스럽게 풀 박스를 따른다" 서술은 같은 파일의 인라인 템플릿과 자기모순이어서 교체). plan-stage의 OPEN_QUESTIONS 템플릿은 "각 항목에 대해 결정을 내려주시면"이라는 일괄 결정 유도 문구가 메타 원칙과 정면 충돌하여 브리핑(F) 정합 구조로 교체했다.
