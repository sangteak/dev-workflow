> 워크플로우 전반의 사용자 경험 일관성을 보장하기 위한 페르소나 피드백 구조, 상태 메타데이터 표준, 사용자 입력 형식 표준을 정의한다.

## 시스템 개요

dev-workflow는 브레인스토밍부터 리뷰까지 여러 단계를 거치며, 각 단계에서 페르소나가 참여하고 설계 문서가 생성된다. 이 과정에서 세 가지 일관성 문제가 발생한다. 첫째, 페르소나의 피드백이 체계적 구조 없이 흩어지면 다관점 검증의 품질이 들쭉날쭉해진다. 둘째, 설계 문서의 상태 메타데이터가 통일되지 않으면 인덱싱과 참조에서 누락이 발생한다. 셋째, 사용자 입력을 받는 지점마다 형식이 다르면 환경과 세션에 따라 경험이 달라진다. 이 domain은 페르소나 피드백의 구조화된 루프, 상태값 표준화, 입력 형식 통일이라는 세 축으로 워크플로우 전반의 UX 일관성을 확보한다.

## 페르소나 피드백 루프

brainstorming 스킬은 원래 국면 2(발견)에서 "충돌/보완" 규칙만 존재했으며, 정형화된 피드백 루프가 없었다. workflow-orchestrator가 "persona-based feedback loops"를 선언하지만 실제 구현은 plan-stage에만 있었고, brainstorming의 발산적 성격에 맞는 별도 메커니즘이 필요했다.

피드백 루프는 매 주제/질문마다 발동되며, 최대 3회로 제한된다. 초과 시 사용자 승인을 받아야 추가 토론이 가능하다. 각 라운드는 `[라운드 N/3]` 카운터로 표기하여 진행 상황을 명시한다.

비판적 관점은 라운드 로빈으로 할당하며, 라운드 시작 시 비판적 역할을 맡는 페르소나를 명시한다. 비판 60%, 건설적 의견 40%의 비율을 유지하되, 비판 시에는 반드시 대안 또는 출구를 함께 제시해야 한다.

매 주제 시작 시 사용자에게 토론 리드를 묻는다. 선택지에 "없음"을 포함하여 사용자 주도권을 보장한다. 리드 선택 시 선정 이유를 한 줄로 명시하고 비판적 비중을 약간 높이며, "없음" 선택 시 모든 페르소나가 동등하게 참여한다. 동일 페르소나가 연속 2회 리드를 맡는 것은 제한된다.

의견 본문은 페르소나 이모지와 이름 접두사를 붙인 자유 형식으로 작성한다. 정형 태그(찬성/반대 등)는 사용하지 않는다. 라운드 종료 시 `**합의:**` 또는 `**미합의:**` 마커와 결론 요약을 붙인다. 이는 brainstorming의 발산적 성격을 보존하면서도 결론을 구조화하기 위한 선택이다.

국면별로 강제 비율은 두지 않되 방향성 가이드를 제공한다. 국면 1에서는 비판보다 호기심을, 국면 3에서는 리스크를 놓치지 않도록 안내한다. phase 파일에는 토론 과정을 제외하고 합의/미합의 결론만 주제별로 그룹핑하여 기록한다.

plan-stage의 피드백 루프와는 의도적으로 별도 유지한다. 스킬 간 의존성을 방지하고 각자 독립적으로 진화할 수 있도록 하기 위함이다.

## 상태 메타데이터 표준

설계 문서 프론트매터의 `status` 필드에 `complete`와 `completed`가 혼용되면서, `design-doc-index`와 `design-summary` 스킬이 `status: complete`만 필터링하여 `completed`로 표기된 문서가 인덱싱에서 누락되는 문제가 발생했다. 근본 원인은 plan.md가 최종 상태를 `completed`로 명시했고, Claude가 `document-consolidation` 스킬의 `complete` 지시보다 plan의 값을 우선 적용한 것이다.

정규 상태값은 `ready-for-plan`과 `complete` 2개로 확정한다. 중간 상태(`in-progress`, `in-development`, `archived`)는 도입하지 않는다. phase 파일(phase1, phase2, phase3, plan.md)이 진행 추적 역할을 대체하므로 상태 라이프사이클은 2단계로 충분하다.

```
ready-for-plan (브레인스토밍 완료) → complete (개발+리뷰+문서통합 완료)
```

이 문제는 쓰기 쪽과 읽기 쪽 양쪽에서 방어한다.

**쓰기 쪽** — `document-consolidation` 스킬의 프론트매터 업데이트 규칙에 우선순위 규칙을 추가한다. plan.md 등 다른 소스에 명시된 값에 관계없이, 프론트매터 status는 반드시 정규값 `complete`를 사용한다.

**읽기 쪽** — `design-doc-index`와 `design-summary` 스킬에 인덱싱 본 로직 전 상태 정규화 전처리 단계를 분리한다. 프론트매터를 순회하여 `completed`를 `complete`로 자동 수정한 뒤 기존 필터링 로직을 그대로 실행한다. 허용 변형은 `completed` 1개만이며, 한국어(`완료`)는 제외한다. 시스템 메타데이터에 한국어 값이 들어갈 가능성이 없기 때문이다.

Markdown 스킬은 코드처럼 함수를 공유할 수 없으므로, 전처리 절차를 `design-doc-index`와 `design-summary` 양쪽에 동일한 표현으로 명시하여 유지보수 시 불일치를 방지한다.

## 입력 형식 표준

dev-workflow 플러그인에서 사용자 입력을 받는 36개 지점이 4가지 서로 다른 패턴(Yes/No 텍스트, 번호/문자 선택, 자유 텍스트, AskUserQuestion 도구)으로 혼재되어 있었다. 스킬 문서에 "무엇을 물어볼지"만 정의되어 있고, "어떤 도구/형식으로 물어볼지"는 미지정이었다. Claude가 비결정적으로 AskUserQuestion 도구 사용 여부를 결정하여 동일 사용자가 데스크톱과 노트북에서 다른 경험을 하는 문제가 발생했다.

모든 스킬에서 사용자 입력 형식을 텍스트 기반 번호 목록으로 통일하고, 스킬 문서에 입력 형식을 명시하여 Claude의 재량 여지를 제거한다. 규칙은 Claude의 출력 형식(질문 제시 방식)에만 적용되며, 사용자는 번호 또는 자연어로 자유롭게 응답할 수 있다.

### 입력 유형별 형식

**닫힌 선택 (선택지가 있는 질문)**
```
[질문 텍스트]

1. 선택지A
2. 선택지B
3. 선택지C
0. 직접 입력          ← 자유 입력 탈출구 필요 시만 추가
```

**Yes/No 질문**
```
[질문 텍스트]

1. Yes
2. No
```

**열린 입력 (선택지 없는 자유 텍스트)**
```
[질문 텍스트]
> 예: "[예시 입력]"
```

### 적용 규칙

- AskUserQuestion 도구를 사용하지 않는다. workflow-orchestrator의 CRITICAL 섹션에 명시적 금지로 배치하여 우선순위를 최대로 높인다.
- 모든 닫힌 선택은 텍스트 기반 번호 목록 형식으로 제시한다.
- Yes/No 질문도 번호 목록으로 통일한다 (`1. Yes  2. No`).
- 열린 입력은 번호 없이 예시 안내 형식으로 제시한다.
- 닫힌 선택에서 직접 입력이 필요한 경우, 0번을 "직접 입력" 예약 번호로 사용한다. 위치가 매번 달라지는 마지막 선택지 방식 대신 고정 번호를 사용하여 일관성을 확보한다.
- 문자 선택(A/B/C/D), 슬래시 구분(Yes / No), 괄호 안 선택지 나열을 금지한다.
- 사용자가 번호, "응", "Yes", "ㅇㅇ", "첫번째" 등으로 답해도 의미가 명확하면 수용한다. 의미가 모호한 경우에만 재확인한다.

### 적용 구조

규칙은 workflow-orchestrator에 Input Format Rules 섹션으로 추가하고, 9개 스킬의 기존 프롬프트 텍스트를 새 형식으로 일괄 수정한다. 규칙 분량이 10~15줄이므로 별도 스킬을 생성하지 않고 orchestrator를 확장하는 것이 적절하다.

```
workflow-orchestrator (SKILL.md)
├── CRITICAL 섹션: AskUserQuestion 도구 사용 금지
├── Input Format Rules 섹션: 입력 형식 표준 정의
└── Output Format Rules 섹션: (기존)

각 스킬 (SKILL.md)
└── 기존 프롬프트 텍스트 → 새 형식으로 교체
```

## 핵심 결정 사항

| 결정 사항 | 선택 | 근거 |
|-----------|------|------|
| brainstorming과 plan-stage 피드백 루프 관계 | 별도 유지 | 스킬 간 의존성 방지, 독립 진화 가능. 통일 시 한쪽 수정이 동기화 문제를 유발 |
| 피드백 출력 형식 | 자유 형식 + 합의/미합의 마커 | brainstorming 발산적 성격 보존. 정형 구분선(plan-stage 방식)은 흐름을 저해 |
| 토론 리드 선정 방식 | 사용자 선택 (없음 허용) | 사용자 주도권 보장, 규칙 단순화. 자동 매칭은 로직 복잡, 고정 리드는 관점 편중 |
| 정규 상태값 | `ready-for-plan`, `complete` 2개 | phase 파일이 진행 추적 대체. 3단계 도입은 중복이며 불필요한 복잡성 |
| 비정규 상태값 처리 | 읽기 쪽 자동 수정(silent fix) | 사용자 개입 불필요. 경고 후 제안 방식은 불필요한 노이즈 |
| 자동 수정 위치 | 인덱싱 전 전처리 분리 | 기존 필터링 로직 불변. 인라인 수정은 부수 효과와 흐름 복잡화 리스크 |
| 허용 변형 범위 | `completed`만 | 시스템 메타데이터에 한국어 불필요. 과도한 관용은 비정규 관행 유발 |
| 입력 도구 | 텍스트 기반 번호 목록 | 환경 의존성 제로, 선택지 개수 제한 없음, 채팅 흐름 유지. AskUserQuestion은 2~4개 선택지 제한으로 모든 상황 적용 불가 |
| Yes/No 처리 | 번호화 (1.Yes / 2.No) | 단일 멘탈 모델 형성. 별도 텍스트 형식은 또 다른 패턴을 만들어 불일관의 원인 |
| 자연어 응답 처리 | 관용 모드 (수용) | Claude는 자연어 해석에 능숙, 사용자 입력 제한 불필요. 엄격 모드는 사용 경험 저하 |
| 직접 입력 탈출구 | 0번 예약 | 닫힌 선택에서 열린 입력으로의 일관된 탈출구. 마지막 선택지 방식은 위치가 매번 달라져 비일관 |
| 입력 형식 규칙 위치 | workflow-orchestrator 확장 | 모든 세션에서 로드, 별도 스킬 불필요. 규칙 분량 10~15줄로 새 스킬은 과잉 |
