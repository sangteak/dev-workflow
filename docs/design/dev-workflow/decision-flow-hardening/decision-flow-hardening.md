# Decision Flow Hardening (v1.10.0)

> v1.9.0 sequential-decision-flow의 배포 환경 리뷰(리뷰어 5인 → 발견 25건 → 검증 → 개선안 3안 → 심사 3인)에서 확정된
> 하이브리드 권고안을 구현한다. 본 문서는 브레인스토밍 국면을 대체한 리뷰 워크플로우 결과를 설계 결정으로 기록한다.

## 1. 배경 (문제 정의)

v1.9.0 리뷰에서 확정된 결함 25건 중 핵심:

1. **[HIGH·실질] SSOT 도달 불가** — `decision-flow.md`는 세션에 주입되지 않고, 4개 스킬의 참조는 리포 상대경로라
   사용자 프로젝트(플러그인 캐시 배포 환경)에서 해소 불가. v1.9.0 핵심 규범 전체가 배포 환경에서 사실상 미동작.
   v1.8.0의 brainstorming 인라인 형식 37줄이 삭제되어 미도달 시 이전 버전보다 퇴행.
2. **[HIGH] 상태 유실** — N/M 카운터·대기열·브리핑 순서가 대화 컨텍스트에만 존재. HANDOFF 템플릿에 필드 없음,
   compact/세션 이관 시 유실 → 모순 결정이 불변 phase 파일에 고착.
3. **[HIGH] 이중 규범** — Input Format Rules(무박스)와 D 풀박스 공존, '결정'의 정의·경계·우선순위 부재.
4. **임계값 8 총체적 붕괴** — 압축 헤더 기호 스왑(N/K/M 의미 역전), "풀 박스 3개 표시"는 D 템플릿과 구조 모순,
   분할 제안 자체가 메타 질문 역설(질문을 줄이겠다며 질문 추가).
5. **사용자 자연 행동 미정의** — 일괄 응답·순서 무시·위임("알아서 해")·즉시 번복·역질문·인터럽트 전부 스펙 밖.
6. **[실질] plan-stage 역행** — OPEN_QUESTIONS 템플릿이 "각 항목에 대해 결정을 내려주시면"으로 일괄 질문을 능동 유도.
7. 닫힌 ASCII 박스는 한글/이모지 전각 폭으로 우측 정렬 유지가 구조적으로 불가능.

전체 발견사항·검증·심사 기록: 리뷰 워크플로우 결과 (wf_48c75dc5-5a6, 2026-07-06).

## 2. 목표

- GOAL-001: 배포 환경(사용자 프로젝트 + 플러그인 캐시)에서 결정 흐름 규범이 실제로 도달·동작하게 한다.
- GOAL-002: "질문 폭탄 방지"라는 원 목표를 지키면서 사용자의 자연스러운 응답 행동을 전부 스펙 안으로 수용한다.
- GOAL-003: 자기모순 메커니즘(임계값 8 분할 제안·압축 헤더)을 수리 대신 제거하여 규칙 순증 없이 모순 계열을 소멸시킨다.
- GOAL-004: 세션 이관·컨텍스트 압축 경로에서 결정 흐름 상태(특히 재논의 대기열 전문)를 보존한다.

## 3. 요구사항

- REQ-001 (도달성 3중 방어): ① 4개 스킬의 수동 참조("따른다")를 능동 Read 지시로 교체 + 경로 해소 절차
  (세션 주입 플러그인 루트 → 스킬 디렉토리 형제 경로 → `find "$HOME/.claude/plugins" -path "*dev-workflow*/skills/..." -not -path "*/worktrees/*"` 동적 탐색 — 캐시의 버전 디렉토리 구조와 worktree 사본 오염에 실측 검증됨)
  ② 매 세션 주입되는 orchestrator SKILL.md에 핵심 형식 인라인 폴백 ③ brainstorming에 토론 출력 최소 규칙 인라인 복원.
  보강: hooks/session-start가 계산해둔 PLUGIN_ROOT 값을 주입 텍스트에 한 줄로 포함하여 1선 경로 해소가 세션 시작부터 성립.
- REQ-002 (응답 주권): "사용자 응답이 형식을 이긴다" 우산 원칙 + 구체 처리 5종(일괄/순서 무시/역질문/위임/인터럽트).
- REQ-003 (진행 모드 3종): 하나씩(기본)/한꺼번에/알아서. 모드는 국면 한정, 복구 시 기본값 리셋, 브리핑 푸터에 안내 접기.
- REQ-004 (임계값 8 대체): 분할 메타 질문·압축 헤더 삭제 → 미결 8개+ 시 브리핑 푸터 권장 1줄. 헤더는 단일 형식만.
- REQ-005 (이벤트 기반 헤더): '매 응답' → 결정 요청/카운터 변경/흐름 복귀 시에만 표시. 기호 1의미 1기호 고정.
  값은 대화 기억이 아닌 브리핑 목록 재집계로 산출.
- REQ-006 (오픈형 박스): 우측 테두리를 닫지 않는 오픈 박스로 교체 — 전각 폭 정렬 실패를 구조적으로 제거.
- REQ-007 (결정의 조작적 정의): 산출물에 기록될 선택지 2개+ 선택만 '결정'. 진행/승인 게이트는 Input Format Rules 담당.
  경계·우선순위(로컬 템플릿 우선, 적용 예외 판단 기준, .claude/rules 충돌 시) 명문화.
- REQ-008 (대기열 완결화): 추가 1회 고지, 노출 박스에 항목 전문 목록, 자연 해소 규칙, 재편입 규칙, 노출 시점 단일화
  (국면 완료 시 + phase 파일 직전 안전망), 합의 항목 이름 부여([=] 참조 ID), 발견 주체 구분(사용자 요청은 즉시 처리).
- REQ-009 (상태 보존 경량판): HANDOFF에 조건부 '결정 흐름 상태' 섹션(대기열은 전문 기록 — 건수 금지).
  기록 시점은 save/compact 직전 + 국면 완료 시로 한정(매 결정마다 파일쓰기 금지). resume 시 복원 절차.
- REQ-010 (목록 변경 규칙): 신규 발견 시 경량 이의 게이트(별도 응답 대기 없음), 무효화 시 고지 후 제거(조용한 제거 금지).
- REQ-011 (국면 대응표): brainstorming=국면 0~4 각각 / plan-stage=Step 1~4 전체 1국면 / 기타=스킬 실행 1회.
- REQ-012 (서브에이전트 규범): 서브에이전트 원문의 결정 요청 문구 미노출 — 메인 컨텍스트가 SSOT 형식으로 재구성.
- REQ-013 (plan-stage 정합): OPEN_QUESTIONS 템플릿을 브리핑(F) 정합 구조 + 모드 안내로 교체.
- REQ-014 (document-consolidation 정합): 13행 자기모순 문장을 명시적 '적용 예외' 선언으로 교체.
- REQ-015 (AskUserQuestion 경계): 금지의 적용 범위(본 워크플로우 주관 단계 한정)를 CRITICAL 섹션에 명시.

## 4. 채택 설계 (심사 합의 하이브리드)

**본체 = 개선안 ③ (상호작용 모드 재설계, 심사 2/3 승자)** — 응답 주권 + 3모드 + 이벤트 기반 헤더 +
깨진 메커니즘 삭제. "규칙 2개 삭제 + 원칙 2개 추가"로 순증 없이 미정의 10여 건을 원칙 수준에서 계열 해소.

**이식 (심사 3인 공통 graft):**
- 개선안 ①에서: 도달성 3중 방어(전원 "필수 선결" 판정), 오픈형 박스, 결정의 조작적 정의, 로컬 템플릿 우선,
  국면 대응표, brainstorming 인라인 최소 규칙 복원, document-consolidation 예외 선언, 합의 항목 이름 부여,
  서브에이전트 재구성 책임, AskUserQuestion 경계.
- 개선안 ②에서: HANDOFF '결정 흐름 상태' 섹션(경량판 — 4-이벤트 상시 기록 대신 save/compact/국면 완료 시점 기록),
  헤더 재산출 규칙, [=] 합의 마커 어휘.

## 6. 성공 기준

- SC-001: 사용자 프로젝트 관점 시뮬레이션에서 결정 요청 전 SSOT Read 시도가 지시문에 의해 트리거되고,
  Read 실패 시에도 인라인 폴백으로 "하나씩 + 오픈 박스 + 헤더" 형식이 유지된다.
- SC-002: 수정된 6개 파일 간 신규 모순이 없다 (검증 에이전트 교차 확인).
- SC-003: v1.8.0 대비 회귀 없음 — 에이전트 태그·합의/미합의 마커가 SSOT 미도달 시에도 brainstorming 본문에서 보장된다.
- SC-004: 일괄 응답·위임·순서 무시·역질문·인터럽트 각각에 대해 스펙이 단일한 동작을 정의한다.
- SC-005: 삭제된 규칙(임계값 8 분할 제안, 압축 헤더, 클러스터 그룹핑 독립 소절)의 잔재 참조가 어디에도 남지 않는다.
- SC-006: 기존 사용자 프로젝트의 docs/design/ 구조가 무변경으로 동작한다 (MINOR 릴리스 적합성).
